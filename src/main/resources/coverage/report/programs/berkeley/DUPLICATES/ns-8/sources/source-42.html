25 /**
26  * A database cursor for a secondary database. Cursors are not thread safe and
27  * the application is responsible for coordinating any multithreaded access to
28  * a single cursor object.
29  *
30  * <p>Secondary cursors are returned by {@link SecondaryDatabase#openCursor
31  * SecondaryDatabase.openCursor} and {@link
32  * SecondaryDatabase#openSecondaryCursor
33  * SecondaryDatabase.openSecondaryCursor}.  The distinguishing characteristics
34  * of a secondary cursor are:</p>
35  *
36  * <ul> <li>Direct calls to <code>put()</code> methods on a secondary cursor
37  * are prohibited.
38  *
39  * <li>The {@link #delete} method of a secondary cursor will delete the primary
40  * record and as well as all its associated secondary records.
41  *
42  * <li>Calls to all get methods will return the data from the associated
43  * primary database.
44  *
45  * <li>Additional get method signatures are provided to return the primary key
46  * in an additional pKey parameter.
47  *
48  * <li>Calls to {@link #dup} will return a {@link SecondaryCursor}.
49  *
50  * </ul>
51  *
52  * <p>To obtain a secondary cursor with default attributes:</p>
53  *
54  * <blockquote><pre>
55  *     SecondaryCursor cursor = myDb.openSecondaryCursor(txn, null);
56  * </pre></blockquote>
57  *
58  * <p>To customize the attributes of a cursor, use a CursorConfig object.</p>
59  *
60  * <blockquote><pre>
61  *     CursorConfig config = new CursorConfig();
62  *     config.setReadUncommitted(true);
63  *     SecondaryCursor cursor = myDb.openSecondaryCursor(txn, config);

185 
186         checkOpenAndState(true);
187 
188         trace(Level.FINEST, "SecondaryCursor.delete: ", null);
189 
190         final CacheMode cacheMode =
191             options != null ? options.getCacheMode() : null;
192 
193         /* Read the primary key (the data of a secondary). */
194         final DatabaseEntry key = new DatabaseEntry();
195         final DatabaseEntry pKey = new DatabaseEntry();
196 
197         /*
198          * Currently we write-lock the secondary before deleting the primary,
199          * which reverses the normal locking order and is deadlock-prone.
200          *

202          * then perform a special delete-if-has-secondary-key operation on the
203          * primary. We must be careful not to delete the primary record if,
204          * after locking it, it does not reference this secondary key.
205          */

207             getCurrentInternal(key, pKey, LockMode.RMW, cacheMode);
208

210             return null;
211         }
212

214         final Database primaryDb = secondaryDb.getPrimary(pKey);
215 
216         if (primaryDb == null) {

218             deleteNoNotify(cacheMode, getDatabaseImpl().getRepContext());

220         }
221

223         final OperationResult priResult =

1667     /**
1668      * Reads the primary record associated with a secondary record.
1669      *
1670      * An approach is used for secondary DB access that avoids deadlocks that
1671      * would occur if locks were acquired on primary and secondary DBs in
1672      * different orders for different operations.  The primary DB lock must
1673      * always be acquired first when doing a write op; for example, when
1674      * deleting a primary record, we don't know what the secondary keys are
1675      * until we read (and lock) the primary record.  However, the natural way
1676      * to read via a secondary DB would be to read (and lock) the secondary
1677      * record first to obtain the primary key, and then read (and lock) the
1678      * primary record. Because this would obtain locks in the reverse order as
1679      * write ops, a different approach is used for secondary reads.
1680      *
1681      * In order to avoid deadlocks, for non-serializable isolation we change
1682      * the natural lock order for reads -- we only lock the primary record and
1683      * then check the secondary record's reference to primary record. The
1684      * initial read of the secondary DB is performed without acquiring locks
1685      * (dirty-read). The primary key is then used to read and lock the
1686      * associated primary record. At this point only the primary record is
1687      * locked.
1688      *
1689      * Then, the secondary reference is checked (see checkReferenceToPrimary in
1690      * Cursor). Note that there is no need to lock the secondary before
1691      * checking its reference to the primary, because during the check the
1692      * secondary is protected from changes by the lock on the primary. If we
1693      * discover that the secondary record has been deleted (for example, due to
1694      * an update to the primary after the secondary dirty-read and before the
1695      * primary locking read), the record will not be returned to the caller (it
1696      * will be skipped) and we will advance to the next record according to the
1697      * operation type. In this case the lock on the primary record is released.
1698      *
1699      * In addition, the READ_UNCOMMITTED_ALL mode is used for the dirty-read
1700      * of the secondary DB.  This ensures that we do not skip uncommitted
1701      * deleted records.  See LockMode.READ_UNCOMMITTED_ALL and
1702      * Cursor.readPrimaryAfterGet for further details.
1703      *
1704      * For a secondary DB with dups, READ_UNCOMMITTED_ALL will return a deleted
1705      * record for an open txn, and we'll discover the deletion when reading
1706      * (and locking) the primary record. The primary lookup is wasted in that
1707      * case, but this should be infrequent. For a secondary DB without dups,
1708      * READ_UNCOMMITTED_ALL will block during the secondary read in this case
1709      * (a deleted record for an open txn) in order to obtain the data (the
1710      * primary key).
1711      *

1713      * using read-uncommitted), or the primary database has been removed from
