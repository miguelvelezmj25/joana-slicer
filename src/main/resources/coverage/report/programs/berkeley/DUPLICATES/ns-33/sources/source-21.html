16 /**
17  * Thrown and handled internally when metadata must be refreshed on a Replica.
18  *
19  * There are several scenarios for refreshing DPL metadata:
20  *
21  * 1. Read entity record on Replica that has stale in-memory metadata.
22  *
23  *    When an entity record that references new metadata (for example, a never
24  *    before encountered class) is written on the Master, the metadata is
25  *    written and replicated prior to writing and replicating the entity
26  *    record.  However, the Replica's in-memory cache of metadata (the
27  *    PersistCatalog object) is not synchronously updated when the metadata is
28  *    replicated.  When the entity record that references the newly replicated
29  *    metadata is read on the Replica, the DPL must refresh the in-memory
30  *    metadata cache by reading it from the catalog database.
31  *
32  *    Note that this scenario occurs even without class evolution/upgrade, for
33  *    two reasons.  First, the Master does not write all metadata at once;
34  *    metadata is added to the catalog incrementally as new persistent classes
35  *    are encountered.  Second, even when all metadata is written intially by
36  *    the Master, the Replica may read the catalog before the Master has
37  *    completed metadata updates.
38  *
39  *    Implementation:
40  *    + PersistCatalog.getFormat(int) throws RefreshException when the given
41  *      format ID is not in the in-memory catalog.
42  *    + The binding method that is calling getFormat catches RefreshException,
43  *      calls RefreshException.refresh to read the updated metadata, and
44  *      retries the operation.
45  *
46  *    Tests:
47  *      c.s.je.rep.persist.test.UpgradeTest.testIncrementalMetadataChanges
48  *      c.s.je.rep.persist.test.UpgradeTest.testUpgrade
49  *
50  * 2. Write entity record on Master that is in Replica Upgrade Mode.
51  *
52  *    When a Replica is upgraded with new persistent classes (see
53  *    evolve/package.html doc) the DPL will evolve the existing metadata and
54  *    update the in-memory metadata cache (PersistCatalog object), but will not
55  *    write the metadata; instead, it will enter Replica Upgrade Mode.  In this
56  *    mode, the Replica will convert old format data to new format data as
57  *    records are read, using the in-memory evolved metadata.  This allows the
58  *    Replica application to perform entity read operations with the new
59  *    persistent classes, during the upgrade process.
60  *
61  *    When this Replica is elected Master, the application will begin writing
62  *    entity records.  Note that the new metadata has not yet been written to
63  *    the catalog database.  In Replica Upgrade Mode, the current in-memory
64  *    metadata cannot be written to disk, since the catalog database may be
65  *    stale, i.e., it have been updated by the Master after the Replica's
66  *    in-memory metadata was evolved.  Therefore, before the first entity
67  *    record is written, the newly elected Master must read the latest
68  *    metadata, perform evolution of the metadata again, write the metadata,
69  *    and then write the entity record.
70  *
71  *    Implementation:
72  *    + The catalog enters Replica Upgrade Mode when a new or evolved format is
73  *      added to the catalog, and a ReplicaWriteException occurs when
74  *      attempting to write the metadata.  Replica Upgrade Mode is defined as
75  *      when the number of in-memory formats is greater than the number of
76  *      stored formats.
77  *    + Before an entity record is written, PersistEntityBinding.objectToData
78  *      is called to convert the entity object to record data.
79  *    + objectToData calls PersistCatalog.checkWriteInReplicaUpgradeMode,
80  *      which throws RefreshExeption in Replica Upgrade Mode.
81  *    + objectToData catches RefreshException, calls RefreshException.refresh
82  *      to read the updated metadata, and retries the operation.
83  *
84  *    Tests:
85  *      c.s.je.rep.persist.test.UpgradeTest.testElectedMasterWithStaleMetadata
86  *      c.s.je.rep.persist.test.UpgradeTest.testRefreshAfterFirstWrite
87  *      c.s.je.rep.persist.test.UpgradeTest.testUpgrade
88  *
89  * 3. Write metadata on Master that is in Replica Upgrade Mode.
90  *
91  *    This third scenario is more unusual than the first two.  It occurs when
92  *    a Replica with stale metadata is elected Master, but is not in Replica
93  *    Upgrade Mode.  The new Master must refresh metadata before writing
94  *    metadata.  See the test case for more information.
95  *
96  *    Implementation:
97  *    + On a Master with stale metadata, the application tries to write an
98  *      entity record that refers to a class that has not been encountered
99  *      before.
100  *    + Before the entity record is written, PersistEntityBinding.objectToData
101  *      is called to convert the entity object to record data.
102  *    + objectToData calls PersistCatalog.addNewFormat during serialization,
103  *      which attempts to write metadata by by calling writeDataCheckStale.
104  *    + writeDataCheckStale reads the existing metadata and detects that it
105  *      has changed since metadata was last read, and throws RefreshExeption.
106  *    + objectToData catches RefreshException, calls RefreshException.refresh
107  *      to read the updated metadata, and retries the operation.
108  *
109  *    Tests: 
110  *      c.s.je.rep.persist.test.UpgradeTest.testRefreshBeforeWrite
111  */
112 public class RefreshException extends Exception {
113 
114     private final Store store;
115     private final PersistCatalog catalog;
116     private final int formatId;
117 
118     RefreshException(final Store store,
119                      final PersistCatalog catalog,
120                      final int formatId) {
121         this.store = store;
122         this.catalog = catalog;
123         this.formatId = formatId;
124     }
125 
126     /**
127      * This method must be called to handle this exception in the binding
128      * methods, after the stack has unwound.  The binding methods should retry
129      * the operation once after calling this method.  If the operation fails
130      * again, then corruption rather than stale metadata is the likely cause
131      * of the problem, and an exception will be thrown to that effect.
132      * [#16655]
133      */
134     public PersistCatalog refresh() {
135         return store.refresh(catalog, formatId, this);
136     }
137 
138     @Override
139     public String getMessage() {
140         return "formatId=" + formatId;
141     }
142 }
