16 /**
17  * Implemented by off-heap memory allocators.
18  *
19  * The allocator is responsible for allocating and freeing a block of memory
20  * efficiently, maintaining the size of a block, identifying a block by a long
21  * integer value (ID), and looking up a memory ID efficiently (in constant
22  * time) in order to copy bytes, return the size, or free the block.
23  * <p>
24  * The allocator is also responsible for compacting memory when necessary to
25  * perform allocations efficiently. A special case is when the off-heap cache
26  * is reduced in size, and memory should be compacted to make memory available
27  * to the OS; the implementation should account for this case.
28  * <p>
29  * Another responsibility of the allocator is to estimate the RAM usage for all
30  * blocks currently allocated, including overhead and space taken by
31  * fragmentation. It is recognized that this may only be a rough estimate in
32  * some implementations (the default allocator, for example). See
33  * {@link #getUsedBytes}.
34  * <p>
35  * Note that with the default allocator, the size is not a built-in property
36  * of each block, and the {@link #size} method will be implemented by storing
37  * the size at the front of the block. The {@code size} method is included in
38  * the interface to allow for implementations where a block size property is
39  * naturally available.
40  * <p>
41  * This interface requires that memory is copied in and out of the Java address
42  * space to make use of the off-heap cache. A future enhancement might involve
43  * adding a way to obtain a ByteBuffer for direct access to the off-heap memory
44  * block, to avoid the copy if this is possible in some implementations. In the
45  * default implementation, this is not practical without using non-public JVM
46  * internals and risking incompatibilities.
47  * <p>
48  * All methods in the allocator must be thread safe, and contention among
49  * threads should be minimized.
50  * <p>
51  * The memory blocks are not assumed to be fast access RAM and in particular
52  * might be NVRAM. JE makes an effort to only copy memory to/from the block
53  * when necessary, and to use single larger copy operations rather than
54  * multiple smaller copy operations.
55  */

65      * use for the off-heap cache, at the gross level of dividing up the RAM on
66      * a machine among processes, the off-heap cache, the file system cache,

94     /**
95      * Allocates a block of a given size and returns its ID.
96      *
97      * The bytes of the memory block must be initialized to zero.
98      * <p>
99      * Note that because the used cache size is only an estimate, and in fact
100      * the maximum size might not actually be available (due to memory use by
101      * other processes when using the default allocator, for example), then the
102      * {@link #allocate} method may fail (thrown an exception) even when the
103      * used size is less than the maximum bytes (the value passed to {@link
104      * #setMaxBytes}). JE handles this situation as follows.
105      * <p>
106      * JE uses an internal off-heap cache size limit to determine when to
107      * perform eviction (which frees off-heap blocks). The limit is
108      * initialized to the value passed to {@link #setMaxBytes}. JE calls
109      * {@link #getUsedBytes()} to determine when to evict memory from the
110      * cache. If the used bytes by grows very close to the limit or exceeds
111      * it, JE will perform off-heap cache eviction. JE will make a best effort
112      * not to call the {@link #allocate} method when the used size exceeds
113      * the limit.
114      * <p>
115      * If an allocation failure occurs (i.e., this method throws an
116      * exception), JE adjusts the limit downward to account for the
117      * inaccuracies discussed above. When a RuntimeException is thrown, the
118      * limit is set to the used size; when an OutOfMemoryError is thrown, the
119      * limit is set to the used size minus the {@link
120      * com.sleepycat.je.EnvironmentConfig#OFFHEAP_EVICT_BYTES}. This adjustment
121      * should ensure that JE eviction occurs and prevent frequent allocation
122      * failures and associated exception handling.
123      * TODO: This never happens because Linux kills the process
124      * <p>
125      *
126      * @return non-zero memory ID, or zero when the memory cannot be allocated.
127      *
128      * @throws OffHeapOverflowException if the block cannot be allocated
129      * because the max size has been reached. The internal off-heap cache
130      * size limit will be set as described above.
131      *

133      * system memory is available. The internal off-heap cache size limit will
134      * be set as described above. In addition, a SEVERE message for the
