517     /**
518      * The size in MiB to be read sequentially at the end of the log in order
519      * to warm the file system cache.
520      * <p>
521      * Making use of sequential reads to warm the file system cache has the
522      * benefit of reducing random reads caused by CRUD operations, and thereby
523      * increasing throughput and latency for these operations. This is
524      * especially true during the initial period after opening an Environment,
525      * when CRUD operations must fetch Btree internal nodes from the file
526      * system in order to populate the JE cache. The fetches due to JE cache
527      * misses typically cause random reads. Often the Btree internal nodes that
528      * are needed appear close to the end of the log because they were written
529      * fairly recently by checkpoints, and this is why warming the cache with
530      * the data at the end of the log is often beneficial.
531      * <p>
532      * The warm-up occurs concurrently with recovery when an Environment is
533      * opened. It may finish before recovery finishes, or continue after
534      * recovery finishes when recovery is brief. In the latter case, the
535      * warm-up is concurrent with the application's CRUD operations. A
536      * dedicated thread is used for the warm-up, and this thread is destroyed
537      * when warm-up is complete.
538      * <p>
539      * Recovery itself will perform at least a partial warm-up implicitly,
540      * since it reads the log (sequentially), and in fact it may read more than
541      * the configured warm-up size. The warm-up thread will only read the
542      * portion of the log not being read by recovery, and only when the warm-up
543      * size is larger than the size read by recovery (i.e., it reads the
544      * difference between these two sizes).
545      * <p>
546      * The size read by recovery is dependent on whether the Environment was
547      * previously closed cleanly (a crash did not occur and the application
548      * called Environment.close), and on the size of the last complete
549      * checkpoint. When the environment is closed cleanly with a small
550      * checkpoint, recovery will only read a small portion of the log, and in
551      * this case the additional reads performed by the warm-up thread can be
552      * very beneficial.
553      * <p>
554      * If the warm-up size is larger than the amount of memory available to the
555      * file system cache, then the warm-up may be counter productive, although
556      * TODO: change text below or change default to 1024
557      * the default warm-up size (1 GiB) was chosen to avoid this problem in
558      * most cases. Applications are advised to change the warm-up size based on
559      * knowledge of the amount of physical memory on the machine and how much
560      * is expected to be available as file system cache. The warm-up may be
561      * disabled by setting the warm-up size to zero, although of course
562      * recovery will continue to do some amount of warm-up implicitly.
563      * <p>
564      * The warm-up thread performs read operations using a single buffer and it
565      * reads as much as will fit in the buffer at a time. The size of the
566      * buffer, and therefore the maximum size of each read, is {@link
567      * #LOG_FILE_WARM_UP_READ_SIZE}. Files are read in the reverse of the order
568      * they were written.
569      *
