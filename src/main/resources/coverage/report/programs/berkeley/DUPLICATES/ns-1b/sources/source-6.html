24 /**
25  * The protocol used to obtain backup files from a LF Feeder. The message
26  * exchange is always initiated by the client.
27  *
28  * The following describes the request/response messages exchanged between the
29  * two nodes:
30  *
31  *      FeederInfoReq -> FeederInfoResp
32  *
33  *      FileListReq -> FileListResp
34  *
35  *      FileInfoReq -> FileInfoResp
36  *
37  *      FileReq -> FileStart <byte stream> FileEnd
38  *
39  *      Done
40  *
41  * So a complete sequence of successful request messages looks like:
42  *
43  * FeederInfoReq FileListReq [[FileInfoReq] [FileReq] ]+ Done
44  *
45  * A response sequence would look like:
46  *
47  * FeederInfoResp FileListResp [[FileInfoResp] [FileStart <byte stream> FileEnd] ]+
48  *
49  * The client may abandon its interaction with the server if it decides the
50  * server is overloaded.
51  *
52  * The client tries to minimize the number of files it actually requests based
53  * upon its current state.
54  *
55  * When a FileReq is received by the server, other files previously requested
56  * (using FileReq) may be deleted by the server. These previously requested
57  * files must not be requested again using FileReq or FileReqInfo.
58  */
