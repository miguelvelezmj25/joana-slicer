19 /**
20  * Thrown when {@code java.lang.InterruptedException} (a thread interrupt) or
21  * {@code java.nio.channels.ClosedChannelException} (which also results from a
22  * thread interrupt) occurs in any JE method.  This occurs when the application,
23  * or perhaps a library or container that the application is using, calls
24  * {@link Thread#interrupt}.
25  *
26  * <p>Calling {@code Thread.interrupt} is not recommended for an active JE
27  * thread if the goal is to stop the thread or do thread coordination. If you
28  * interrupt a thread that is executing a JE operation, the state of the
29  * environment will be undefined.  That's because JE might have been in the
30  * middle of I/O activity when the operation was aborted midstream, and it
31  * becomes very difficult to detect and handle all possible outcomes.</p>
32  *
33  * <p>When JE detects the interrupt, it will mark the environment invalid and
34  * will throw a {@code ThreadInterruptedException}. This tells you that you
35  * must close the environment and re-open it before using it again. This is
36  * necessary, because if JE didn't throw {@code ThreadInterruptedException}, it
37  * is very likely that you would get some other exception that is less
38  * meaningful, or simply see corrupted data.</p>
39  * 
40  * <p>Instead, applications should use other mechanisms like {@code
41  * Object.notify} and {@code wait} to coordinate threads. For example, use a
42  * {@code keepRunning} variable of some kind in each thread. Check this
43  * variable in your threads, and return from the thread when it is false. Set
44  * it to false when you want to stop the thread. If this thread is waiting to
45  * be woken up to do another unit of work, use {@code Object.notify} to wake it
46  * up. This is the recommended technique.</p>
47  *
48  * <p>However, if the use of {@code Thread.interrupt} is unavoidable, be sure
49  * to use it only when shutting down the environment.  In this situation,
50  * the {@code ThreadInterruptedException} should be expected.  Note that
51  * by shutting down the environment abnormally, recovery time will be longer
52  * when the environment is subsequently opened, because a final checkpoint was
53  * not performed.</p>
54  *
55  * <p>Existing {@link Environment} handles are invalidated as a result of this
56  * exception.</p>
57  *
58  * @since 4.0
59  */
