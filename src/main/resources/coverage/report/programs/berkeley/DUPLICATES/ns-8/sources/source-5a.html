22 /**
23  * Options for calling methods that write (insert, update or delete) records.
24  *
25  * <h3><a name="ttl">Time-To-Live</a></h3>
26  *
27  * <p>When performing a 'put' operation, a TTL may be specified using
28  * {@link #setTTL(int, TimeUnit)} or {@link #setTTL(int)}.</p>
29  *
30  * <p>By default, the TTL property is zero, meaning there is no automatic
31  * expiration. A non-zero TTL may be specified to cause an inserted record
32  * to expire. The expiration time may also be changed for an existing
33  * record by updating the record and specifying a different TTL, including
34  * specifying zero to prevent the record from expiring. However, the TTL of
35  * an existing record is updated only if {@link #setUpdateTTL(boolean)} is
36  * explicitly set to true. When deleting a record, the TTL parameter is
37  * ignored.</p>
38  *
39  * <p>Records expire on day or hour boundaries, depending on the {@code
40  * timeUnit} parameter. At the time of the write operation, the TTL
41  * parameter is used to compute the record's expiration time by first
42  * converting it from days (or hours) to milliseconds, and then adding it
43  * to the current system time. If the resulting expiration time is not
44  * evenly divisible by the number of milliseconds in one day (or hour), it
45  * is rounded up to the nearest day (or hour).</p>
46  *
47  * <p>Passing TimeUnit.DAYS, rather than TimeUnit.HOURS, for the timeUnit
48  * parameter is recommended to minimize storage requirements (memory
49  * and disk). Because the expiration time is stored in the JE Btree
50  * internally, when using the TTL feature, the additional memory and disk
51  * space required for storing Btree internal nodes (INs) is twice as much
52  * when using TimeUnit.HOURS as when using TimeUnit.DAYS. Using
53  * TimeUnit.DAYS adds about 5% to the space needed for INs, while
54  * TimeUnit.HOURS adds about 10%.</p>
55  *
56  * <p>Note that JE stores the expiration time of the record and not the
57  * original TTL value that was specified. The expiration time of a record
58  * is available when reading (or writing) records via {@link
59  * OperationResult#getExpirationTime()}.</p>
60  *
61  * <p>A summary of the behavior of expired records is as follows.</p>
62  * <ul>
63  *     <li>Space for expired records will be purged in the background by
64  *     the JE cleaner, and expired records will be filtered out of queries
65  *     even if they have not been purged.<p></li>
66  *
67  *     <li>Expired records are removed individually: there is no guarantee
68  *     that records with the same expiration time will be removed
69  *     simultaneously.<p></li>
70  *
71  *     <li>Records with expiration times support repeatable-read semantics
72  *     in most cases, but with some exceptions (described below).<p></li>
73  * </ul>
74  *
75  * <p>A more detailed description is below, including some information on
76  * how expired records are handled internally.</p>
77  * <ul>
78  *     <li>Expired records will be purged in order to reclaim disk space.
79  *     This happens in background over time, and there is no guarantee that
80  *     the space for a record will be reclaimed at any particular time.
81  *     Purging of expired records occurs during the normal JE cleaning
82  *     process. The goals of the purging process are:
83  *     <ol>
84  *         <li>to minimize the cost of purging;</li>
85  *         <li>to keep disk utilization below the {@link
86  *         EnvironmentConfig#CLEANER_MIN_UTILIZATION} threshold, as usual,
87  *         but taking into account expired data; and</li>
88  *         <li>to reclaim expired data gradually and avoid spikes in
89  *         cleaning on day and hour boundaries.</li>
90  *     </ol>
91  *
92  *     <li>Expired records that have not been purged will be filtered out
93  *     of queries and will not be returned to the application. In a
94  *     replicated environment, purging and filtering occur independently on
95  *     each node. For queries to return consistent results on all nodes,
96  *     the system clocks on all nodes must be synchronized.<p></li>
97  *
98  *     <li>Repeatable-read semantics are supported for records that expire
99  *     after being read. If a lock of any kind is held on a record and the
100  *     record expires, when accessing it again using the same transaction
101  *     or cursor, it will be accessed as if it is not expired. In other
102  *     words, locking a record prevents it from expiring, from the
103  *     viewpoint of that transaction or cursor. However, there are some
104  *     caveats and exceptions to this rule:
105  *        <ul>
106  *           <li>A lock by one transaction or cursor will not prevent a
107  *           record from being seen as expired when accessing it using a
108  *           different transaction or cursor.<p></li>
109  *
110  *           <li>In the unlikely event that the system clock is changed,
111  *           locking a record may not guarantee that the record's data has
112  *           not been purged, if the data is not read at the time the
113  *           record is locked. This is because the record's key and its
114  *           data are purged independently. It is possible to lock a record
115  *           without reading its data by passing null for the 'data'
116  *           parameter. If a record is locked in this manner, and the data
117  *           was previously purged because the system clock was changed,
118  *           then one of the following may occur, even when using the same
119  *           transaction or cursor that was used to lock the record:
120  *           <p></li>
121  *              <ul>
122  *              <li>If the record is read again with a non-null data
123  *              parameter, the operation may fail (return null) because the
124  *              data cannot be read.<p></li>
125  *
126  *              <li>If a partial update is attempted (passing a {@link
127  *              DatabaseEntry#setPartial(int,int,boolean) partial} 'data'
128  *              parameter), the operation may fail (return null)
129  *              because the pre-existing data cannot be read.<p></li>
130  *              </ul>
131  *        </ul>
132  *
133  *     <li>Even when multiple records have the same expiration time, JE
134  *     does not provide a way for them to expire atomically, as could be
135  *     done by explicitly deleting multiple records in a single
136  *     transaction. This restriction is for performance reasons; if records
137  *     could expire atomically, they could not be purged efficiently using
138  *     the JE cleaning process. Instead, each record expires individually,
139  *     as if each were deleted in a separate transaction. This means that
140  *     even when a set of records is inserted or updated atomically, a
141  *     query may return some but not not all of the records, when any of
142  *     the records expire at a time very close to the time of the query.
143  *     This is because the system clock is checked for each record
144  *     individually at the time it is read by the query, and because
145  *     expired records may be purged by other threads.<p></li>
146  *
147  *     <li>There are several special cases of the above rule that involve
148  *     access to primary and secondary databases. Because a given primary
149  *     record and its associated secondary records are normal records in
150  *     most respects, this set of records does not expire atomically. For
151  *     most read and write operations, JE treats the expiration of any
152  *     record in this set as if all records have expired, and in these
153  *     cases there is no special behavior to consider. For example:<p>
154  *        <ul>
155  *            <li>As long as the primary and secondary databases are
156  *            transactional, JE ensures that the expiration times of a
157  *            given primary record and all its associated secondary records
158  *            are the same.<p></li>
159  *
160  *            <li>When reading a primary record via a secondary key, JE
161  *            first reads the secondary record and then the primary. If
162  *            either record expires during this process, both records are
163  *            treated as expired.<p></li>
164  *
165  *            <li>When updating or deleting a primary record, JE first
166  *            reads the primary record to obtain the secondary keys and
167  *            then deletes/updates/inserts the secondary records as
168  *            needed. If a secondary record expires during this process,
169  *            this will not cause a {@link SecondaryIntegrityException}, as
170  *            would normally happen when an expected associated record is
171  *            missing.<p></li>
172  *
173  *            <li>When a primary and/or secondary record expires after
174  *            being read, with few exceptions, repeatable-read semantics
175  *            are supported as described above, i.e., locks prevent
176  *            expiration from the viewpoint of the locking transaction or
177  *            cursor. Exceptions to this rule are described below.<p></li>
178  *        </ul>
179  *
180  *     However, there are several cases where such treatment by JE is not
181  *     practical, and the user should be aware of special behavior when
182  *     primary or secondary records expire. These are not common use cases,
183  *     but it is important to be aware of them. In the cases described
184  *     below, let us assume a primary database has two associated secondary
185  *     databases, and a particular primary record with primary key X has
186  *     two secondary records with keys A and B, one in each secondary
187  *     database.<p>
188  *        <ul>
189  *            <li>After a transaction or cursor reads and locks the primary
190  *            record via primary key X, reading via primary key X again
191  *            with the same transaction or cursor will also be successful
192  *            even if the record has expired, i.e., repeatable-read is
193  *            supported. However, if the record expires and the same
194  *            transaction or cursor attempts to read via key A or B, the
195  *            record will not be found. This is because the secondary
196  *            records for key A and B were not locked and they expire
197  *            independently of the primary record.<p></li>
198  *
199  *            <li>Similarly, after a transaction or cursor reads and locks
200  *            the primary record via secondary key A successfully, reading
201  *            via key A again with the same transaction or cursor will also
202  *            be successful even if the record has expired. Reading via
203  *            primary key X will also be successful, even if the record has
204  *            expired, because the primary record was locked. However, if
205  *            the record expires and the same transaction or cursor
206  *            attempts to read via key B, the record will not be found.
207  *            This is because the secondary record for key B was not locked
208  *            and it expires independently of the primary record and the
209  *            secondary record for key A.<p></li>
210  *
211  *            <li>When reading via a secondary database, it is possible to
212  *            read the only the secondary key and primary key (which are
213  *            both contained in the secondary record), but not the primary
214  *            record, by passing null for the 'data' parameter. In this
215  *            case the primary record is not locked. Therefore, if the
216  *            record expires and the same transaction or cursor attempts to
217  *            read the primary record (via any secondary key or the primary
218  *            key), the record will not be found.</li>
219  *
220  *            <li>When a record expires, if its database serves as
221  *            a {@link SecondaryConfig#setForeignKeyDatabase foreign key
222  *            database}, the {@link
223  *            SecondaryConfig#setForeignKeyDeleteAction foreign key delete
224  *            action} will not be enforced. Therefore, setting a TTL for a
225  *            record in a foreign key database is not recommended. The same
226  *            is true when using the DPL and a foreign key database is
227  *            specified using {@link
228  *            com.sleepycat.persist.model.SecondaryKey#relatedEntity()}.
229  *            <p></li>
230  *        </ul>
231  *     <p></li>
232  *
233  *     <li>When JE detects what may be an internal integrity error, it
234  *     tries to determine whether an expired record, rather than a true
235  *     integrity error, is the underlying cause. To prevent internal errors
236  *     when small changes in the system clock time are made, if a record
237  *     has expired within {@link EnvironmentConfig#ENV_TTL_CLOCK_TOLERANCE}
238  *     (two hours, by default), JE treats the record as deleted and no
239  *     exception is thrown.
240  *
241  *     <p>When an integrity error does cause an exception to be thrown, the
242  *     record's expiration time will be included in the exception message
243  *     and this can help to diagnose the problem. This includes the
244  *     following exceptions:
245  *        <ul>
246  *            <li>{@link SecondaryIntegrityException}</li>
247  *            <li>{@link EnvironmentFailureException} with
248  *            LOG_FILE_NOT_FOUND in the message.</li>
249  *        </ul>
250  *     </p>
251  *
252  *     <p>In cases where the clock has been changed by more than one hour
253  *     and integrity exceptions occur because of this, it may be possible
254  *     to avoid the exceptions by setting the {@link
255  *     EnvironmentConfig#ENV_TTL_CLOCK_TOLERANCE} configuration parameter
256  *     to a larger value.</p></li>
257  * </ul>
258  *
259  * <p>In order to use the TTL feature in a ReplicatedEnvironment, all nodes
260  * must be upgraded to JE 7.0 or later. If one or more nodes in a group
261  * uses an earlier version, an IllegalStateException will be thrown when
262  * attempting a put operation with a non-zero TTL. Also, once records with
263  * a non-zero TTL have been written, a node using an earlier version of JE
264  * may not join the group; if this is attempted, the node will fail during
265  * open with an EnvironmentFailureException.</p>
266  *
267  * @since 7.0
268  */

413     /**
414      * A convenience method to set the TTL based on a given expiration time
415      * and the current system time.
416      * <p>
417      * Given a desired expiration time and {@link TimeUnit} (DAYS or HOURS),
418      * sets the TTL to a value that will cause a record to expire at or after
419      * the given time, if the record is stored at the current time. The
420      * intended use case is to determine the TTL when writing a record and the
421      * desired expiration time, rather than the TTL, is known.
422      * <p>
423      * This method determines the TTL by taking the difference between the
424      * current time and the given time, converting it from milliseconds to
425      * days (or hours), and rounding up if it is not evenly divisible by
426      * the number of milliseconds in one day (or hour).
427      * <p>
428      * A special use case is when the expiration time was previously obtained
429      * from {@link OperationResult#getExpirationTime()}, for example, when
430      * performing an export followed by an import. To support this, null can be
431      * passed for the timeUnit parameter and the time unit will be determined
432      * as follows.
433      * <ul>
434      *     <li>This method first converts the expiration time to a TTL in
435      *     hours, as described above. If the expiration time was obtained by
436      *     calling {@link OperationResult#getExpirationTime}, then it will be
437      *     evenly divisible by the number of milliseconds in one hour and no
438      *     rounding will occur.</li>
439      *
440      *     <li>If the resulting TTL in hours is an even multiple of 24,
441      *     {@code DAYS} is used; otherwise, {@code HOURS} is used. For example,
442      *     when performing an import, if the original expiration time was
443      *     specified in {@code DAYS}, and obtained by calling
444      *     {@link OperationResult#getExpirationTime}, the unit derived by this
445      *     method will also be {@code DAYS}.</li>
446      * </ul>
447      * <p>
448      * Note that when a particular time unit is desired, null should not be
449      * passed for the timeUnit parameter. Normally {@link TimeUnit#DAYS} is
450      * recommended instead of {@link TimeUnit#HOURS}, to minimize storage
451      * requirements (memory and disk). When the desired unit is known, the unit
452      * should be passed explicitly.
453      *
454      * @param expirationTime is the desired expiration time in milliseconds
455      * (UTC), or zero for no automatic expiration.
456      *
457      * @param timeUnit is {@link TimeUnit#DAYS} or {@link TimeUnit#HOURS}, or
458      * null to derive the time unit as described above.
459      *
460      * @throws IllegalArgumentException if ttlUnits is not DAYS, HOURS or null.
461      *
462      * @see <a href="#ttl">Time-To-Live</a>
463      */
464     public WriteOptions setExpirationTime(final long expirationTime,
465                                           TimeUnit timeUnit) {
466         if (expirationTime == 0) {
467             return setTTL(0, timeUnit);
468         }
469 
470         final boolean hours;
471 
472         if (timeUnit == TimeUnit.DAYS) {
473             hours = false;
474         } else if (timeUnit == TimeUnit.HOURS) {
475             hours = true;
476         } else if (timeUnit == null) {
477             hours = TTL.isSystemTimeInHours(expirationTime);
478             timeUnit = hours ? TimeUnit.HOURS : TimeUnit.DAYS;
479         } else {
480             throw new IllegalArgumentException(
481                 "ttlUnits not allowed: " + timeUnit);
482         }
483 
484         setTTL(
485             TTL.systemTimeToExpiration(
486                 expirationTime - TTL.currentSystemTime(), hours),
487             timeUnit);
488 
489         return this;
490     }
