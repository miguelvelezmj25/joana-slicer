289  * <p>The following sections describe each group of stats along with some

291  *
292  * <h3><a name="cache">Cache Statistics</a></h3>
293  *
294  * <p style="margin-left: 2em">Group Name: {@value
295  * EvictorStatDefinition#GROUP_NAME}
296  * <br/>Description: {@value
297  * EvictorStatDefinition#GROUP_DESC}</p>
298  *
299  * <p style="margin-left: 2em">Group Name: {@value
300  * OffHeapStatDefinition#GROUP_NAME}
301  * <br/>Description: {@value
302  * OffHeapStatDefinition#GROUP_DESC}</p>
303  *
304  * <p>The JE cache consists of the main (in-heap) cache and and optional
305  * off-heap cache. The vast majority of the cache is occupied by Btree nodes,
306  * including internal nodes (INs) and leaf nodes (LNs). INs contain record keys
307  * while LNs contain record data.</p>
308  *
309  * <p>Each IN refers to a configured maximum number of child nodes ({@link
310  * EnvironmentConfig#NODE_MAX_ENTRIES}). The INs form a Btree of at least 2
311  * levels. With a large data set the Btree will normally have 4 or 5 levels.
312  * The top level is a single node, the root IN. Levels are numbered from the
313  * bottom up, starting with level 1 for bottom level INs (BINs). Levels are
314  * added at the top when the root IN splits.</p>
315  *
316  * <p>When an off-heap cache is configured, it serves as an overflow for the
317  * main cache. See {@link EnvironmentConfig#MAX_OFF_HEAP_MEMORY}.</p>
318  *
319  * <h4><a name="cacheSizing">Cache Statistics: Sizing</a></h4>
320  *
321  * <p>Operation performance is often directly proportional to how much of the
322  * active data set is cached. BINs and LNs form the vast majority of the cache.
323  * Caching of BINs and LNs have different performance impacts, and behavior
324  * varies depending on whether an off-heap cache is configured and which {@link
325  * CacheMode} is used.</p>
326  *
327  * <p>Main cache current usage is indicated by the following stats. Note that
328  * there is currently no stat for the number of LNs in the main cache.</p>
329  * <ul>
330  *     <li>{@link #getCacheTotalBytes()}</li>
331  *     <li>{@link #getNCachedBINs()}</li>
332  *     <li>{@link #getNCachedBINDeltas()}</li>
333  *     <li>{@link #getNCachedUpperINs()}</li>
334  * </ul>
335  *
336  * <p>Off-heap cache current usage is indicated by:</p>
337  * <ul>
338  *     <li>{@link #getOffHeapTotalBytes()}</li>
339  *     <li>{@link #getOffHeapCachedLNs()}</li>
340  *     <li>{@link #getOffHeapCachedBINs()}</li>
341  *     <li>{@link #getOffHeapCachedBINDeltas()}</li>
342  * </ul>
343  * </p>
344  *
345  * <p>A cache miss is considered a miss only when the object is not found in
346  * either cache. Misses often result in file I/O and are a primary indicator
347  * of cache performance. Fetches (access requests) and misses are indicated
348  * by:</p>
349  * <ul>
350  *     <li>{@link #getNLNsFetch()}</li>
351  *     <li>{@link #getNLNsFetchMiss()}</li>
352  *     <li>{@link #getNBINsFetch()}</li>
353  *     <li>{@link #getNBINsFetchMiss()}</li>
354  *     <li>{@link #getNBINDeltasFetchMiss()}</li>
355  *     <li>{@link #getNFullBINsMiss()}</li>
356  *     <li>{@link #getNUpperINsFetch()}</li>
357  *     <li>{@link #getNUpperINsFetchMiss()}</li>
358  * </ul>
359  *
360  * <p>When the number of LN misses ({@code nLNsFetchMiss}) or the number of
361  * BIN misses ({@code nBINsFetchMiss + nFullBINsMiss}) are significant, the
362  * JE cache may be undersized, as discussed below. But note that it is not
363  * practical to correlate the number of fetches and misses directly to
364  * application operations, because LNs are sometimes
365  * {@link EnvironmentConfig#TREE_MAX_EMBEDDED_LN embedded}, BINs are sometimes
366  * accessed multiple times per operation, and internal Btree accesses are
367  * included in the stat values.</p>
368  *
369  * <p>Ideally, all BINs and LNs for the active data set should fit in cache so
370  * that operations do not result in fetch misses, which often perform random
371  * read I/O. When this is not practical, which is often the case for large
372  * data sets, the next best thing is to ensure that all BINs fit in cache,
373  * so that an operation will perform at most one random read I/O to fetch
374  * the LN. The {@link DbCacheSize} javadoc describes how to size the cache
375  * to ensure that all BINs and/or LNs fit in cache.</p>
376  *
377  * <p>Normally {@link EnvironmentConfig#MAX_MEMORY_PERCENT} determines the JE
378  * cache size as a value relative to the JVM heap size, i.e., the heap size
379  * determines the cache size.</p>
380  *
381  * <p>For configuring cache size and behavior, see:</p>
382  * <ul>
383  *     <li>{@link EnvironmentConfig#MAX_MEMORY_PERCENT}</li>
384  *     <li>{@link EnvironmentConfig#MAX_MEMORY}</li>
385  *     <li>{@link EnvironmentConfig#MAX_OFF_HEAP_MEMORY}</li>
386  *     <li>{@link EnvironmentConfig#setCacheMode(CacheMode)}</li>
387  *     <li>{@link CacheMode}</li>
388  *     <li>{@link DbCacheSize}</li>
389  * </ul>
390  *
391  * <p>When using Oracle NoSQL DB, a sizing exercise and {@link DbCacheSize} are
392  * used to determine the cache size needed to hold all BINs in memory. The
393  * memory available to each node is divided between a 32 GB heap for the JVM
394  * process (so that CompressedOops may be used) and the off-heap cache (when
395  * more than 32 GB of memory is available).</p>
396  *
397  * <p>It is also important not to configured the cache size too large, relative
398  * to the JVM heap size. If there is not enough free space in the heap, Java
399  * GC pauses may become a problem. Increasing the default value for {@code
400  * MAX_MEMORY_PERCENT}, or setting {@code MAX_MEMORY} (which overrides {@code
401  * MAX_MEMORY_PERCENT}), should be done carefully.</p>
402  *
403  * <p>Java GC performance may also be improved by using {@link
404  * CacheMode#EVICT_LN}. Record data sizes should also be kept below 1 MB to
405  * avoid "humongous objects" (see Java GC documentation).</p>
406  *
407  * <p>When using Oracle NoSQL DB, by default, {@code MAX_MEMORY_PERCENT} is
408  * set to 70% and {@link CacheMode#EVICT_LN} is used. The LOB (large object)
409  * API is implemented using multiple JE records per LOB where the data size of
410  * each record is 1 MB or less.</p>
411  *
412  * <p>When a shared cache is configured, the main and off-heap cache may be
413  * shared by multiple JE Environments in a single JVM process. See:</p>
414  * <ul>
415  *     <li>{@link EnvironmentConfig#SHARED_CACHE}</li>
416  *     <li>{@link #getSharedCacheTotalBytes()}</li>
417  *     <li>{@link #getNSharedCacheEnvironments()}</li>
418  * </ul>
419  *
420  * <p>When using Oracle NoSQL DB, the JE shared cache feature is not used
421  * because each node only uses a single JE Environment.</p>
422  *
423  * <h4><a name="cacheSizeOptimizations">Cache Statistics: Size
424  * Optimizations</a></h4>
425  *
426  * <p>Since a large portion of an IN consists of record keys, JE uses
427  * {@link DatabaseConfig#setKeyPrefixing(boolean) key prefix compression}.
428  * Ideally, key suffixes are small enough to be stored using the {@link
429  * EnvironmentConfig#TREE_COMPACT_MAX_KEY_LENGTH compact key format}. The
430  * following stat indicates the number of INs using this compact format:</p>
431  * <ul>
432  *     <li>{@link #getNINCompactKeyIN()}</li>
433  * </ul>
434  *
435  * <p>Configuration params impacting key prefixing and the compact key format
436  * are:</p>
437  * <ul>
438  *     <li>{@link DatabaseConfig#setKeyPrefixing(boolean)}</li>
439  *     <li>{@link EnvironmentConfig#TREE_COMPACT_MAX_KEY_LENGTH}</li>
440  * </ul>
441  *
442  * <p>Enabling key prefixing for all databases is strongly recommended. When
443  * using Oracle NoSQL DB, key prefixing is always enabled.</p>
444  *
445  * <p>Another configuration param impacting BIN cache size is {@code
446  * TREE_MAX_EMBEDDED_LN}. There is currently no stat indicating the number of
447  * embedded LNs. See:</p>
448  * <ul>
449  *     <li>{@link EnvironmentConfig#TREE_MAX_EMBEDDED_LN}</li>
450  * </ul>
451  *
452  * <h4><a name="cacheUnexpectedSizes">Cache Statistics: Unexpected
453  * Sizes</a></h4>
454  *
455  * <p>Although the Btree normally occupies the vast majority of the cache, it
456  * is possible that record locks occupy unexpected amounts of cache when
457  * large transactions are used, or when cursors or transactions are left open
458  * due to application bugs. The following stat indicates the amount of cache
459  * used by record locks:</p>
460  * <ul>
461  *     <li>{@link #getLockBytes()}</li>
462  * </ul>
463  *
464  * <p>To reduce the amount of memory used for record locks:</p>
465  * <ul>
466  *     <li>Use a small number of write operations per transaction. Write
467  *     locks are held until the end of a transaction.</li>
468  *     <li>For transactions using Serializable isolation or RepeatableRead
469  *     isolation (the default), use a small number of read operations per
470  *     transaction.</li>
471  *     <li>To read large numbers of records, use {@link
472  *     LockMode#READ_COMMITTED} isolation or use a null Transaction (which
473  *     implies ReadCommitted). With ReadCommitted isolation, locks are
474  *     released after each read operation. Using {@link
475  *     LockMode#READ_UNCOMMITTED} will also avoid record locks, but does not
476  *     provide any transactional guarantees.</li>
477  *     <li>Ensure that all cursors and transactions are closed
478  *     promptly.</li>
479  * </ul>
480  *
481  * <p>Note that the above guidelines are also important for reducing contention
482  * when records are accessed concurrently from multiple threads and
483  * transactions. When using Oracle NoSQL DB, the application should avoid
484  * performing a large number of write operations in a single request. For read
485  * operations, NoSQL DB uses ReadCommitted isolation to avoid accumulation of
486  * locks.</p>
487  *
488  * <p>Another unexpected use of cache is possible when using a {@link
489  * DiskOrderedCursor} or when calling {@link Database#count()}. The amount of
490  * cache used by these operations is indicated by:</p>
491  * <ul>
492  *     <li>{@link #getDOSBytes()}</li>
493  * </ul>
494  *
495  * <p>{@code DiskOrderedCursor} and {@code Database.count} should normally be
496  * explicitly constrained to use a maximum amount of cache memory. See:</p>
497  * <ul>
498  *     <li>{@link DiskOrderedCursorConfig#setInternalMemoryLimit(long)}</li>
499  *     <li>{@link Database#count(long)}</li>
500  * </ul>
501  *
502  * <p>Oracle NoSQL DB does not currently use {@code DiskOrderedCursor} or
503  * {@code Database.count}.</p>
504  *
505  * <h4><a name="cacheEviction">Cache Statistics: Eviction</a></h4>
506  *
507  * <p>Eviction is removal of Btree node from the cache in order to make room
508  * for newly added nodes. See {@link CacheMode} for a description of
509  * eviction.</p>
510  *
511  * <p>Normally eviction is performed via background threads in the eviction
512  * thread pools. Disabling the eviction pool threads is not recommended.</p>
513  * <ul>
514  *     <li>{@link EnvironmentConfig#ENV_RUN_EVICTOR}</li>
515  *     <li>{@link EnvironmentConfig#ENV_RUN_OFFHEAP_EVICTOR}</li>
516  * </ul>
517  *
518  * <p>Eviction stats are important indicator of cache efficiency and provide a
519  * deeper understanding of cache behavior. Main cache eviction is indicated
520  * by:</p>
521  * <ul>
522  *     <li>{@link #getNLNsEvicted()}</li>
523  *     <li>{@link #getNNodesMutated()}</li>
524  *     <li>{@link #getNNodesEvicted()}</li>
525  *     <li>{@link #getNDirtyNodesEvicted()}</li>
526  * </ul>
527  *
528  * <p>Note that objects evicted from the main cache are moved to the off-heap
529  * cache whenever possible.</p>
530  *
531  * <p>Off-heap cache eviction is indicated by:</p>
532  * <ul>
533  *     <li>{@link #getOffHeapLNsEvicted()}</li>
534  *     <li>{@link #getOffHeapNodesMutated()}</li>
535  *     <li>{@link #getOffHeapNodesEvicted()}</li>
536  *     <li>{@link #getOffHeapDirtyNodesEvicted()}</li>
537  * </ul>
538  *
539  * <p>When analyzing Java GC performance, the most relevant stats are {@code
540  * NLNsEvicted}, {@code NNodesMutated} and {@code NNodesEvicted}, which all
541  * indicate eviction from the main cache based on LRU. Large values for these
542  * stats indicate that many old generation Java objects are being GC'd, which
543  * is often a cause of GC pauses.</p>
544  *
545  * <p>Note that {@link CacheMode#EVICT_LN} is used or when LNs are {@link
546  * EnvironmentConfig#TREE_MAX_EMBEDDED_LN embedded}, {@code NLNsEvicted} will
547  * be close to zero because LNs are not evicted based on LRU. And if an
548  * off-heap cache is configured, {@code NNodesMutated} will be close to zero
549  * because BIN mutation takes place in the off-heap cache. If any of the three
550  * values are large, this points to a potential GC performance problem. The GC
551  * logs should be consulted to confirm this.</p>
552  *
553  * <p>Large values for {@code NDirtyNodesEvicted} or {@code
554  * OffHeapDirtyNodesEvicted} indicate that the cache is severely undersized and
555  * there is a risk of using all available disk space and severe performance
556  * problems. Dirty nodes are evicted last (after evicting all non-dirty nodes)
557  * because they must be written to disk. This causes excessive writing and JE
558  * log cleaning may be unproductive.</p>
559  *
560  * <p>Note that when an off-heap cache is configured, {@code
561  * NDirtyNodesEvicted} will be zero because dirty nodes in the main cache are
562  * moved to the off-heap cache if they don't fit in the main cache, and are
563  * evicted completely and written to disk only when they don't fit in the
564  * off-heap cache.</p>
565  *
566  * <p>Another type of eviction tuning for the main cache involves changing the
567  * number of bytes evicted each time an evictor thread is awoken:</p>
568  * <ul>
569  *     <li>{@link EnvironmentConfig#EVICTOR_EVICT_BYTES}</li>
570  * </ul>
571  *
572  * <p>If the number of bytes is too large, it may cause a noticeable spike in
573  * eviction activity, reducing resources available to other threads. If the
574  * number of bytes is too small, the overhead of waking the evictor threads
575  * more often may be noticeable. The default values for this parameter is
576  * generally a good compromise. This parameter also impacts critical eviction,
577  * which is described next.</p>
578  *
579  * <p>Note that the corresponding parameter for the off-heap cache, {@link
580  * EnvironmentConfig#OFFHEAP_EVICT_BYTES}, works differently and is described
581  * in the next section.</p>
582  *
583  * <h4><a name="cacheCriticalEviction">Cache Statistics: Critical
584  * Eviction</a></h4>
585  *
586  * <p>The following stats indicate that critical eviction is occurring:</p>
587  * <ul>
588  *     <li>{@link #getNBytesEvictedCritical()}</li>
589  *     <li>{@link #getNBytesEvictedCacheMode()}</li>
590  *     <li>{@link #getNBytesEvictedDeamon()}</li>
591  *     <li>{@link #getNBytesEvictedEvictorThread()}</li>
592  *     <li>{@link #getNBytesEvictedManual()}</li>
593  *     <li>{@link #getOffHeapCriticalNodesTargeted()}</li>
594  *     <li>{@link #getOffHeapNodesTargeted()}</li>
595  * </ul>
596  *
597  * <p>Eviction is performed by eviction pool threads, calls to {@link
598  * Environment#evictMemory()} in application background threads, or via {@link
599  * CacheMode#EVICT_LN} or {@link CacheMode#EVICT_BIN}. If these mechanisms are
600  * not sufficient to evict memory from cache as quickly as CRUD operations are
601  * adding memory to cache, then critical eviction comes into play. Critical
602  * eviction is performed in-line in the thread performing the CRUD operation,
603  * which is very undesirable since it increases operation latency.</p>
604  *
605  * <p>Critical eviction in the main cache is indicated by large values for
606  * {@code NBytesEvictedCritical}, as compared to the other {@code
607  * NBytesEvictedXXX} stats. Critical eviction in the off-heap cache is
608  * indicated by large values for {@code OffHeapCriticalNodesTargeted} compared
609  * to {@code OffHeapNodesTargeted}.</p>
610  *
611  * <p>Additional stats indicating that background eviction threads may be
612  * insufficient are:</p>
613  * <ul>
614  *     <li>{@link #getNThreadUnavailable()}</li>
615  *     <li>{@link #getOffHeapThreadUnavailable()}</li>
616  * </ul>
617  *
618  * <p>Critical eviction can sometimes be reduced by changing {@link
619  * EnvironmentConfig#EVICTOR_CRITICAL_PERCENTAGE} or modifying the eviction
620  * thread pool parameters.</p>
621  * <ul>
622  *     <li>{@link EnvironmentConfig#EVICTOR_CRITICAL_PERCENTAGE}</li>
623  *     <li>{@link EnvironmentConfig#EVICTOR_CORE_THREADS}</li>
624  *     <li>{@link EnvironmentConfig#EVICTOR_MAX_THREADS}</li>
625  *     <li>{@link EnvironmentConfig#EVICTOR_KEEP_ALIVE}</li>
626  *     <li>{@link EnvironmentConfig#OFFHEAP_CORE_THREADS}</li>
627  *     <li>{@link EnvironmentConfig#OFFHEAP_MAX_THREADS}</li>
628  *     <li>{@link EnvironmentConfig#OFFHEAP_KEEP_ALIVE}</li>
629  * </ul>
630  *
631  * <p>When using Oracle NoSQL DB, {@code EVICTOR_CRITICAL_PERCENTAGE} is set to
632  * 20% rather than using the JE default of 0%.</p>
633  *
634  * <p>In the main cache, critical eviction uses the same parameter as
635  * background eviction for determining how many bytes to evict at one
636  * time:</p>
637  * <ul>
638  *     <li>{@link EnvironmentConfig#EVICTOR_EVICT_BYTES}</li>
639  * </ul>
640  *
641  * <p>Be careful when increasing this value, since this will cause longer
642  * operation latencies when critical eviction is occurring in the main
643  * cache.</p>
644  *
645  * <p>The corresponding parameter for the off-heap cache, {@code
646  * OFFHEAP_EVICT_BYTES}, works differently:</p>
647  * <ul>
648  *     <li>{@link EnvironmentConfig#OFFHEAP_EVICT_BYTES}</li>
649  * </ul>
650  *
651  * <p>Unlike in the main cache, {@code OFFHEAP_EVICT_BYTES} defines the goal
652  * for background eviction to be below {@code MAX_OFF_HEAP_MEMORY}. The
653  * background evictor threads for the off-heap cache attempt to maintain the
654  * size of the off-heap cache at {@code MAX_OFF_HEAP_MEMORY -
655  * OFFHEAP_EVICT_BYTES}. If the off-heap cache size grows larger than {@code
656  * MAX_OFF_HEAP_MEMORY}, critical off-heap eviction will occur. The default
657  * value for {@code OFFHEAP_EVICT_BYTES} is fairly large to ensure that
658  * critical eviction does not occur. Be careful when lowering this value.</p>
659  *
660  * <p>This approach is intended to prevent the off-heap cache from exceeding
661  * its maximum size. If the maximum is exceeded, there is a danger that the
662  * JVM process will be killed by the OS. See {@link
663  * #getOffHeapAllocFailures()}.</p>
664  *
665  * <h4><a name="cacheLRUListContention">Cache Statistics: LRU List
666  * Contention</a></h4>
667  *
668  * <p>Another common tuning issue involves thread contention on the cache LRU
669  * lists, although there is no stat to indicate such contention. Since each
670  * time a node is accessed it must be moved to the end of the LRU list, a
671  * single LRU list would cause contention among threads performing CRUD
672  * operations. By default there are 4 LRU lists for each cache. If contention
673  * is noticeable on internal Evictor.LRUList or OffHeapCache.LRUList methods,
674  * consider increasing the number of LRU lists:</p>
675  * <ul>
676  *     <li>{@link EnvironmentConfig#EVICTOR_N_LRU_LISTS}</li>
677  *     <li>{@link EnvironmentConfig#OFFHEAP_N_LRU_LISTS}</li>
678  * </ul>
679  *
680  * <p>However, note that increasing the number of LRU lists will decrease the
681  * accuracy of the LRU.</p>

1211      * CleanerStatDefinition#GROUP_NAME}
1212      * <br/>Name: {@value
1213      * CleanerStatDefinition#CLEANER_TWO_PASS_RUNS_NAME}</p>
1214      *
1215      * Number of cleaner two-pass runs, which are a subset of the
1216      * {@link #getNCleanerRuns() total cleaner runs}. The {@link
1217      * #getCurrentMinUtilization() minimum} and {@link
1218      * #getCurrentMaxUtilization() maximum} utilization values are used to
1219      * drive cleaning.
1220      * <p>
1221      * The obsolete portion of the log includes data that has expired. An
1222      * expiration histogram is stored for each file and used to compute the
1223      * expired size. The minimum and maximum utilization are the lower and
1224      * upper bounds of computed utilization. They are different only when the
1225      * TTL feature is used, and some data in the file has expired while other
1226      * data has become obsolete for other reasons, such as record updates,
1227      * record deletions or checkpoints. In this case the strictly obsolete size
1228      * and the expired size may overlap because they are maintained separately.
1229      * <p>

1231      * while if there is no overlap then the maximum utilization is correct.

1239      * equal. If the file should still be cleaned, based on the recomputed
1240      * utilization, it is cleaned as usual, and in this case the number of
1241      * two-pass runs (this statistic) is incremented.
1242      * <p>
1243      * If the file should not be cleaned because its recomputed utilization is
1244      * higher than expected, the file will not be cleaned. Instead, its
1245      * recomputed expiration histogram, which has size information that now
1246      * does not overlap with the strictly obsolete data, is stored for future
1247      * use. By storing the revised histogram, the cleaner can select the most
1248      * appropriate files for cleaning in the future. In this case the number of
1249      * {@link #getNCleanerRevisalRuns() revisal runs} is incremented, and the
1250      * number of {@link #getNCleanerRuns() total runs} is not incremented.

1836      * keys in the protected log size map are the names of the protecting
1837      * entities, and the values are the number of bytes protected by each
1838      * entity. The type and format of the entity names are as follows:</p>
1839      *
1840      * <pre>
1841      *    Backup-N
1842      *    DatabaseCount-N
1843      *    DiskOrderedCursor-N

1856      *         are not reserved files ond only appear in the map if they are

1858      *         are not protected if they have been copied and

1868      *         {@code Database.count}.
1869      *         {@code N} is a sequentially assigned integer.
1870      *     </li>

1880      *     </li>
1881      *     <li>

1883      *         a master and replica node in an HA replication group to
