23 /**
24  * DiskOrderedCursor returns records in unsorted order in exchange for
25  * generally faster retrieval times. Instead of key order, an approximation of
26  * disk order is used, which results in less I/O. This can be useful when the
27  * application needs to scan all records in one or more databases, and will be
28  * applying filtering logic which does not need key ordered retrieval.
29  * A DiskOrderedCursor is created using the {@link
30  * Database#openCursor(DiskOrderedCursorConfig)} method or the {@link
31  * Environment#openDiskOrderedCursor(Database[], DiskOrderedCursorConfig)}
32  * method.
33  * <p>
34  * <em>WARNING:</em> After opening a DiskOrderedCursor, deletion of log files
35  * by the JE log cleaner will be disabled until {@link #close()} is called.  To
36  * prevent unbounded growth of disk usage, be sure to call {@link #close()} to
37  * re-enable log file deletion.
38  * <p>
39  * <em>Optional configurations:</em> the following options are available to
40  * tune the DiskOrderedCursor.
41  * <p>
42  * The DiskOrderedCursor creates a background producer thread which prefetches
43  * some target records and inserts them in a queue for use by the cursor. The
44  * parameter {@link EnvironmentConfig#DOS_PRODUCER_QUEUE_TIMEOUT} applies to
45  * this background thread, and controls the timeout which governs the blocking
46  * queue.
47  * <p>
48  * See {@link DiskOrderedCursorConfig} for additional options.
49  * <p>
50  * <h3>Consistency Guarantees</h3>
51  * <p>
52  * The consistency guarantees provided by a DiskOrderedCursor are, at best, the
53  * same as those provided by READ_UNCOMMITTED (see {@link LockMode}).  With
54  * READ_UNCOMMITTED, changes made by all transactions, including uncommitted
55  * transactions, may be returned by the scan.  Also, a record returned by the
56  * scan is not locked, and may be modified or deleted by the application after
57  * it is returned, including modification or deletion of the record at the
58  * cursor position.
59  * <p>
60  * In other words, the records returned by the scan correspond to the state
61  * of the database (as if READ_UNCOMMITTED were used) at the beginning of the
62  * scan plus some, but not all, changes made by the application after the start
63  * of the scan.  The user should not rely on the scan returning any changes
64  * made after the start of the scan.  For example, if the record referred to by
65  * the DiskOrderedCursor is deleted after the DiskOrderedCursor is positioned
66  * at that record, getCurrent() will still return the key and value of that
67  * record and OperationStatus.SUCCESS.
68  * 
69  * If a transactionally correct data set is required (as defined by
70  * READ_COMMITTED), the application must ensure that all transactions that
71  * write to the database are committed before the beginning of the scan.
72  * During the scan, no records in the database of the scan may be
73  * inserted, deleted, or modified.  While this is possible, it is not the
74  * expected use case for a DiskOrderedCursor.
75  * <p>
76  * <h3>Performance Considerations</h3>
77  * <p>
78  * The internal algorithm used to approximate disk ordered reads is as follows.
79  * For simplicity, the algorithm description assumes that a single database is
80  * being scanned, but the algorithm is almost the same when multiple databases
81  * are involved.  
82  * An internal producer thread is used to scan the database. This thread is
83  * created and started when the {@code DiskOrderedCursor} is created, and is
84  * destroyed by {@link DiskOrderedCursor#close}. Scanning consists of two
85  * phases.  In phase I the in-cache Btree of the scanned database is traversed
86  * in key order.  The LSNs (physical record addresses) of the data to be
87  * fetched are accumulated in a memory buffer.  Btree latches are held during
88  * the traversal, but only for short durations.  In phase II the accumulated
89  * LSNs are sorted into disk order, fetched one at a time in that order, and
90  * the fetched data is added to a blocking queue.  The {@code getNext} method
91  * in this class removes the next entry from the queue.  This approach allows
92  * concurrent access to the Database during both phases of the scan, including
93  * access by the application's consumer thread (the thread calling {@code
94  * getNext}).
95  * <p>
96  * Phase I does not always process the entire Btree.  During phase I if the
97  * accumulation of LSNs causes the {@link
98  * DiskOrderedCursorConfig#setInternalMemoryLimit internal memory limit} or
99  * {@link DiskOrderedCursorConfig#setLSNBatchSize LSN batch size} to be
100  * exceeded, phase I is ended and phase II begins.  In this case, after phase
101  * II finishes, phase I resumes where it left off in the Btree traversal.
102  * Phase I and II are repeated until the entire database is scanned.
103  * <p>
104  * By default, the internal memory limit and LSN batch size are unbounded (see
105  * {@link DiskOrderedCursorConfig}).  For a database with a large number of
106  * records, this could cause an {@code OutOfMemoryError}.  Therefore, it is
107  * strongly recommended that either the internal memory limit or LSN batch size
108  * is configured to limit the use of memory during the scan.  On the other
109  * hand, the efficiency of the scan is proportional to the amount of memory
110  * used.  If enough memory is available, the ideal case would be that the
111  * database is scanned in in a single iteration of phase I and II.  The more
112  * iterations, the more random IO will occur.
113  * <p>
114  * Another factor is the {@link DiskOrderedCursorConfig#setQueueSize queue
115  * size}.  During the phase I Btree traversal, data that is resident in the JE
116  * cache will be added to the queue immediately, rather than waiting until
117  * phase II and fetching it, but only if the queue is not full.  Therefore,
118  * increasing the size of the queue can avoid fetching data that is resident in
119  * the JE cache.  Also, increasing the queue size can improve parallelism of
120  * the work done by the producer and consumer threads.
121  * <p>
122  * Also note that a {@link DiskOrderedCursorConfig#setKeysOnly keys-only} scan
123  * is much more efficient than the default keys-and-data scan.  With a
124  * keys-only scan, only the BINs (bottom internal nodes) of the Btree need to
125  * be fetched; the LNs (leaf nodes) do not.  This is also true of databases
126  * {@link DatabaseConfig#setSortedDuplicates configured for duplicates}, even
127  * for a keys-and-data scan, since internally the key and data are both
128  * contained in the BIN.
129  *
130  * @since 5.0
131  */

144     DiskOrderedCursor(
145         final Database[] dbHandles,
146         final DiskOrderedCursorConfig config) {
147 
148         this.dbHandles = dbHandles;
149         this.config = config;
150 
151         assert(dbHandles != null && dbHandles.length > 0);
152 
153         dbImpls = new DatabaseImpl[dbHandles.length];
154 
155         boolean dups = false;
156         int i = 0;
157 
158         try {
159             for (; i < dbHandles.length; ++i) {
160 
161                 Database db = dbHandles[i];
162                 DatabaseImpl dbImpl;
163 
164                 synchronized (db) {
165                     db.addCursor(this);
166                     dbImpl = db.getDbImpl();
167                 }
168 
169                 assert(dbImpl != null);
170 
171                 if (i == 0) {
172                     dups = dbImpl.getSortedDuplicates();
173 
174                 } else if (dbImpl.getSortedDuplicates() != dups) {
175                     throw new IllegalArgumentException(
176                         "In a multi-database disk ordered cursor " +
177                         "either all or none of the databases should support " +
178                         "duplicates");
179                 }
180 
181                 dbImpls[i] = dbImpl;
182             }
183 
184             dosCursorImpl = new DiskOrderedCursorImpl(dbImpls, config);
185 
186             this.logger = dbImpls[0].getEnv().getLogger();
187 
188         } catch (final Throwable e) {
189             for (int j = 0; j < i; ++j) {
190                 dbHandles[j].removeCursor(this);
191             }
192 
193             throw e;
194         }
195     }
