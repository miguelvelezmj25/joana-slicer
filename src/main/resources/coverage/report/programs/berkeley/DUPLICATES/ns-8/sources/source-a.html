475      * <p>Note that there are two ways to set the comparator: by specifying the

481      *
482      * <p>If a comparator needs to be initialized before it is used or needs

484      * {@link DatabaseComparator} interface.</p>
485      *
486      * <p>The Comparator.compare() method is passed the byte arrays that are
487      * stored in the database. If you know how your data is organized in the
488      * byte array, then you can write a comparison routine that directly
489      * examines the contents of the arrays. Otherwise, you have to reconstruct
490      * your original objects, and then perform the comparison.  See the <a
491      * href="{@docRoot}/../GettingStartedGuide/comparator.html"
492      * target="_top">Getting Started Guide</a> for examples.</p>
493      *
494      * <p><em>WARNING:</em> There are several special considerations that must
495      * be taken into account when implementing a comparator.<p>
496      * <ul>
497      *   <li>Comparator instances are shared by multiple threads and comparator
498      *   methods are called without any special synchronization. Therefore,
499      *   comparators must be thread safe.  In general no shared state should be
500      *   used and any caching of computed values must be done with proper
501      *   synchronization.</li>
502      *
503      *   <li>Because records are stored in the order determined by the
504      *   Comparator, the Comparator's behavior must not change over time and
505      *   therefore should not be dependent on any state that may change over
506      *   time.  In addition, although it is possible to change the comparator
507      *   for an existing database, care must be taken that the new comparator
508      *   provides compatible results with the previous comparator, or database
509      *   corruption will occur.</li>
510      *
511      *   <li>JE uses comparators internally in a wide variety of circumstances,
512      *   so custom comparators must be sure to return valid values for any two
513      *   arbitrary keys.  The user must not make any assumptions about the
514      *   range of key values that might be compared. For example, it's possible
515      *   for the comparator may be used against previously deleted values.</li>
516      * </ul>
517      *
518      * <p>A special type of comparator is a <em>partial comparator</em>, which
519      * allows for the keys of a database to be updated, but only if the updates
520      * do not change the relative order of the keys. For example, if a database
521      * uses strings as keys and a case-insensitive comparator, it is possible
522      * to change the case of characters in the keys, as this will not change
523      * the ordering of the keys. Another example is when the keys contain
524      * multiple fields but uniquely identify each record with a single field.
525      * The partial comparator could then compare only the single identifying
526      * field, allowing the rest of the fields to be updated. A query
527      * ({@link Cursor#getSearchKey Cursor.getSearchKey}, for example) could
528      * then be performed by passing a partial key that contains only the
529      * identifying field.
530      *
531      * <p><strong>WARNING:</strong> To allow for key updates in situations
532      * like those described above, all partial comparators must implement the
533      * {@link PartialComparator} tag interface. Otherwise, BDB JE will raise
534      * an exception if an attempt is made to update a key in a database whose
535      * comparators do not implement PartialComparator. See "Upgrading from JE

538      * <p>
