107 /**
108  *
109  * Overview
110  * --------
111  *
112  * The Evictor is responsible for managing the JE cache. The cache is
113  * actually a collection of in-memory btree nodes, implemented by the
114  * com.sleepycat.je.dbi.INList class. A subset of the nodes in te INList
115  * are candidates for eviction. This subset is tracked in one or more
116  * LRULists, which are maintained by the Evictor. When a node is evicted,
117  * it is detached from its containing BTree and then removed from the INList
118  * and from its containing LRUList. Once all references to an evicted node
119  * are removed, it can be GC'd by the JVM.
120  *
121  * The Evictor owns a pool of threads that are available to handle eviction
122  * tasks. The eviction pool is a standard java.util.concurrent thread pool,
123  * and can be mutably configured in terms of core threads, max threads, and
124  * keepalive times.
125  *
126  * Eviction is carried out by three types of threads:
127  * 1. An application thread, in the course of doing critical eviction.
128  * 2. Daemon threads, such as the cleaner or INCompressor, in the course of
129  *    doing their respective duties.
130  * 3. Eviction pool threads.
131  *
132  * Memory consumption is tracked by the MemoryBudget. The Arbiter, which is
133  * also owned by the Evictor, is used to query the MemoryBudget and determine
134  * whether eviction is actually needed, and if so, how many bytes should be
135  * evicted by an evicting thread.
136  *
137  * Multiple threads can do eviction concurrently. As a result, it's important
138  * that eviction is both thread safe and as parallel as possible.  Memory
139  * thresholds are generally accounted for in an unsynchronized fashion, and are
140  * seen as advisory. The only point of true synchronization is around the
141  * selection of a node for eviction. The act of eviction itself can be done
142  * concurrently.
143  *
144  * The eviction method is not reentrant, and a simple concurrent hash map
145  * of threads is used to prevent recursive calls.
146  *
147  * Details on the implementation of the LRU-based eviction policy
148  * --------------------------------------------------------------
149  *
150  * ------------------
151  * Data structures
152  * ------------------
153  *
154  * An LRU eviction policy is approximated by one or more LRULists. An LRUList
155  * is a doubly linked list consisting of BTree nodes. If a node participates
156  * in an LRUList, then whenever it is accessed, it moves to the "back" of the
157  * list. When eviction is needed, the evictor evicts the nodes at the "front"
158  * of the LRULists.
159  *
160  * An LRUList is implemented as 2 IN references: a "front" ref pointing to the
161  * IN at the front of the list and a "back" ref, pointing to the IN at the back
162  * of the list. In addition, each IN has "nextLRUNode" and "prevLRUNode" refs
163  * for participating in an LRUList. This implementation works because an IN can
164  * belong to at most 1 LRUList at a time. Furthermore, it is the responsibility
165  * of the Evictor to know which LRUList a node belongs to at any given time
166  * (more on this below). As a result, each LRUList can assume that a node will
167  * either not be in any list at all, or will belong to "this" list. This way,
168  * membership of a node to an LRUList can be tested by just checking that
169  * either the nextLRUNode or prevLRUNode field of the node is non-null.
170  *
171  * The operations on an LRUList are:
172  *
173  * - addBack(IN) : 
174  * Insert an IN at the back of the list. Assert that the node does not belong
175  * to an LRUList already.
176  *
177  * - addFront(IN) : 
178  * Insert an IN at the front of the list.  Assert that the node does not belong
179  * to an LRUList already.
180  *
181  * - moveBack(IN) :
182  * Move an IN to the back of the list, if it is in the list already. Noop
183  * if the node is not in the list.
184  *
185  * - moveFront(IN) : 
186  * Move an IN to the front of the list, if it is in the list already. Noop
187  * if the node is not in the list.
188  *
189  * - removeFront() : 
190  * Remove the IN at the front of the list and return it to the caller. 
191  * Return null if the list is empty.
192  *
193  * - remove(IN) : 
194  * Remove the IN from the list, if it is there. Return true if the node was
195  * in the list, false otherwise.
196  *
197  * - contains(IN):
198  * Return true if the node is contained in the list, false otherwise.
199  *
200  * All of the above methods are synchronized on the LRUList object. This may
201  * create a synchronization bottleneck. To alleviate this, the Evictor uses
202  * multiple LRULists, which taken together comprise a logical LRU list, called
203  * an LRUSet. The number of LRULists per LRUSet (numLRULists) is fixed and
204  * determined by a config parameter (max of 64). The LRULists are stored in
205  * an array whose length is numLRULists.
206  *
207  * The Evictor actually maintains 2 LRUSets: priority-1 and priority-2.
208  * Within an LRUSet, the nodeId is used to place a node to an LRUList: a
209  * node with id N goes to the (N % numLRULists)-th list. In addition, each
210  * node has a flag (isInPri2LRU) to identify which LRUSet it belongs to.
211  * This way, the Evictor knows which LRUList a node should belong to, and 
212  * accesses the appropriate LRUList instance when it needs to add/remove/move
213  * a node within the LRU.
214  *
215  * Access to the isInPri2LRU flag is synchronized via the SH/EX node latch.
216  *
217  * When there is no off-heap cache configured, the priority-1 LRU is the
218  * "mixed" one and the priority-2 LRU is the "dirty" one. When there is an
219  * off-heap cache configured, the priority-1 LRU is the "normal" one and the
220  * priority-2 LRU is the "level-2" one.
221  *
222  * Justification for the mixed and dirty LRUSets: We would like to keep dirty
223  * INs in memory as much as possible to achieve "write absorption". Ideally,
224  * dirty INs should be logged by the checkpointer only. So, we would like to
225  * have the option in the Evictor to chose a clean IN to evict over a dirty
226  * IN, even if the dirty IN is colder than the clean IN. In this mode, having
227  * a single LRUSet will not perform very well in the situation when most (or
228  * a lot) or the INs are dirty (because each time we get a dirty IN from an
229  * LRUList, we would have to put it back to the list and try another IN until
230  * we find a clean one, thus spending a lot of CPU time trying to select an
231  * eviction target).
232  *
233  * Justification for the normal and level-2 LRUSets: With an off-heap cache,
234  * if level-2 INs were not treated specially, the main cache evictor may run
235  * out of space and (according to LRU) evict a level 2 IN, even though the IN
236  * references off-heap BINs (which will also be evicted). The problem is that
237  * we really don't want to evict the off-heap BINs (or their LNs) when the
238  * off-heap cache is not full. Therefore we only evict level-2 INs with
239  * off-heap children when there are no other nodes that can be evicted. A
240  * level-2 IN is moved to the priority-2 LRUSet when it is encountered by the
241  * evictor in the priority-1 LRUSet.
242  *
243  * Within each LRUSet, picking an LRUList to evict from is done in a round-
244  * robin fashion. To this end, the Evictor maintains 2 int counters:
245  * nextPri1LRUList and nextPri2LRUList. To evict from the priority-1 LRUSet, an
246  * evicting thread picks the (nextPri1LRUList % numLRULists)-th list, and
247  * then increments nextPri1LRUList. Similarly, to evict from the priority-2
248  * LRUSet, an evicting thread picks the (nextPri2LRUList % numLRULists)-th
249  * list, and then increments nextPri2LRUList. This does not have to be done in
250  * a synchronized way.
251  *
252  * A new flag (called hasCachedChildren) is added to each IN to indicate
253  * whether the IN has cached children or not. This flag is used and maintained
254  * for upper INs (UINs) only. The need for this flag is explained below.
255  * Access to this flag is synchronized via the SH/EX node latch.
256  *
257  * ---------------------------------------------------------------------------
258  * LRUSet management: adding/removing/moving INs in/out of/within the LRUSets
259  * ---------------------------------------------------------------------------
260  *
261  * We don't want to track upper IN (UIN) nodes that have cached children.
262  * There are 2 reasons for this: (a) we cannot evict UINs with cached children
263  * (the children must be evicted first) and (b) UINs will normally have high
264  * access rate, and would add a lot of CPU overhead if they were tracked. 
265  *
266  * The hasCachedChildren flag is used as a quick way to determine whether a
267  * UIN has cached children or not.
268  *
269  * Adding a node to the LRU.
270  * -------------------------
271  *
272  * A IN N is added in an LRUSet via one of the following Evictor methods:
273  * addBack(IN), addFront(IN), pri2AddBack(IN), or pri2AddFront(IN). The
274  * first 2 add the node to the priority-1 LRUSet and set its isInPri2LRU flag
275  * to false. The last 2 add the node to the priority-2 LRUSet and set its
276  * isInPri2LRU flag to true.
277  *
278  * Note: DINs and DBINs are never added to the LRU.
279  *
280  * A node N is added to the LRU in the following situations:
281  *
282  * 1. N is fetched into memory from the log. Evictor.addBack(N) is called
283  *    inside IN.postfetchInit() (just before N is connected to its parent).
284  *
285  * 2. N is a brand new node created during a split, and either N is a BIN or
286  *    N does not get any cached children from its split sibling.
287  *    Evictor.addFront(N) is called if N is a BIN and the cachemode is
288  *    MAKE_COLD or EVICT_BIN. Otherwise, Evictor.addBack(child) is called.
289  *
290  * 3. N is a UIN that is being split, and before the split it had cached
291  *    children, but all its cached children have now moved to its newly
292  *    created sibling. Evictor.addBack(N) is called in this case.
293  *
294  * 4. N is a UIN that looses its last cached child (either because the child is
295  *    evicted or it is deleted). Evictor.addBack(N) is called inside
296  *    IN.setTarget(), if the target is null, N is a UIN, N's hasCachedChildren
297  *    flag is true, and N after setting the target to null, N has no remaining
298  *    cached children.
299  *
300  * 5. N is the 1st BIN in a brand new tree. In this case, Evictor.addBack(N)
301  *    is called inside Tree.findBinForInsert().
302  *
303  * 6. N is a node visited during IN.rebuildINList() and N is either a BIN or
304  *    a UIN with no cached children.
305  *
306  * 7. An evicting thread T removes N from the LRU, but after T EX-latches N,
307  *    it determines that N is not evictable or should not be evicted, and
308  *    should be put back in the LRU. T puts N back to the LRU using one of
309  *    the above 4 methods (for details, read about the eviction processing
310  *    below), but ONLY IF (a) N is still in the INList, and (b) N is not in
311  *    the LRU already.
312  *
313  *    Case (b) can happen if N is a UIN and after T removed N from the LRU
314  *    but before T could latch N, another thread T1 added a child to N and
315  *    removed that child. Thus, by item 4 above, T1 adds N back to the LRU.
316  *    Furthermore, since N is now back in the LRU, case (a) can now happen
317  *    as well if another thread can evict N before T latches it.
318  *
319  * 8. When the checkpointer (or any other thread/operation) cleans a dirty IN,
320  *    it must move it from the priority-2 LRUSet (if there) to the priority-1
321  *    one. This is done via the Evictor.moveToPri1LRU(N) method: If the
322  *    isInPri2LRU flag of N is true, LRUList.remove(N) is called to remove
323  *    the node from the priority-2 LRUSet. If N was indeed in the priority-2
324  *    LRUSet (i.e., LRUList.remove() returns true), addBack(N) is called to
325  *    put it in the priority-1 LRUSet.
326  *
327  *    By moving N to the priority-1 LRUSet only after atomically removing it
328  *    from the priority-2 LRUSet and checking that it was indeed there, we
329  *    prevent N from being added into the LRU if N has been or would be removed
330  *    from the LRU by a concurrently running evicting thread.
331  *    
332  * In cases 2, 3, 4, 5, 7, and 8 N is EX-latched. In case 1, the node is not
333  * latched, but it is inaccessible by any other threads because it is not
334  * connected to its parent yet and the parent is EX-latched (but N has already
335  * been inserted in the INList; can this create any problems ?????). In case
336  * 6 there is only one thread running. So, in all cases it's ok to set the
337  * isInPri2LRU flag of the node.
338  *
339  * Question: can a thread T try to add a node N, seen as a Java obj instance,
340  * into the LRU, while N is already there? I believe not, and LRUList addBack()
341  * and addFront() methods assert that this cannot happen. In cases 1, 2, and 5
342  * above N is newly created node, so it cannot be in the LRU already. In cases
343  * 3 and 4, N is a UIN that has cached children, so it cannot be in the LRU.
344  * In case 6 there is only 1 thread. Finally, in cases 7 and 8, T checks that
345  * N is not in the LRU before attempting to add it (and the situation cannot
346  * change between tis check and the insertion into the LRU because N is EX-
347  * latched).
348  *
349  * Question: can a thread T try to add a node N, seen as a logical entity
350  * represented by its nodeId, into the LRU, while N is already there?
351  * Specifically, (a) can two Java instances, N1 and N2, of the same node
352  * N exist in memory at the same time, and (b) while N1 is in the LRU, can
353  * a thread T try to add N2 in the LRU? The answer to (a) is "yes", and as
354  * far as I can think, the answer to (b) is "no", but there is no explicit
355  * check in the code for this. Consider the following sequence of events:
356  * Initially only N1 is in memory and in the LRU. An evicting thread T1
357  * removes N1 from the LRU, thread T2 adds N1 in the LRU, thread T3 removes
358  * N1 from the LRU and actually evicts it, thread T4 fetches N from the log,
359  * thus creating instance N2 and adding N2 to the LRU, thread T1 finally
360  * EX-latches N1 and has to decide what to do with it. The check in case
361  * 7a above makes sure that N1 will not go back to the LRU. In fact the
362  * same check makes sure that N1 will not be evicted (i.e., logged, if
363  * dirty). T1 will just skip N1, thus allowing it to be GCed.
364  *
365  * Removing a node from the LRU
366  * ----------------------------
367  *
368  * A node is removed from the LRU when it is selected as an eviction target
369  * by an evicting thread. The thread chooses an LRUList list to evict from
370  * and calls removeFront() on it. The node is not latched when it is removed
371  * from the LRU in this case. The evicting thread is going to EX-latch the
372  * node shortly after the removal. But as explain already earlier, between
373  * the removal and the latching, another thread may put the node back to the
374  * LRU, and as a result, another thread may also choose the same node for
375  * eviction. The node may also be detached from the BTree, or its database
376  * closed, or deleted.
377  * 
378  * A node may also be removing from the LRU by a non-evicting thread. This
379  * is done via the Evictor.remove(IN) method. The method checks the node's
380  * isInDrtryLRU flag to determine which LRUSet the node belongs to (if any)
381  * and then calls LRUList.remove(N). The node must be at least SH latched
382  * when the method is called. The method is a noop if the node is not in the
383  * LRU. The node may not belong to any LRUList, because it has been selected
384  * for eviction by another thread (and thus removed from LRU), but the
385  * evicting thread has not yet latched the node. There are 3 cases (listed
386  * below) where Evictor.remove(N) is called. In the first two cases
387  * Evictor.remove(N) is invoked from INList.removeInternal(N). This makes
388  * sure that N is removed from the LRU whenever it it removed from the
389  * INList (to guarantee that the nodes in the LRU are always a subset of
390  * the nodes in the INList).
391  *
392  * 1. When a tree branch containing N gets detached from its tree. In this
393  *    case, INList.remove(N) is invoked inside accountForSubtreeRemoval() or
394  *    accountForDeferredWriteSubtreeRemoval().
395  * 
396  * 2. When the database containing N gets deleted or truncated. In this case,
397  *    INList.iter.remove() is called in DatabaseImpl.finishDeleteProcessing().
398  *
399  * 3. N is a UIN with no cached children (hasCachedChildren flag is false)
400  *    and a new child for N is fetched. The call to Evictor.remove(N) is
401  *    done inside IN.setTarget().
402  *
403  * Moving a node within the LRU
404  * ----------------------------
405  *
406  * A node N is moved within its containing LRUList (if any) via the Evictor
407  * moveBack(IN) and moveFront(IN) methods. The methods check the isInPri2LRU
408  * flag of the node to determine the LRUSet the node belongs to and then move
409  * the node to the back or to the front of the LRUList. The node will be at
410  * least SH latched when these methods are called. Normally, the IN will be
411  * in an LRUList. However, it may not belong to any LRUList, because it has
412  * been selected for eviction by another thread (and thus removed from LRU),
413  * but the evicting thread has not yet EX-latched the node. In this case,
414  * these methods are is a noop. The methods are called in the following
415  * situations:
416  *
417  * 1. N is latched with cachemode DEFAULT, KEEP_HOT, or EVICT_LN and N is a
418  *    BIN or a UIN with no cached children (the hasCachedChildren flag is
419  *    used to check if the UIN has cached children, so we don't need to
420  *    iterate over all of the node's child entries). In this case,
421  *    Evictor.moveBack(N) .
422  *
423  * 2. N is latched with cachemode MAKE_COLD or EVICT_BIN and N is a BIN.
424  *    In this case, Evictor.moveFront(N) is called.
425  * 
426  * -------------------
427  * Eviction Processing
428  * -------------------
429  *
430  * A thread can initiate eviction by invoking the Evictor.doEviction() method.
431  * This method implements an "eviction run". An eviction run consists of a
432  * number of "eviction passes", where each pass is given as input a maximum
433  * number of bytes to evict. An eviction pass is implemented by the
434  * Evictor.evictBatch() method. 
435  *
436  * Inside Evictor.evictBatch(), an evicting thread T:
437  * 
438  * 1. Picks the priority-1 LRUset initially as the "current" LRUSet to be
439  *    processed,
440  *
441  * 2. Initializes the max number of nodes to be processed per LRUSet to the
442  *    current size of the priority-1 LRUSet,
443  *
444  * 3. Executes the following loop:
445  *
446  * 3.1. Picks a non-empty LRUList from the current LRUSet in a round-robin
447  *      fashion, as explained earlier, and invokes LRUList.removeFront() to
448  *      remove the node N at the front of the list. N becomes the current
449  *      eviction target.
450  *
451  * 3.2. If the DB node N belongs to has been deleted or closed, skips this node,
452  *      i.e., leaves N outside the LRU and goes to 3.4.
453  *
454  * 3.3. Calls ProcessTarget(N) (see below)
455  *
456  * 3.4. If the current LRUset is the priority-1 one and the number of target nodes
457  *      processed reaches the max number allowed, the priority-2 LRUSet becomes
458  *      the current one, the max number of nodes to be processed per LRUSet is
459  *      set to the current size of the priority-2 LRUSet, and the number of
460  *      nodes processed is reset to 0.
461  *
462  * 3.5. Breaks the loop if the max number of bytes to evict during this pass
463  *      has been reached, or memConsumption is less than (maxMemory - M) (where
464  *      M is a config param), or the number of nodes that have been processed
465  *      in the current LRUSet reaches the max allowed.
466  *
467  * --------------------------
468  * The processTarget() method
469  * --------------------------
470  *
471  * This method is called after a node N has been selected for eviction (and as
472  * result, removed from the LRU). The method EX-latches N and determines
473  * whether it can/should really be evicted, and if not what is the appropriate
474  * action to be taken by the evicting thread. Before returning, the method
475  * unlatches N. Finally, it returns the number of bytes evicted (if any).
476  *
477  * If a decision is taken to evict N or mutate it to a BINDelta, N must first
478  * be unlatched and its parent must be searched within the tree. During this
479  * search, many things can happen to the unlatched N, and as a result, after
480  * the parent is found and the N is relatched, processTarget() calls itself
481  * recursively to re-consider all the possible actions for N.
482  *
483  * Let T be an evicting thread running processTarget() to determine what to do
484  * with a target node N. The following is the list of possible outcomes:
485  *
486  * 1. SKIP - Do nothing with N if:
487  *    (a) N is in the LRU. This can happen if N is a UIN and while it is
488  *        unlatched by T, other threads fetch one or more of N's children,
489  *        but then all of N's children are removed again, thus causing N to
490  *        be put back to the LRU.
491  *    (b) N is not in the INList. Given than N can be put back to the LRU while
492  *        it is unlatched by T, it can also be selected as an eviction target
493  *        by another thread and actually be evicted.
494  *    (c) N is a UIN with cached children. N could have acquired children
495  *        after the evicting thread removed it from the LRU, but before the
496  *        evicting thread could EX-latch it.
497  *    (d) N is the root of the DB naming tree or the DBmapping tree. 
498  *    (e) N is dirty, but the DB is read-only.
499  *    (f) N's environment used a shared cache and the environment has been
500  *        closed or invalidated.
501  *    (g) If a decision was taken to evict od mutate N, but the tree search
502  *        (using N's keyId) to find N's parent, failed to find the parent, or
503  *        N itself. This can happen if during the search, N was evicted by
504  *        another thread, or a branch containing N was completely removed
505  *        from the tree.
506  *
507  * 2. PUT BACK - Put N to the back of the LRUSet it last belonged to, if:
508  *    (a) It is a BIN that was last accessed with KEEP_HOT cache mode.
509  *    (b) N has an entry with a NULL LSN and a null target.
510  *
511  * 3. PARTIAL EVICT - perform partial eviction on N, if none of the cases
512  *    listed above is true. Currently, partial eviction applies to BINs only
513  *    and involves the eviction (stripping) of evictable LNs. If a cached LN
514  *    is not  evictable, the whole BIN is not evictable as well. Currently,
515  *    only MapLNs may be non-evictable (see MapLN.isEvictable()).
516  *
517  *    After partial eviction is performed the following outcomes are possible:
518  *
519  * 4. STRIPPED PUT BACK - Put N to the back of the LRUSet it last belonged to,
520  *    if partial eviction did evict any bytes, and N is not a BIN in EVICT_BIN
521  *    or MAKE_COLD cache mode.
522  *
523  * 5. PUT BACK - Put N to the back of the LRUSet it last belonged to, if
524  *    no bytes were stripped, but partial eviction determined that N is not
525  *    evictable.
526  *
527  * 6. MUTATE - Mutate N to a BINDelta, if none of the above apply and N is a
528  *    BIN that can be mutated.
529  *
530  * 7. MOVE DIRTY TO PRI-2 LRU - Move N to the front of the priority-2 LRUSet,
531  *    if none of the above apply and N is a dirty node that last belonged to
532  *    the priority-1 LRUSet, and a dirty LRUSet is used (meaning that no
533  *    off-heap cache is configured).
534  *
535  * 8. MOVE LEVEL-2 TO PRI-2 LRU - Move N to the front of the priority-2 LRUSet,
536  *    if none of the above apply and N is a level-2 node with off-heap BINs
537  *    that last belonged to the priority-1 LRUSet.
538  *
539  * 9. EVICT - Evict N is none of the above apply.
540  *
541  * -------
542  * TODO:
543  * -------

547  *

979                 assert(node.getPrevLRUNode().getNextLRUNode() == node);
980

983                 assert(size > 1);
984

988                     assert(node.getNextLRUNode().getPrevLRUNode() == node);

992                 } else {

994                     assert(front != node && back != node);
995                     assert(node.getPrevLRUNode().getNextLRUNode() == node);

1006                 back = node;
1007             }
