23 /**
24  * The abstract class that defines the various formats used to represent
25  * the keys associated with the IN node. The class is also used to store
26  * embedded records, where the actual key and the data portion of a record are
27  * stored together as a single byte sequence.
28  * 
29  * There are currently two supported representations:
30  * <ol>
31  * <li>A default representation <code>Default</code> that's capable of holding
32  * any set of keys.</li>
33  * <li>
34  * A compact representation <code>MaxKeySize</code> that's more efficient for
35  * holding small keys (LTE 16 bytes) in length. If key prefixing is in use this
36  * represents the unprefixed part of the key, since that's what is stored in
37  * this array.</li>
38  * </ol>
39  * <p>
40  * The choice of representation is made when an IN node is first read in from
41  * the log. The <code>MaxKeySize</code> representation is only used when it is
42  * more storage efficient than the default representation for the set of keys
43  * currently associated with the IN.
44  * <p>
45  * Note that no attempt is currently made to optimize the storage
46  * representation as keys are added to, or removed from, the
47  * <code>Default</code> representation to minimize the chances of transitionary
48  * "back and forth" representation changes that could prove to be expensive.
49  */

291 
292             if (keys.length > MaxKeySize.MAX_KEYS) {
293                 return this;
294             }
295 
296             final int compactMaxKeyLength = parent.getCompactMaxKeyLength();
297             if (compactMaxKeyLength <= 0) {
298                 return this;
299             }
300 
301             int keyCount = 0;
302             int maxKeyLength = 0;
303             int defaultKeyBytes = 0;
304 
305             for (byte[] key : keys) {
306                 if (key != null) {
307                     keyCount++;
308                     if (key.length > maxKeyLength) {
309                         maxKeyLength = key.length;
310                         if (maxKeyLength > compactMaxKeyLength) {
311                             return this;
312                         }
313                     }
314                     defaultKeyBytes += MemoryBudget.byteArraySize(key.length);
315                 }
316             }
317 
318             if (keyCount == 0) {
319                 return this;
320             }
321

323 
324             if (defaultSizeWithKeys >
325                 MaxKeySize.calculateMemorySize(keys.length, maxKeyLength)) {
326                 return compactToMaxKeySizeRep(maxKeyLength, parent);

373      * The compact representation that can be used to represent keys LTE 16
374      * bytes in length. The keys are all represented inside a single byte array
375      * instead of having one byte array per key. Within the array, all keys are
376      * assigned a storage size equal to that taken up by the longest key, plus
377      * one byte to hold the actual key length. This makes key retreival fast.
378      * However, insertion and deletion for larger keys moves bytes proportional
379      * to the storage length of the keys. This is why the representation is
380      * restricted to keys LTE 16 bytes in size.
381      *
382      * On a 32 bit VM the per key overhead for the Default representation is 4
383      * bytes for the pointer + 16 bytes for each byte array key object, for a
384      * total of 20 bytes/key. On a 64 bit machine the overheads are much
385      * larger: 8 bytes for the pointer plus 24 bytes per key.
386      *
387      * The more fully populated the IN the more the savings with this
388      * representation since the single byte array is sized to hold all the keys
389      * regardless of the actual number of keys that are present.
390      *
391      * It's worth noting that the storage savings here are realized in addition
392      * to the storage benefits of key prefixing, since the keys stored in the
393      * key array are the smaller key values after the prefix has been stripped,
394      * reducing the length of the key and making it more likely that it's small
395      * enough for this specialized representation.
396      */
