62 /**
63  * DbTree represents the database directory for this environment. DbTree is
64  * itself implemented through two databases. The nameDatabase maps
65  * databaseName-> an internal databaseId. The idDatabase maps
66  * databaseId->DatabaseImpl.
67  *
68  * For example, suppose we have two databases, foo and bar. We have the
69  * following structure:
70  *
71  *           nameDatabase                          idDatabase
72  *               IN                                    IN
73  *                |                                     |
74  *               BIN                                   BIN
75  *    +-------------+--------+            +---------------+--------+
76  *  .               |        |            .               |        |
77  * NameLNs         NameLN    NameLN      MapLNs for   MapLN        MapLN
78  * for internal    key=bar   key=foo     internal dbs key=53       key=79
79  * dbs             data=     data=                    data=        data=
80  *                 dbId79    dbId53                   DatabaseImpl DatabaseImpl
81  *                                                        |            |
82  *                                                   Tree for foo  Tree for bar
83  *                                                        |            |
84  *                                                     root IN       root IN
85  *
86  * Databases, Cursors, the cleaner, compressor, and other entities have
87  * references to DatabaseImpls. It's important that object identity is properly
88  * maintained, and that all constituents reference the same DatabaseImpl for
89  * the same db, lest they develop disparate views of the in-memory database;
90  * corruption would ensue. To ensure that, all entities must obtain their
91  * DatabaseImpl by going through the idDatabase.
92  *
93  * DDL type operations such as create, rename, remove and truncate get their
94  * transactional semantics by transactionally locking the NameLN appropriately.
95  * A read-lock on the NameLN, called a handle lock, is maintained for all DBs
96  * opened via the public API (openDatabase).  This prevents them from being
97  * renamed or removed while open.  See HandleLocker for details.
98  *
99  * However, for internal database operations, no handle lock on the NameLN is
100  * acquired and MapLNs are locked with short-lived non-transactional Lockers.
101  * An entity that is trying to get a reference to the DatabaseImpl gets a short
102  * lived read lock just for the fetch of the MapLN, and a DatabaseImpl usage
103  * count is incremented to prevent eviction; see getDb and releaseDb. A write
104  * lock on the MapLN is taken when the database is created, deleted, or when
105  * the MapLN is evicted. (see DatabaseImpl.isInUse())
106  *
107  * The nameDatabase operates pretty much as a regular application database in
108  * terms of eviction and recovery. The idDatabase requires special treatment
109  * for both eviction and recovery.
110  *
111  * The issues around eviction of the idDatabase center on the need to ensure
112  * that there are no other current references to the DatabaseImpl other than
113  * that held by the mapLN. The presence of a current reference would both make
114  * the DatabaseImpl not GC'able, and more importantly, would lead to object
115  * identity confusion later on. For example, if the MapLN is evicted while
116  * there is a current reference to its DatabaseImpl, and then refetched, there
117  * will be two in-memory versions of the DatabaseImpl. Since locks on the
118  * idDatabase are short lived, DatabaseImpl.useCount acts as a reference count
119  * of active current references. DatabaseImpl.useCount must be modified and
120  * read in conjunction with appropriate locking on the MapLN. See
121  * DatabaseImpl.isInUse() for details.
122  *
123  * This reference count checking is only needed when the entire MapLN is
124  * evicted. It's possible to evict only the root IN of the database in
125  * question, since that doesn't interfere with the DatabaseImpl object
126  * identity.
127  *
128  * Another dependency on usage counts was introduced to prevent MapLN deletion
129  * during cleaner and checkpointer operations that are processing entries for a
130  * DB.  (Without usage counts, this problem would have occurred even if DB
131  * eviction were never implemented.)  When the usage count is non-zero it
132  * prohibits deleteMapLN from running. The deleted state of the MapLN must not
133  * change during a reader operation (operation by a thread that has called
134  * getDb and not yet called releaseDb).
135  *
136  * Why not just hold a MapLN read lock during a reader operation?
137  * --------------------------------------------------------------
138  * Originally this was not done because of cleaner performance.  We afraid that  * either of the following solutions would not perform well:
139  *   + If we get (and release) a MapLN lock for every entry in a log file, this
140  *     adds a lot of per-entry overhead.
141  *   + If we hold the MapLN read lock for the duration of a log file cleaning
142  *     (the assumption is that many entries are for the same DB), then we block
143  *     checkpoints during that period, when they call modifyDbRoot.
144  * Therefore, the usage count is incremented once per DB encountered during log
145  * cleaning, and the count is decremented at the end.  This caching approach is
146  * also used by the HA replayer.  In both cases, we do not want to lock the
147  * MapLN every entry/operation, and we do not want to block checkpoints or
148  * other callers of modifyDbRoot.  It is acceptable, however, to block DB
149  * naming operations.
150  *
151  * In addition we allow modifyDbRoot to run when even the usage count is
152  * non-zero, which would not be possible using a read-write locking strategy.
153  * I'm not sure why this was done originally, perhaps to avoid blocking.  But
154  * currently, it is necessary to prevent a self-deadlock.  All callers of
155  * modifyDbRoot first call getDb, which increments the usage count.  So if
156  * modifyDbRoot was to check the usage count and retry if non-zero (like
157  * deleteMapLN), then it would loop forever.
158  *
159  * Why are the retry loops necessary in the DbTree methods?
160  * --------------------------------------------------------
161  * Three methods that access the MapLN perform retries (forever) when there is
162  * a lock conflict: getDb, modifyDbRoot and deleteMapLN.  Initially the retry
163  * loops were added to compensate for certain slow operations. To solve that
164  * problem, perhaps there are alternative solutions (increasing the lock
165  * timeout).  However, the deleteMapLN retry loop is necessary to avoid
166  * deleting it when the DB is in use by reader operations.
167  *
168  * Tendency to livelock
169  * --------------------
170  * Because MapLN locks are short lived, but a reader operation may hold a
171  * MapLN/DatabaseImpl for a longer period by incrementing the usage count,
172  * there is the possibility of livelock.  One strategy for avoiding livelock is
173  * to avoid algorithms where multiple threads continuously call getDb and
174  * releaseDb, since this could prevent completion of deleteMapLN.  [#20816]
175  */

764      *

766      * exist; if false, silently do nothing.
767      */

769         DatabaseImpl db,
770         long ifBeforeLsn,
771         boolean mustExist)
772         throws DatabaseException {

775          * Do not write LNs in read-only env.  This method is called when
776          * recovery causes a root split. [#21493]
777          */
778         if (envImpl.isReadOnly() && envImpl.isInInit()) {
779             return;
780         }

783             db.getId().equals(NAME_DB_ID)) {
784             envImpl.logMapTreeRoot();

787

789              * Retry indefinitely in the face of lock timeouts since the
790              * lock on the MapLN is only supposed to be held for short

802

805                             throw new EnvironmentFailureException(
806                                 envImpl,
807                                 EnvironmentFailureReason.LOG_INTEGRITY,
808                                 "Can't find database ID: " + db.getId());
809                         }
810                         /* Do nothing silently. */
811                         break;
812                     }
813 
814                     /* Check BIN LSN while latched. */
815                     if (ifBeforeLsn == DbLsn.NULL_LSN ||
816                         DbLsn.compareTo(
817                             cursor.getCurrentLsn(), ifBeforeLsn) < 0) {
818 
819                         MapLN mapLN = (MapLN) cursor.getCurrentLN(
820                             true, /*isLatched*/ true/*unlatch*/);
821 
822                         assert mapLN != null; /* Should be locked. */
823 
824                         /* Perform rewrite. */
825                         RewriteMapLN writeMapLN = new RewriteMapLN(cursor);
826                         mapLN.getDatabase().getTree().withRootLatchedExclusive(
827                             writeMapLN);
828 
829                         operationOk = true;
830                     }
831                     break;
832                 } catch (LockConflictException e) {
833                     /* Continue loop and retry. */
834                 } finally {
835                     if (cursor != null) {
836                         cursor.releaseBIN();
837                         cursor.close();
838                     }
839                     if (idDbLocker != null) {
840                         idDbLocker.operationEnd(operationOk);
841                     }
842                 }
843             }
844         }

897     /**
898      * Helper for database operations. This method positions a cursor
899      * on the NameLN that represents this database and write locks it.
900      *
901      * Do not evict (do not call CursorImpl.setAllowEviction(true)) during low
902      * level DbTree operation. [#15176]
903      *
904      * @throws IllegalStateException via
905      * Environment.remove/rename/truncateDatabase
906      */
907     private NameLockResult lockNameLN(Locker locker,
908                                       String databaseName,
909                                       String action,
910                                       GetRepContext getRepContext) 
911         throws DatabaseNotFoundException, NeedRepLockerException {
912 
913         /*
914          * We have to return both a cursor on the naming tree and a
915          * reference to the found DatabaseImpl.
916          */
917         NameLockResult result = new NameLockResult();
918 
919         /* Find the existing DatabaseImpl and establish a cursor. */
920         result.dbImpl = getDb(locker, databaseName, null, true);
921         if (result.dbImpl == null) {
922             throw new DatabaseNotFoundException
923                 ("Attempted to " + action + " non-existent database " +
924                  databaseName);
925         }
926 
927         boolean success = false;
928         try {
929             /* Get effective rep context and check for replica write. */
930             result.repContext = getRepContext.get(result.dbImpl);
931             checkReplicaWrite(locker, result.repContext);
932 
933             /*
934              * Check for an incorrect locker created via auto-commit.  This
935              * check is made after we have the DatabaseImpl and can check
936              * whether it is replicated.  See Environment.DbNameOperation.
937              */
938             if (envImpl.isReplicated() &&
939                 result.dbImpl.isReplicated() &&
940                 locker.getTxnLocker() != null &&
941                 locker.getTxnLocker().isAutoTxn() &&
942                 !locker.isReplicated()) {
943                 throw new NeedRepLockerException();
944             }
945 
946             result.nameCursor = new CursorImpl(nameDatabase, locker);
947 
948             /* Position the cursor at the specified NameLN. */
949             DatabaseEntry key =
950                 new DatabaseEntry(StringUtils.toUTF8(databaseName));
951             /* See [#16210]. */
952             boolean found = result.nameCursor.searchExact(key, LockType.WRITE);
953 
954             if (!found) {
955                 throw new DatabaseNotFoundException(
956                     "Attempted to " + action + " non-existent database " +
957                     databaseName);
958             }
959 
960             /* Call lockAndGetCurrentLN to write lock the nameLN. */
961             result.nameLN = (NameLN) result.nameCursor.getCurrentLN(
962                 true, /*isLatched*/ true/*unlatch*/);
963             assert result.nameLN != null; /* Should be locked. */
964 
965             /*
966              * Check for open handles after we have the write lock and no other
967              * transactions can open a handle.  After obtaining the write lock,
968              * other handles may be open only if (1) we preempted their locks,
969              * or (2) a handle was opened with the same transaction as used for
970              * this operation.  For (1), we mark the handles as preempted to
971              * cause a DatabasePreemptedException the next time they are
972              * accessed.  For (2), we throw IllegalStateException.
973              */
974             if (locker.getImportunate()) {
975                 /* We preempted the lock of all open DB handles. [#17015] */
976                 final String msg =
977                     "Database " + databaseName +
978                     " has been forcibly closed in order to apply a" +
979                     " replicated " + action + " operation.  This Database" +
980                     " and all associated Cursors must be closed.  All" +
981                     " associated Transactions must be aborted.";
982                 for (Database db : result.dbImpl.getReferringHandles()) {
983                     DbInternal.setPreempted(db, databaseName, msg);
984                 }
985             } else {
986                 /* Disallow open handles for the same transaction. */
987                 int handleCount = result.dbImpl.getReferringHandleCount();
988                 if (handleCount > 0) {
989                     throw new IllegalStateException
990                         ("Can't " + action + " database " + databaseName +
991                          ", " + handleCount + " open Database handles exist");
992                 }
993             }
994             success = true;
995         } finally {
996             if (!success) {
997                 releaseDb(result.dbImpl);
998                 if (result.nameCursor != null) {
999                     result.nameCursor.releaseBIN();
1000                     result.nameCursor.close();
1001                 }
1002             }
1003         }
1004 
1005         return result;
1006     }
