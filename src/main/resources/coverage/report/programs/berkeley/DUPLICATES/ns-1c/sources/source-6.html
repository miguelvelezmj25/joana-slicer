79 /**
80  * There is an instance of a Feeder for each client that needs a replication
81  * stream. Either a master, or replica (providing feeder services) may
82  * establish a feeder.
83  *
84  * A feeder is created in response to a request from a Replica, and is shutdown
85  * either upon loss of connectivity, or upon a change in mastership.
86  *
87  * The protocol used to validate and negotiate a connection is synchronous, but
88  * once this phase has been completed, the communication between the feeder and
89  * replica is asynchronous. To handle the async communications, the feeder has
90  * two threads associated with it:
91  *
92  * 1) An output thread whose sole purpose is to pump log records (and if
93  * necessary heart beat requests) down to the replica as fast as the network
94  * will allow it
95  *
96  * 2) An input thread that listens for responses to transaction commits and
97  * heart beat responses.
98  *
99  * <p>The feeder maintains several statistics that provide information about
100  * the replication rate for each replica.  By comparing this information to
101  * information about master replication maintained by the FeederTxns class, it
102  * is also possible to estimate the lag between replicas and the master.
103  *
104  * <p>The statistics facilities do not expect the set of available statistics
105  * to change dynamically.  To handle recording statistics about the changing
106  * set of replicas, the statistics are represented as maps that associated node
107  * names with statistics.  Each feeder adds individual statistics in these maps
108  * at startup, and removes them at shutdown time to make sure that the
109  * statistics in the map only reflect up-to-date information.
110  *
111  * <p>Some notes about the specific statistics:<dl>
112  *
113  * <dt>replicaDelay
114  *
115  * <dd>The difference between the commit times of the latest transaction
116  * committed on the master and the transaction most recently processed by the
117  * replica.  The master timestamp comes from the lastCommitTimestamp statistic
118  * maintained by FeederTxns.  The feeder determines the commit timestamp of the
119  * replica's most recently processed transaction by obtaining timestamps from
120  * commit records being sent to the replica, and noting the last one prior to
121  * sending a heartbeat.  When a heartbeat response is received, if the latest
122  * replica VLSN included in the response is equal or greater to the one
123  * recorded when the heartbeat request was sent, then the delay is computed by
124  * comparing the commit timestamp for that most recently sent transaction with
125  * the timestamp of the master's latest transaction.  Replicas can send
126  * heartbeat responses on their own, so comparing the VLSNs is necessary to
127  * make sure that the response matches the request.  Note that this arrangement
128  * depends on the fact that the replica processes transactions and heartbeats
129  * in order, and only sends a heartbeat response once all preceding
130  * transactions have been processed.  If the master processes transactions at a
131  * fast enough rate that additional transactions are generated while waiting
132  * for a heartbeat response, then the value of this statistic will not reach
133  * zero, but will represent the total time for sending a commit operation to
134  * the replica and receiving the associated response, including the roundtrip
135  * latency of the network and any time spent due to buffering of replication
136  * data.
137  *
138  * <dt>replicaLastCommitTimestamp
139  *
140  * <dd>The commit timestamp of the last transaction committed before the most
141  * recent heartbeat for which a heartbeat response has been received.  This
142  * statistic represents the commit time on the master of the most recent data
143  * known to have been processed on the replica.  It provides the information
144  * used for the replica component of the replicaDelay statistic.
145  *
146  * <dt>replicaLastCommitVLSN
147  *
148  * <dd>The VLSN of the committed transaction described for
149  * replicaLastCommitTimestamp.  This statistic provides the information used
150  * for the replica component of the replicaVLSNLag statistic.
151  *
152  * <dt>replicaVLSNLag
153  *
154  * <dd>The difference between the VLSN of the latest transaction committed on
155  * the master and the one most recently processed by the replica.  The master
156  * VLSN comes from the lastCommitVLSN statistic maintained by FeederTxns.  This
157  * statistic is similar to replicaDelay, but provides information about the
158  * VLSN lag rather than the time delay.
159  *
160  * <dt>replicaVLSNRate
161  *
162  * <dd>An exponential moving average of the rate of change of the
163  * replicaLastCommitVLSN statistic over time, averaged over a 10 second time
164  * period.  This statistic provides information about how quickly the replica
165  * is processing replication data, which can be used, along with the vlsnRate
166  * statistic maintained by FeederTxns, to estimate the amount of time it will
167  * take for the replica to catch up with the master.</dl>
168  */

576     }

767         /*
768          * This method deals with responses from the Replica. There are exactly
769          * two types of responses from the Replica:
770          *
771          * 1) Responses acknowledging a successful commit by the Replica.
772          *
773          * 2) Responses to heart beat messages.
774          *
775          * This loop (like the loop in the OutputThread) is terminated under
776          * one of the following conditions:
777          *
778          * 1) The thread detects a change in masters.
779          * 2) There is network connection issue (which might also be an
780          *    indication of an unfolding change in masters).
781          * 3) If the replica closes its connection -- variation of the above.
782          *
783          * In addition, the loop will also exit if it gets a ShutdownResponse
784          * message sent in response to a ShutdownRequest sent by the
785          * OutputThread.
786          */
787         private void runResponseLoop()
788             throws IOException, MasterSyncException {
789 
790             /*
791              * Start the acknowledgment loop. It's very important that this
792              * loop be wait/contention free.
793              */
794             while (!checkShutdown()) {
795                 Message response = protocol.read(feederReplicaChannel);
796                 if (checkShutdown()) {
797 
798                     /*
799                      * Shutdown quickly, in particular, don't update sync
800                      * VLSNs.
801                      */
802                     break;
803                 }
804                 masterStatus.assertSync();
805 
806                 lastResponseTime = System.currentTimeMillis();
807 
808                 if (response.getOp() == Protocol.HEARTBEAT_RESPONSE) {
809                     processHeartbeatResponse(response);
810                 } else if (response.getOp() == Protocol.ACK) {
811 
812                     /*
813                      * Check if a commit has been waiting for this
814                      * acknowledgment and signal any waiters.
815                      */
816                     long txnId = ((Ack) response).getTxnId();
817                     if (logger.isLoggable(Level.FINE)) {
818                         LoggerUtils.fine(logger, repImpl, "Ack for: " + txnId);
819                     }
820                     deemAcked(txnId);
821                 }  else if (response.getOp() == Protocol.GROUP_ACK) {
822                     final long txnIds[] = ((GroupAck) response).getTxnIds();
823 
824                     for (long txnId : txnIds) {
825                         if (logger.isLoggable(Level.FINE)) {
826                             LoggerUtils.fine(logger, repImpl,
827                                              "Group Ack for: " + txnId);
828                         }
829                         deemAcked(txnId);
830                     }
831                 } else if (response.getOp() == Protocol.SHUTDOWN_RESPONSE) {
832                     LoggerUtils.info(logger, repImpl,
833                                      "Shutdown confirmed by replica " +
834                                      replicaNameIdPair.getName());
835                     /* Exit the loop and the thread. */
836                     break;
837                 } else if (response.getOp() == Protocol.REAUTHENTICATE) {
838                     processReauthenticate(response);
839                 } else {
840                     throw EnvironmentFailureException.unexpectedState
841                         ("Unexpected message: " + response);
842                 }
843             }
844         }

846         private void processHeartbeatResponse(Message response) {
847             /* Last response has been updated, keep going. */
848             final HeartbeatResponse hbResponse =
849                 (Protocol.HeartbeatResponse)response;
850 
851             /*
852              * For arbiters we do not process the response, but it is still
853              * important for preventing the channel from timing out.
854              */
855             if (replicaNode.getType().isArbiter()) {
856                 return;
857             }
858 
859             /*
860              * When the global CBVLSN is not defunct, update it for a data node
861              * (replicaCBVLSN is null for non-data nodes).
862              */
863             if (replicaCBVLSN != null) {
864                 replicaCBVLSN.updateForReplica(hbResponse);
865             }
866 
867             final VLSN replicaTxnVLSN = hbResponse.getTxnEndVLSN();
868 
869             /* All further work requires the replica's VLSN */
870             if (replicaTxnVLSN == null) {
871                 return;
872             }
873 
874             replicaTxnEndVLSN = replicaTxnVLSN;
875             final long replicaTxnVLSNSeq = replicaTxnVLSN.getSequence();
876 
877             feederManager.updateDTVLSN(replicaTxnVLSNSeq);
878 
879             if (replicaTxnVLSN.compareTo(
880                 repNode.getCurrentTxnEndVLSN()) >= 0) {
881 
882                 caughtUp = true;
883                 adviseMasterTransferProgress();
884             }
885 
886             /*
887              * Only tally statistics for the commit VLSN and timestamp if both
888              * values were recorded when the heartbeat was requested.  Make
889              * computations based directly on the measured heartbeat delay if
890              * the heartbeat reply confirms that the requested VLSN has been
891              * processed.  Otherwise, use the master VLSN rate to estimate the
892              * delay.
893              */
894             final long commitVLSN = lastHeartbeatCommitVLSN;
895             final long commitTimestamp = lastHeartbeatCommitTimestamp;
896             if ((commitVLSN == 0) || (commitTimestamp == 0)) {
897                 return;
898             }
899             final long statCommitVLSN = (commitVLSN <= replicaTxnVLSNSeq) ?
900                 commitVLSN : replicaTxnVLSNSeq;
901 
902             /* Set the depended-on stats first */
903             replicaLastCommitVLSN.set(statCommitVLSN);
904             replicaVLSNLag.set(statCommitVLSN, lastResponseTime);
905             replicaVLSNRate.add(statCommitVLSN, lastResponseTime);
906 
907             final long statCommitTimestamp;
908             if (commitVLSN <= replicaTxnVLSNSeq) {
909                 statCommitTimestamp = commitTimestamp;
910             } else {
911 
912                 /* Adjust the commit timestamp based on the VLSN rate */
913                 final long vlsnRatePerMinute = vlsnRate.get();
914                 if (vlsnRatePerMinute <= 0) {
915                     return;
916                 }
917                 final long vlsnLag = commitVLSN - replicaTxnVLSNSeq;
918                 final long timeLagMillis =
919                     (long) (60000.0 * ((double) vlsnLag / vlsnRatePerMinute));
920                 statCommitTimestamp = commitTimestamp - timeLagMillis;
921             }
922             replicaLastCommitTimestamp.set(statCommitTimestamp);
923             replicaDelay.set(statCommitTimestamp, lastResponseTime);
924         }
