38 /**
39  * DbBackup is a helper class for stopping and restarting JE background
40  * activity in an open environment in order to simplify backup operations. It
41  * also lets the application create a backup which can support restoring the
42  * environment to a specific point in time.
43  * <p>
44  * <b>Backing up without DbBackup</b>
45  * <p>
46  * Because JE has an append only log file architecture, it is always possible
47  * to do a hot backup without the use of DbBackup by copying all log files
48  * (.jdb files) to your archival location. As long as the log files are copied
49  * in alphabetical order, (numerical in effect) <i>and</i> all log files are
50  * copied, the environment can be successfully backed up without any need to
51  * stop database operations or background activity. This means that your
52  * backup operation must do a loop to check for the creation of new log files
53  * before deciding that the backup is finished. For example:
54  * <pre>
55  * time    files in                    activity
56  *         environment
57  *
58  *  t0     000000001.jdb     Backup starts copying file 1
59  *         000000003.jdb
60  *         000000004.jdb
61  *
62  *  t1     000000001.jdb     JE log cleaner migrates portion of file 3 to newly
63  *         000000004.jdb     created file 5 and deletes file 3. Backup finishes
64  *         000000005.jdb     file 1, starts copying file 4. Backup MUST include
65  *                           file 5 for a consistent backup!
66  *
67  *  t2     000000001.jdb     Backup finishes copying file 4, starts and
68  *         000000004.jdb     finishes file 5, has caught up. Backup ends.
69  *         000000005.jdb
70  *</pre>
71  * <p>
72  * In the example above, the backup operation must be sure to copy file 5,
73  * which came into existence after the backup had started. If the backup
74  * stopped operations at file 4, the backup set would include only file 1 and
75  * 4, omitting file 3, which would be an inconsistent set.
76  * <p>
77  * Also note that log file 5 may not have filled up before it was copied to
78  * archival storage. On the next backup, there might be a newer, larger version
79  * of file 5, and that newer version should replace the older file 5 in archive
80  * storage.
81  * <p>
82  * Using the approach above, as opposed to using DbBackup, will copy all files
83  * including {@link EnvironmentStats#getReservedLogSize reserved files} as well
84  * as {@link EnvironmentStats#getActiveLogSize active files}. A large number of
85  * reserved files may be present in an HA Environment, and they are essentially
86  * wasted space in a backup. Using DbBackup is strongly recommended for this
87  * reason, as well as to reduce the complexity of file copying.
88  * <p>
89  * <b>Backing up with DbBackup</b>
90  * <p>
91  * DbBackup helps simplify application backup by defining the set of {@link
92  * EnvironmentStats#getActiveLogSize active files} that must be copied for each
93  * backup operation. If the environment directory has read/write protection,
94  * the application must pass DbBackup an open, read/write environment handle.
95  * <p>
96  * When entering backup mode, JE determines the set of active files needed for
97  * a consistent backup, and freezes all changes to those files. The application
98  * can copy that defined set of files and finish operation without checking for
99  * the ongoing creation of new files. Also, there will be no need to check for
100  * a newer version of the last file on the next backup.
101  * <p>
102  * In the example above, if DbBackup was used at t0, the application would only
103  * have to copy files 1, 3 and 4 to back up. On a subsequent backup, the
104  * application could start its copying at file 5. There would be no need to
105  * check for a newer version of file 4.
106  * <p>
107  * When it is important to minimize the time that it takes to recover using a
108  * backup, a checkpoint should be performed immediately before calling {@link
109  * #startBackup}.  This will reduce recovery time when opening the environment
110  * with the restored log files.  A checkpoint is performed explicitly by
111  * calling {@link Environment#checkpoint} using a config object for which
112  * {@link CheckpointConfig#setForce setForce(true)} has been called.
113  * <p>
114  * <b>Performing simple/full backups</b>
115  * <p>
116  * The following examples shows how to perform a full backup.  A checkpoint is
117  * performed to minimize recovery time.
118  * <pre class="code">
119  * void myBackup(Environment env, File destDir) {
120  *     DbBackup backupHelper = new DbBackup(env);
121  *
122  *     // Optional: Do a checkpoint to reduce recovery time after a restore.
123  *     env.checkpoint(new CheckpointConfig().setForce(true));
124  *
125  *     // Start backup, find out what needs to be copied.
126  *     backupHelper.startBackup();
127  *     try {
128  *         // Copy the necessary files to archival storage.
129  *         String[] filesToCopy = backupHelper.getLogFilesInBackupSet();
130  *         myCopyFiles(env, backupHelper, filesToCopy, destDir);
131  *     } finally {
132  *         // Remember to exit backup mode, or the JE cleaner cannot delete
133  *         // log files and disk usage will grow without bounds.
134  *        backupHelper.endBackup();
135  *     }
136  * }
137  *
138  * void myCopyFiles(
139  *     Environment env,
140  *     DbBackup backupHelper,
141  *     String[] filesToCopy,
142  *     File destDir) {
143  *
144  *     for (String fileName : filesToCopy) {
145  *         // Copy fileName to destDir.
146  *         // See {@link LogVerificationReadableByteChannel} and
147  *         // {@link LogVerificationInputStream}.
148  *         ....
149  *
150  *         // Remove protection to allow file to be deleted in order to reclaim
151  *         // disk space.
152  *         backupHelper.removeFileProtection(fileName);
153  *     }
154  * }
155  * </pre>
156  * When copying files to the backup directory, it is critical that each file is
157  * verified before or during the copy.  If a file is copied that is corrupt
158  * (due to an earlier disk failure that went unnoticed, for example), the
159  * backup will be invalid and provide a false sense of security.
160  * <p>
161  * The {@link LogVerificationInputStream example here} shows how to implement
162  * the {@code myCopyFiles} method using {@link
163  * LogVerificationInputStream}. A {@link LogVerificationReadableByteChannel}
164  * could also be used for higher performance copying. A filter input stream is
165  * used to verify the file efficiently as it is being read.  If you choose to
166  * use a script for copying files, the {@link DbVerifyLog} command line tool
167  * can be used instead.
168  * <p>
169  * Assuming that the full backup copied files into an empty directory, to
170  * restore you can simply copy these files back into another empty directory.
171  * <p>
172  * Always start with an empty directory as the destination for a full backup or
173  * a restore, to ensure that no unused files are present.  Unused files --
174  * perhaps the residual of an earlier environment or an earlier backup -- will
175  * take up space, and they will never be deleted by the JE log cleaner.  Also
176  * note that such files will not be used by JE for calculating utilization and
177  * will not appear in the {@link DbSpace} output.
178  * <p>
179  * <b>Performing incremental backups</b>
180  * <p>
181  * Incremental backups are used to reduce the number of files copied during
182  * each backup.  Compared to a full backup, there are two additional pieces of
183  * information needed for an incremental backup: the number of the last file in
184  * the previous backup, and a list of the active files in the environment
185  * directory at the time of the current backup, i.e., the current snapshot.
186  * Their purpose is explained below.
187  * <p>
188  * The number of the last file in the previous backup is used to avoid copying
189  * files that are already present in the backup set.  This file number must be
190  * obtained before beginning the backup, either by checking the backup archive,
191  * or getting this value from a stored location.  For example, the last file
192  * number could be written to a special file in the backup set at the time of a
193  * backup, and then read from the special file before starting the next backup.
194  * <p>
195  * The list of files in the current snapshot, which should be obtained by
196  * calling {@link #getLogFilesInSnapshot} (after calling {@link #startBackup}),
197  * is used to avoid unused files after a restore, and may also be used to
198  * reduce the size of the backup set.  How to use this list is described below.
199  * <p>
200  * Some applications need the ability to restore to the point in time of any of
201  * the incremental backups that were made in the past, and other applications
202  * only need to restore to the point in time of the most recent backup.
203  * Accordingly, the list of current files (that is made at the time of the
204  * backup), should be used in one of two ways.
205  * <ol>
206  *   <li>If you only need to restore to the point in time of the most recent
207  *   backup, then the list should be used to delete unused files from the
208  *   backup set.  After copying all files during the backup, any file that is
209  *   <em>not</em> present in the list may then be deleted from the backup set.
210  *   This both reduces the size of the backup set, and ensures that unused
211  *   files will not be present in the backup set and therefore will not be
212  *   restored.</li>
213  *   <li>If you need to keep all log files from each backup so you can restore
214  *   to more than one point in time, then the list for each backup should be
215  *   saved with the backup file set so it can be used during a restore. During
216  *   the restore, only the files in the list should be copied, starting with an
217  *   empty destination directory.  This ensures that unused files will not be
218  *   restored.</li>
219  * </ol>
220  * <p>
221  * The following two examples shows how to perform an incremental backup.  In
222  * the first example, the list of current files is used to delete files from
223  * the backup set that are no longer needed.
224  * <pre class="code">
225  * void myBackup(Environment env, File destDir) {
226  *
227  *     // Get the file number of the last file in the previous backup.
228  *     long lastFileInPrevBackup =  ...
229  *
230  *     DbBackup backupHelper = new DbBackup(env, lastFileInPrevBackup);
231  *
232  *     // Optional: Do a checkpoint to reduce recovery time after a restore.
233  *     env.checkpoint(new CheckpointConfig().setForce(true));
234  *
235  *     // Start backup, find out what needs to be copied.
236  *     backupHelper.startBackup();
237  *     try {
238  *         // Copy the necessary files to archival storage.
239  *         String[] filesToCopy = backupHelper.getLogFilesInBackupSet();
240  *         myCopyFiles(env, backupHelper, filesToCopy, destDir);
241  *
242  *         // Delete files that are no longer needed.
243  *         // WARNING: This should only be done after copying all new files.
244  *         String[] filesInSnapshot = backupHelper.getLogFilesInSnapshot();
245  *         myDeleteUnusedFiles(destDir, filesInSnapshot);
246  *
247  *         // Update knowledge of last file saved in the backup set.
248  *         lastFileInPrevBackup = backupHelper.getLastFileInBackupSet();
249  *         // Save lastFileInPrevBackup persistently here ...
250  *     } finally {
251  *         // Remember to exit backup mode, or the JE cleaner cannot delete
252  *         // log files and disk usage will grow without bounds.
253  *        backupHelper.endBackup();
254  *     }
255  * }
256  *
257  * void myDeleteUnusedFiles(File destDir, String[] filesInSnapshot) {
258  *     // For each file in destDir that is NOT in filesInSnapshot, it should
259  *     // be deleted from destDir to save disk space in the backup set, and to
260  *     // ensure that unused files will not be restored.
261  * }
262  *
263  * See myCopyFiles further above.
264  * </pre>
265  * <p>
266  * When performing backups as shown in the first example above, to restore you
267  * can simply copy all files from the backup set into an empty directory.
268  * <p>
269  * In the second example below, the list of current files is saved with the
270  * backup set so it can be used during a restore.  The backup set will
271  * effectively hold multiple backups that can be used to restore to different
272  * points in time.
273  * <pre class="code">
274  * void myBackup(Environment env, File destDir) {
275  *
276  *     // Get the file number of the last file in the previous backup.
277  *     long lastFileInPrevBackup =  ...
278  *
279  *     DbBackup backupHelper = new DbBackup(env, lastFileInPrevBackup);
280  *
281  *     // Optional: Do a checkpoint to reduce recovery time after a restore.
282  *     env.checkpoint(new CheckpointConfig().setForce(true));
283  *
284  *     // Start backup, find out what needs to be copied.
285  *     backupHelper.startBackup();
286  *     try {
287  *         // Copy the necessary files to archival storage.
288  *         String[] filesToCopy = backupHelper.getLogFilesInBackupSet();
289  *         myCopyFiles(env, backupHelper, filesToCopy, destDir);
290  *
291  *         // Save current list of files with backup data set.
292  *         String[] filesInSnapshot = backupHelper.getLogFilesInSnapshot();
293  *         // Save filesInSnapshot persistently here ...
294  *
295  *         // Update knowledge of last file saved in the backup set.
296  *         lastFileInPrevBackup = backupHelper.getLastFileInBackupSet();
297  *         // Save lastFileInPrevBackup persistently here ...
298  *     } finally {
299  *         // Remember to exit backup mode, or the JE cleaner cannot delete
300  *         // log files and disk usage will grow without bounds.
301  *        backupHelper.endBackup();
302  *     }
303  * }
304  *
305  * See myCopyFiles further above.
306  * </pre>
307  * <p>
308  * When performing backups as shown in the second example above, to restore you
309  * must choose one of the file lists that was saved.  You may choose the list
310  * written by the most recent backup, or a list written by an earlier backup.
311  * To restore, the files in the list should be copied into an empty destination
312  * directory.
313  * <p>
314  * <b><a href="restore">Restoring from a backup</a></b>
315  * <p>
316  * As described in the sections above, the restore procedure is to copy the
317  * files from a backup set into an empty directory.  Depending on the type of
318  * backup that was performed (see above), either all files from the backup set
319  * are copied, or only the files on a list that was created during the backup.
320  * <p>
321  * There is one additional consideration when performing a restore, under the
322  * following condition:
323  * <ul>
324  *     <li>Incremental backups are used, AND
325  *     <ul>
326  *         <li>the backup was created using DbBackup with JE 6.2 or earlier,
327  *         OR</li>
328  *         <li>the backup was created in a read-only JE environment.</li>
329  *     </ul>
330  *     </li>
331  * </ul>
332  * <p>
333  * If the above condition holds, after copying the files an additional step is
334  * needed. To enable the creation of future incremental backups using the
335  * restored files, the {@link
336  * com.sleepycat.je.EnvironmentConfig#ENV_RECOVERY_FORCE_NEW_FILE} parameter
337  * should be set to true when opening the JE Environment for the first time
338  * after the restore.  When this parameter is set to true, the last .jdb file
339  * restored will not be modified when opening the Environment, and the next
340  * .jdb file will be created and will become the end-of-log file.
341  * <p>
342  * WARNING: When the above special condition is true and this property is
343  * <em>not</em> set to true when opening the environment for the first time
344  * after a restore, then the backup set that was restored may not be used as
345  * the basis for future incremental backups. If a future incremental backup
346  * were performed based on this backup set, it would be incomplete and data
347  * would be lost if that incremental backup were restored.
348  * <p>
349  * When JE 6.3 or later is used to create the backup, and the backup is created
350  * in a read-write environment (the usual case), this extra step is
351  * unnecessary. In this case, {@link #startBackup} will have added an
352  * "immutable file" marker to the last file in the backup and this will prevent
353  * that file from being modified, just as if the
354  * {@code ENV_RECOVERY_FORCE_NEW_FILE} parameter were set to true.
355  */

453     /**
454      * Start backup mode in order to determine the definitive backup set needed
455      * at this point in time.
456      *
457      * <p>This method determines the last file in the backup set, which is the
458      * last log file in the environment at this point in time.  Following this
459      * method call, all new data will be written to other, new log files.  In
460      * other words, the last file in the backup set will not be modified after
461      * this method returns.</p>
462      *
463      * <p><em>WARNING:</em> After calling this method, deletion of log files in
464      * the backup set by the JE log cleaner will be disabled until {@link
465      * #endBackup()} is called.  To prevent unbounded growth of disk usage, be
466      * sure to call {@link #endBackup()} to re-enable log file deletion.
467      * Additionally, the Environment can't be closed until endBackup() is
468      * called.
469      * </p>
470      *
471      * @throws com.sleepycat.je.rep.LogOverwriteException if a replication
472      * operation is overwriting log files. The backup can not proceed because
473      * files may be invalid. The backup may be attempted at a later time.
474      *
475      * @throws EnvironmentFailureException if an unexpected, internal or
476      * environment-wide failure occurs.
477      *
478      * @throws IllegalStateException if a backup is already in progress
479      */
480     public synchronized void startBackup()
481         throws DatabaseException {
482 
483         if (backupStarted) {
484             throw new IllegalStateException("startBackup was already called");
485         }
486 
487         /* Throw a LogOverwriteException if the Environment is rolling back. */
488         if (!envImpl.addDbBackup(this)) {
489             throw envImpl.createLogOverwriteException
490                 ("A replication operation is overwriting log files. The " +
491                  "backup can not proceed because files may be invalid. The " +
492                  "backup may be attempted at a later time.");
493         }
494 
495         final FileProtector fileProtector = envImpl.getFileProtector();
496 
497         /*
498          * For a network restore we use a different backup name/id and also
499          * protect the 2 newest (highest numbered) reserved files. This is a
500          * safeguard to ensure that the restored node can function as a master.
501          * In a future release this approach may be improved by additionally
502          * copying all reserved files to the restored node, but only after it
503          * has recovered using the active files [#25783].
504          */
505         final long backupId = networkRestore ?
506             envImpl.getNodeSequence().getNextNetworkRestoreId() :
507             envImpl.getNodeSequence().getNextBackupId();
508 
509         final String backupName =
510             (networkRestore ?
511              FileProtector.NETWORK_RESTORE_NAME :
512              FileProtector.BACKUP_NAME) +
513             "-" + backupId;
514 
515         final int nReservedFiles = networkRestore ? 2 : 0;
516 
517         /*
518          * Protect all files from deletion momentarily, while determining the
519          * last file and protecting the active files.
520          */
521         final ProtectedFileRange allFilesProtected =
522             fileProtector.protectFileRange(backupName + "-init", 0);
523 
524         try {
525             if (envIsReadOnly) {
526                 /*
527                  * All files are currently immutable, so the backup list is the
528                  * current set of files. However, we can't add a marker to the
529                  * last file in list, and therefore it will not be immutable
530                  * after being restored to a read-write directory (unless the
531                  * user sets ENV_RECOVERY_FORCE_NEW_FILE after restoring).
532                  */
533                 lastFileInBackup = envImpl.getFileManager().getLastFileNum();
534             } else {
535                 /*
536                  * Flip the log so that all files in the backup list are
537                  * immutable. But first, write an "immutable file" marker in
538                  * the last file in the backup, so it cannot be modified after
539                  * it is restored. Recovery enforces this rule.
540                  */
541                 LogEntry marker = new SingleItemEntry<>(
542                     LogEntryType.LOG_IMMUTABLE_FILE, new EmptyLogEntry());
543 
544                 long markerLsn = envImpl.getLogManager().log(
545                     marker, ReplicationContext.NO_REPLICATE);
546 
547                 envImpl.forceLogFileFlip();
548                 lastFileInBackup = DbLsn.getFileNumber(markerLsn);
549             }
550 
551             /*
552              * Protect all active files from deletion. This includes files
553              * prior to firstFileInBackup, in order to get a snapshot of all
554              * active files. New files do not need protection, since the backup
555              * set does not include them. lastFileInBackup will be protected by
556              * protectActiveFiles because it is not the last file in the env.
557              */
558             protectedFileSet = fileProtector.protectActiveFiles(
559                 backupName, nReservedFiles, false /*protectNewFiles*/);
560 
561         } finally {
562             fileProtector.removeFileProtection(allFilesProtected);
563         }
564 
565         /* At this point, endBackup must be called to undo file protection. */
566         backupStarted = true;
567 
568         /* Files after lastFileInBackup do not need protection. */
569         protectedFileSet.truncateTail(lastFileInBackup);
570 
571         /* Snapshot the complete backup file set. */
572         snapshotFiles = protectedFileSet.getProtectedFiles();
573 
574         /*
575          * Now that we have the snapshot, files before firstFileInBackup do not
576          * need protection.
577          */
578         protectedFileSet.truncateHead(firstFileInBackup);
579     }
