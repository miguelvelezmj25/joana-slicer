39 /**
40  * A VLSNBucket instance represents a set of VLSN->LSN mappings. Buckets are
41  * usually not updated, except at times when the replication stream may have
42  * been reduced in size, by log cleaning or syncup. The VLSNBuckets in the
43  * VLSNIndex's VLSNTracker are written to disk and are persistent. There are
44  * also VLSNBuckets in the temporary recovery-time tracker that are used for
45  * collecting mappings found in the log during recovery.
46  *
47  * VLSNBuckets only hold mappings from a single log file. A single log file
48  * may be mapped by multiple VLSNBuckets though.
49  *
50  * As a tradeoff in space vs time, a VLSNBucket only stores a sparse set of
51  * mappings and the caller must use a VLSNReader to scan the log file and
52  * find any log entries not mapped directly by the bucket. In addition,
53  * the VLSN is not actually stored. Only the offset portion of the LSN is
54  * stored, and the VLSN is intuited by a stride field. Each VLSNBucket
55  * only maps a single file, though a single file may be mapped by several
56  * VLSNBuckets.
57  *
58  * For example, suppose a node had these VLSN->LSN mappings:
59  *
60  * VLSN            LSN (file/offset)
61  * 9               10/100
62  * 10              10/110
63  * 11              10/120
64  * 12              10/130
65  * 13              10/140
66  * 14              11/100
67  * 15              11/120
68  *
69  * The mappings in file 10 could be represented by a VLSNBucket with
70  * a stride of 4. That means the bucket would hold the mappings for
71  *  9              10/100,
72  * 13              10/140
73  *
74  * And since the target log file number and the stride is known, the mappings
75  * can be represented in by the offset alone in this array: {100, 140}, rather
76  * than storing the whole lsn.
77  *
78  * Each bucket can also provide the mapping for the first and last VLSN it
79  * covers, even if the lastVLSN is not divisible by the stride. This is done to
80  * support forward and backward scanning. From the example above, the completed
81  * bucket can provide 9->10/100, 13->10/140, 15 -> 10/160 even though 15 is not
82  * a stride's worth away from 13.
83  *
84  * Because registering a VLSN->LSN mapping is done outside the log write latch,
85  * any inserts into the VLSNBucket may not be in order. However, when any
86  * VLSN is registered, we can assume that all VLSNs < that value do exist in
87  * the log. It's just an accident of timing that they haven't yet been
88  * registered. Note that out of order inserts into the buckets can create holes
89  * in the bucket's offset array, or cause the array to be shorter than
90  * anticipated.
91  *
92  * For example, if the insertion order into the bucket is vlsns 9, 15, we'll
93  * actually only keep an offset array of size 1. We have to be able to handle
94  * holes in the bucket, and can't count on filling them in when the lagging
95  * vlsn arrives, because it is possible that a reading thread will access the
96  * bucket before the laggard inserter arrives, or that the bucket might be
97  * flushed to disk, and become immutable.
98  */

180     /**
181      * Record the LSN location for this VLSN.
182      *
183      * One key issue is that puts() are not synchronized, and the VLSNs may
184      * arrive out of order. If an out of order VLSN does arrive, we can still
185      * assume that the earlier VLSNs have been successfully logged. If a VLSN
186      * arrives that is divisible by the stride, and should be recorded in the
187      * fileOffsets, but is not the next VLSN that should be recorded, we'll pad
188      * out the fileOffsets list with placeholders.
189      *
190      * For example, suppose the stride is 3, and the first VLSN is 2. Then this
191      * bucket should record VLSN 2, 5, 8, ... etc.  If VLSN 8 arrives before
192      * VLSN 5, VLSN 8 will be recorded, and VLSN 5 will have an offset
193      * placeholder of NO_OFFSET. It is a non-issue if VLSNs 3, 4, 6, 7 arrive
194      * out of order, because they would not have been recorded anyway. This
195      * should not happen often, because the stride should be fairly large, and
196      * the calls to put() should be close together. If the insertion order is
197      * vlsn 2, 5, 9, then the file offsets array will be a little short, and
198      * will only have 2 elements, instead of 3.
199      *
200      * We follow this policy because we must always have a valid begin and end
201      * point for the range. We must handle placeholders in all cases, and can't
202      * count of later vlsn inserts, because a bucket can become immutable at
203      * any time if it is flushed to disk.
204      *
205      * @return false if this bucket will not accept this VLSN. Generally, a
206      * refusal might happen because the bucket was full or the mapping is too
207      * large a distance away from the previous mapping. In that case, the
208      * tracker will start another bucket.
209      */
210     synchronized boolean put(VLSN vlsn, long lsn) {
211 
212         if (closed) {
213             return false;
214         }
215 
216         if (!belongs(vlsn, lsn)) {
217             return false;
218         }
219 
220         /*
221          * Add it to the fileOffset if it's on a stride boundary and is the
222          * next mapping in the fileOffset list.
223          */
224         if (isModulo(vlsn)) {
225             int index = getIndex(vlsn);
226             int listLen = fileOffsets.size();
227             if (index < listLen) {
228                 fileOffsets.set(index, DbLsn.getFileOffsetAsInt(lsn));
229             } else if (index == listLen) {
230                 fileOffsets.add(DbLsn.getFileOffsetAsInt(lsn));
231             } else {
232                 for (int i = listLen; i < index; i++) {
233                     fileOffsets.add(NO_OFFSET);
234                 }
235                 fileOffsets.add(DbLsn.getFileOffsetAsInt(lsn));
236             }
237             dirty = true;
238         }
239 
240         /* If the lastLsn is less, or not initialized, set it to this VLSN. */
241         if ((lastVLSN.compareTo(vlsn) < 0) ||
242             (lastLsn == DbLsn.NULL_LSN)) {
243             lastVLSN = vlsn;
244             lastLsn = lsn;
245             dirty = true;
246         }
247 
248         return true;
249     }

268      * @return true if this VLSN->LSN mapping should go into this bucket.
269      */
270     private boolean belongs(VLSN vlsn, long lsn) {
271         assert vlsn.compareTo(firstVLSN) >= 0 :
272             "firstVLSN = " + firstVLSN + " should not be greater than " + vlsn;
273 
274         if (DbLsn.getFileNumber(lsn) != fileNumber) {
275             /* Mappings must be for same file. */
276             return false;
277         }
278 
279         if (emptyInternal()) {
280             return true;
281         }
282 
283         /*
284          * Some other thread beat us to the put() call and inserted a later
285          * mapping, so we know for sure that we fit in this bucket
286          */
287         if (lastVLSN.compareTo(vlsn) > 0) {
288             return true;
289         }
290 
291         boolean onStrideBoundary = isModulo(vlsn);
292         if (onStrideBoundary && (fileOffsets.size() >= maxMappings)) {
293             /* Too full. */
294             return false;
295         }
296 
297         /*
298          * Will this VLSN be next one recorded in the fileOffsets? If so,
299          * calculate the scan distance.
300          */
301         if ((onStrideBoundary && (getIndex(vlsn) == fileOffsets.size())) ||
302             lastVLSN.compareTo(vlsn) < 0) {
303             /* This VLSN is going in at the tail of the bucket. */
304             int lastOffset = fileOffsets.get(fileOffsets.size() - 1);
305             if ((DbLsn.getFileOffset(lsn) -
306                  DbLsn.convertIntFileOffsetToLong(lastOffset)) >
307                 maxDistance) {
308                 /* The scan distance is exceeded. */
309                 return false;
310             }
311         }
312 
313         return true;
314     }

575

585             return null;

593             fileOffsets = null;
594             firstVLSN = VLSN.NULL_VLSN;
595             lastVLSN = VLSN.NULL_VLSN;

603         /*
604          * Find the mappings that still belong. Using the example above, we
605          * should find that we can delete fileOffset[0] and fileOffset[1] and
606          * preserve fileOffset[2]
607          */
608         for (int i = 0; i < fileOffsets.size(); i++) {
609             if ((indexVLSN.compareTo(lastDuplicate) > 0) &&
610                 (fileOffsets.get(i) != NO_OFFSET)) {
611                 newFirstIndex = i;
612                 break;
613             }
614             indexVLSN = new VLSN(indexVLSN.getSequence() + stride);
615         }
616 
617         VLSNBucket remainder = null;
618         int lastOffset;
619         if (newFirstIndex == -1) {
620 
621             /*
622              * None of the VLSNs represented by the strided file offsets are
623              * needed anymore. This bucket consists solely of the last
624              * VLSN->LSN pair.
625              */
626             lastOffset = fileOffsets.get(fileOffsets.size() - 1);
627             fileOffsets = new TruncateableList<Integer>();
628             fileOffsets.add(DbLsn.getFileOffsetAsInt(lastLsn));
629             firstVLSN = lastVLSN;
630         } else {
631             /* Move the still-valid mappings to a new list. */
632             assert (newFirstIndex > 0);
633             lastOffset = fileOffsets.get(newFirstIndex - 1);
634             TruncateableList<Integer> newFileOffsets =
635                 new TruncateableList<Integer>
636                 (fileOffsets.subList(newFirstIndex, fileOffsets.size()));
637             fileOffsets = newFileOffsets;
638             firstVLSN =  new VLSN((newFirstIndex * stride) +
639                                   firstVLSN.getSequence());
640         }
641 
642         if (!firstVLSN.equals(lastDuplicate.getNext())) {
643 
644             /*
645              * If lastDuplicate was not on the same stride boundary as our old
646              * bucket, we may have a broken bucket of mappings to preserve.
647              * Using our example numbers above, we still need to make sure
648              * there's a bucket that matches VLSNs 17, 18 19.
649              */
650             long scanStart =  DbLsn.makeLsn(fileNumber, lastOffset);
651             remainder = scanForNewBucket(envImpl,
652                                          lastDuplicate.getNext(),
653                                          firstVLSN.getPrev(),
654                                          scanStart);
655         }
656 
657         dirty = true;
658         return remainder;
659     }

688     /**
689      * Remove the mappings from this bucket that are for VLSNs >=
690      * startOfDelete.  Unlike removing from the head, we need not worry about
691      * breaking a bucket stride interval.
692      *
693      * If prevLsn is NULL_VLSN, we don't have a good value to cap the bucket.
694      * Instead, we'll have to delete the bucket back to whatever was the next
695      * available lsn. For example, suppose the bucket has these mappings.  This
696      * strange bucket (stride 25 is missing) is possible if vlsn 26 arrived
697      * early, out of order.
698      * 
699      *  in fileOffset: 10 -> 101
700      *  in fileOffset: 15 -> no offset
701      *  in fileOffset: 20 -> 201
702      *  lastVLSN->lastnLsn mapping  26 -> 250 
703      *
704      * If we have a prevLsn and the startOfDelete is 17, then we can create
705      * a new mapping
706      *  in fileOffset: 10 -> 101
707      *  in fileOffset: 15 -> no offset
708      *  lastVLSN->lastnLsn mapping  17 -> 190
709      *
710      * If we don't have a prevLsn, then we know that we have to cut the bucket
711      * back to the largest known mapping, losing many mappings along the way.
712      *  in fileOffset: 10 -> 101
713      *  lastVLSN->lastnLsn mapping  10 -> 101
714      *
715      * If we are deleting in the vlsn area between the last stride and the
716      * last offset, (i.e. vlsn 23 is the startOfDelete) the with and without
717      * prevLSn cases would look like this:
718      *
719      * (there is a prevLsn, and 23 is startDelete. No need to truncate
720      * anything) 
721      *  in fileOffset: 10 -> 101
722      *  in fileOffset: 15 -> no offset
723      *  in fileOffset: 20 -> 201
724      *  lastVLSN->lastnLsn mapping  23 -> prevLsn
725      *
726      * (there is no prevLsn, and 23 is startDelete) 
727      *  in fileOffset: 10 -> 101
728      *  in fileOffset: 15 -> no offset
729      *  in fileOffset: 20 -> 201
730      *  lastVLSN->lastnLsn mapping  20 -> 201
731      *
732      * @param startOfDelete is the VLSN that begins the range to delete,
733      *        inclusive
734      * @param prevLsn is the lsn of startOfDelete.getPrev(). We'll be using it
735      * to cap off the end of the bucket, by assigning it to the lastLsn field.
736      */
737     void removeFromTail(VLSN startOfDelete, long prevLsn) {
738 
739         if (empty()) {
740             return;
741         }
742 
743         if (lastVLSN.compareTo(startOfDelete) < 0) {
744             return;
745         }
746 
747         /* Delete all the mappings. */
748         if (firstVLSN.compareTo(startOfDelete) >= 0) {
749             lastVLSN = firstVLSN;
750             lastLsn = DbLsn.NULL_LSN;
751             fileOffsets.clear();
752             return;
753         }
754 
755         /* Delete some of the mappings. */
756         int deleteIndex = getGTEIndex(startOfDelete);
757 
758         /* 
759          * This should never happen, because the startOfDelete should be a vlsn
760          * that is >= the first vlsn and we handled the case where
761          * startOfDelete == firstVLSN already.)  Throw this exception to make
762          * debugging info available.
763          */
764         if (deleteIndex <= 0) {
765             throw EnvironmentFailureException.unexpectedState
766                 ("deleteIndex=" + deleteIndex +
767                  " startOfDelete=" + startOfDelete +
768                  " bucket=" + this);
769         }
770 
771         /* See if there are any fileoffsets to prune off. */
772         if (deleteIndex < fileOffsets.size()) {
773 
774             /*
775              * The startOfDeleteVLSN is a value between the firstVLSN and
776              * the last file offset.
777              */
778             if (prevLsn == DbLsn.NULL_LSN) {
779                 int lastPopulatedIndex = 
780                     findPopulatedIndex(deleteIndex-1, false);
781                 if (lastPopulatedIndex != (deleteIndex -1)) {
782                     deleteIndex = lastPopulatedIndex + 1;
783                 }
784             }
785             fileOffsets.truncate(deleteIndex);
786         } else {
787             /* 
788              * The startOfDelete vlsn is somewhere between the last file offset
789              * and the lastVLSN.
790              */
791             if (prevLsn == DbLsn.NULL_LSN) {
792                 int lastIndex = fileOffsets.size() - 1;
793                 int lastPopulatedIndex = findPopulatedIndex(lastIndex, false);
794                 if (lastPopulatedIndex < lastIndex) {
795                     fileOffsets.truncate(lastPopulatedIndex);
796                 }
797             }
798         }
799 
800         /* Now set the lastVLSN -> lastLSN mapping. */
801         if (prevLsn == DbLsn.NULL_LSN) {
802             lastVLSN = new VLSN(((fileOffsets.size()-1) * stride) +
803                                 firstVLSN.getSequence());
804             Integer lastOffset = fileOffsets.get(fileOffsets.size() - 1);
805             assert lastOffset != null;
806             lastLsn = DbLsn.makeLsn(fileNumber,  lastOffset);
807         } else {
808             lastVLSN = startOfDelete.getPrev();
809             lastLsn = prevLsn;
810         }
811         dirty = true;
812     }

847     /**
848      * Write this bucket to the mapping database using a cursor.  Note that
849      * this method must disable critical eviction. Critical eviction makes the
850      * calling thread search for a target IN node to evict. That target IN node
851      * may or may not be in the internal VLSN db.
852      * 
853      * For example, when a new, replicated LN is inserted or modified, a
854      * new VLSN is allocated. To do so, the app thread that is executing the
855      * operation 
856      *  A1. Takes a BIN latch on a BIN in a replicated db
857      *  A2. Takes the VLSNINdex mutex
858      *
859      * Anyone calling writeDatabase() has to take these steps:
860      *  B1. Take the VLSNIndex mutex
861      *  B2. Get a BIN latch for a BIN in the internal vlsn db.
862      *
863      * This difference in locking hierarchy could cause a deadlock except for
864      * the fact that A1 and B2 are guaranteed to be in different databases.  If
865      * writeDatabase() also did critical eviction, it would have a step where
866      * it tried to get a BIN latch on a replicated db, and we'd have a
867      * deadlock. [#18475]
868      */
869     void writeToDatabase(EnvironmentImpl envImpl, Cursor cursor) {
870 
871         if (!dirty) {
872             return;
873         }
874 
875         DatabaseEntry key = new DatabaseEntry();
876         DatabaseEntry data = new DatabaseEntry();
877 
878         LongBinding.longToEntry(firstVLSN.getSequence(), key);
879         VLSNBucketBinding bucketBinding = new VLSNBucketBinding();
880         bucketBinding.objectToEntry(this, data);
881 
882         DbInternal.getCursorImpl(cursor).setAllowEviction(false);
883         OperationStatus status = cursor.put(key, data);
884 
885         if (status != OperationStatus.SUCCESS) {
886             throw EnvironmentFailureException.unexpectedState
887                  (envImpl, "Unable to write VLSNBucket for file " +
888                  fileNumber + " status=" + status);
889         }
890         dirty = false;
891     }
