21 /**
22  * Indicates that a failure has occurred that could impact the {@code
23  * Environment} as a whole.  For failures that impact only the current
24  * operation and/or transaction, see {@link OperationFailureException}).  For
25  * an overview of all exceptions thrown by JE, see {@link DatabaseException}.
26  *
27  * <p>Depending on the nature of the failure, this exception may indicate that
28  * {@link Environment#close} must be called. The application should catch
29  * {@code EnvironmentFailureException} and then call {@link
30  * Environment#isValid}.  If {@code false} is returned, all {@code Environment}
31  * handles (instances) must be closed and re-opened in order to run recovery
32  * and continue operating.  If {@code true} is returned, the {@code
33  * Environment} can continue operating without being closed and re-opened.
34  * Also note that {@link Environment#isValid} may be called at any time, not
35  * just during exception handling.</p>
36  *
37  * <p>The use of the {@link Environment#isValid} method allows JE to determine
38  * dynamically whether the failure requires recovery or not, and allows for
39  * this determination to change in future releases.  Over time, internal
40  * improvements to error handling may allow more error conditions to be handled
41  * without invalidating the {@code Environment}.</p>
42  *
43  * <p>(Although this exception class extends {@link RunRecoveryException}, it
44  * does not always indicate that recovery is necessary, as described above.
45  * {@code RunRecoveryException} has been deprecated and {@code
46  * EnvironmentFailureException} should be used instead.)</p>
47  *
48  * <p>If an {@code EnvironmentFailureException} consistently occurs soon after
49  * opening the Environment, this may indicate a persistent problem.  It may
50  * indicate a system problem or a persistent storage problem.  In this case,
51  * human intervention is normally required and restoring from a backup may be
52  * necessary.</p>
53  *
54  * <p>Note that subclasses of {@code EnvironmentFailureException} indicate how
55  * to handle the exception in more specific ways.</p>
56  * <ul>
57  * <li>If {@code Thread.interrupt} is called for a thread performing JE
58  * operations, a {@link ThreadInterruptedException} is thrown.  Since
59  * interrupting a thread is intentional, it does not indicate a persistent
60  * problem and human intervention is not normally required.
61  * </li>
62  * <li>If an {@code IOException} occurs while writing to the JE log, a
63  * {@link LogWriteException} is thrown.  Although an {@code IOException} can
64  * occur for different reasons, it is a hint that the disk may be full and
65  * applications may wish to attempt recovery after making more disk space
66  * available.
67  * </li>
68  * <li>For replicated environments, see the subclasses of {@code
69  * EnvironmentFailureException} in the {@link com.sleepycat.je.rep} package for
70  * more information.  Such exceptions may require special handling.
71  * </li>
72  * </ul>
73  *
74  * <p>If {@link Environment#close} is not called after an {@code
75  * EnvironmentFailureException} invalidates the {@code Environment}, all
76  * subsequent method calls for the {@code Environment} will throw the same
77  * exception.  This provides more than one opportunity to catch and handle the
78  * specific exception subclass that caused the failure.</p>
79  */

152     public EnvironmentFailureException(EnvironmentImpl envImpl,
153                                        EnvironmentFailureReason reason,
154                                        String message,
155                                        Throwable cause) {
156         super(makeMsg(envImpl, reason, message, cause), cause);
157         this.reason = reason;
158 
159         if (reason.invalidatesEnvironment()) {
160             /* 
161              * If the environment exists, invalidate it. Note that there are
162              * cases, such as if an exception occurred during recovery, that
163              * the environment will not exist.
164              */
165             if (envImpl == null) {
166                 if (reason.envShouldExist()) {
167                     assert envImpl != null;
168                 }
169             } else {
170                 envImpl.invalidate(this);
171             }
172         }
173     }

175     private static String makeMsg(EnvironmentImpl envImpl,
176                                   EnvironmentFailureReason reason,
177                                   String message,
178                                   Throwable cause) {
179         StringBuilder s = new StringBuilder(300);
180         if (envImpl != null) {
181             s.append(envImpl.getName()).append(" ");
182         }
183 
184         if (message != null) {
185             s.append(message);
186             s.append(' ');
187         } else if (cause != null) {
188             s.append(cause.toString());
189             s.append(' ');
190         }
191         assert reason != null;
192         s.append(reason);
193 
194         /*
195          * Use the current environment status for reporting in the exception
196          * message.  This is more information than simply whether this
197          * exception caused an invalidate, since previous exceptions may have
198          * occurred.
199          */
200         if (reason.invalidatesEnvironment() ||
201             (envImpl != null && !envImpl.isValid())) {
202             s.append(" Environment is invalid and must be closed.");
203         }
204         return s.toString();
205     }

269      * Whether the EnvironmentFailureException indicates that the log is
270      * corrupt, meaning that a network restore (or restore from backup) should
271      * be performed.
272      * <p>
273      * This method currently returns true only when corruption has been
274      * detected and is persistent. This may have been detected by verifying
275      * checksums in the disk data log, and in this case the corruption
276      * indicates a media/disk failure. The checksum error may have
277      * been detected when accessing data normally via the JE API, or by the
278      * background data verifier (see {@link EnvironmentConfig#VERIFY_LOG}).
279      * Or a persistent Btree corruption may have been detected by the data
280      * verifier (see {@link EnvironmentConfig#VERIFY_BTREE}) or by the
281      * {@link Environment#verify(VerifyConfig, PrintStream)} or
282      * {@link Database#verify(VerifyConfig)} methods. This method will
283      * returns true in all such cases.
284      * <p>
285      * Additionally, when a persistent corruption is detected and the
286      * Environment is open for read-write access,  a marker file named
287      * 7fffffff.jdb is created in the Environment directory that will
288      * prevent re-opening the environment. If an attempt is made to
289      * re-open the Environment, the original EnvironmentFailureException
290      * will be thrown. This is meant to safeguard against using a corrupt
291      * environment when the original exception is accidentally overlooked.
292      * While the marker file can be deleted to allow re-opening the
293      * environment, this is normally unsafe and is not recommended.
294      *
295      * @return true if the environment is corrupt.
296      *
297      * @since 7.3
298      */

301                reason == EnvironmentFailureReason.BTREE_CORRUPTION;
302     }
