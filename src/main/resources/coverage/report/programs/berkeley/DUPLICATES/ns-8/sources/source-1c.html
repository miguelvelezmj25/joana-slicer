27 /**
28  * Specifies the attributes of an environment.
29  *
30  * <p>To change the default settings for a database environment, an application
31  * creates a configuration object, customizes settings and uses it for
32  * environment construction. The set methods of this class validate the
33  * configuration values when the method is invoked.  An
34  * IllegalArgumentException is thrown if the value is not valid for that
35  * attribute.</p>
36  *
37  * <p>Most parameters are described by the parameter name String constants in
38  * this class. These parameters can be specified or individually by calling
39  * {@link #setConfigParam}, through a Properties object passed to {@link
40  * #EnvironmentConfig(Properties)}, or via properties in the je.properties
41  * files located in the environment home directory.</p>
42  *
43  * <p>For example, an application can change the default btree node size
44  * with:</p>
45  *
46  * <pre>
47  *     envConfig.setConfigParam(EnvironmentConfig.LOCK_TIMEOUT, "250 ms");
48  * </pre>
49  *
50  * <p>Some commonly used environment attributes have convenience setter/getter
51  * methods defined in this class.  For example, to change the default
52  * lock timeout setting for an environment, the application can instead do
53  * the following:</p>
54  * <pre class=code>
55  *     // customize an environment configuration
56  *     EnvironmentConfig envConfig = new EnvironmentConfig();
57  *     // will throw if timeout value is invalid
58  *     envConfig.setLockTimeout(250, TimeUnit.MILLISECONDS);
59  *     // Open the environment using this configuration.
60  *     Environment myEnvironment = new Environment(home, envConfig);
61  * </pre>
62  *
63  * <p>Parameter values are applied using this order of precedence:</p>
64  * <ol>
65  *     <li>Configuration parameters specified in je.properties take first
66  *      precedence.</li>
67  *     <li>Configuration parameters set in the EnvironmentConfig object used at
68  *     Environment construction are next.</li>
69  *     <li>Any configuration parameters not set by the application are set to
70  *     system defaults, described along with the parameter name String
71  *     constants in this class.</li>
72  * </ol>
73  *
74  * <p>However, a small number of parameters do not have string constants in
75  * this class, and cannot be set using {@link #setConfigParam}, a Properties
76  * object, or the je.properties file. These parameters can only be changed
77  * via the following setter methods:</p>
78  * <ul>
79  *     <li>{@link #setAllowCreate}</li>
80  *     <li>{@link #setCacheMode}</li>
81  *     <li>{@link #setClassLoader}</li>
82  *     <li>{@link #setCustomStats}</li>
83  *     <li>{@link #setExceptionListener}</li>
84  *     <li>{@link #setLoggingHandler}</li>
85  *     <li>{@link #setNodeName}</li>
86  *     <li>{@link #setRecoveryProgressListener}</li>
87  * </ul>
88  *
89  * <p>An EnvironmentConfig can be used to specify both mutable and immutable
90  * environment properties.  Immutable properties may be specified when the
91  * first Environment handle (instance) is opened for a given physical
92  * environment.  When more handles are opened for the same environment, the
93  * following rules apply:</p>
94  * <ol>
95  *     <li>Immutable properties must equal the original values specified when
96  * constructing an Environment handle for an already open environment.  When a
97  * mismatch occurs, an exception is thrown.</li>
98  *     <li>Mutable properties are ignored when constructing an Environment handle
99  * for an already open environment.</li>
100  * </ol>
101  *
102  * <p>After an Environment has been constructed, its mutable properties may be
103  * changed using {@link Environment#setMutableConfig}.  See {@link
104  * EnvironmentMutableConfig} for a list of mutable properties; all other
105  * properties are immutable.  Whether a property is mutable or immutable is
106  * also described along with the parameter name String constants in this
107  * class.</p>
108  *
109  * <h4>Getting the Current Environment Properties</h4>
110  *
111  * To get the current "live" properties of an environment after constructing it
112  * or changing its properties, you must call {@link Environment#getConfig} or
113  * {@link Environment#getMutableConfig}.  The original EnvironmentConfig or
114  * EnvironmentMutableConfig object used to set the properties is not kept up to
115  * date as properties are changed, and does not reflect property validation or
116  * properties that are computed.
117  *
118  * <h4><a name="timeDuration">Time Duration Properties</a></h4>
119  *
120  * <p>Several environment and transaction configuration properties are time
121  * durations.  For these properties, a time unit is specified along with an
122  * integer duration value.</p>
123  *
124  * <p>When specific setter and getter methods exist for a time duration
125  * property, these methods have a {@link TimeUnit} argument.  Examples are
126  * {@link #setLockTimeout(long,TimeUnit)} and {@link
127  * #getLockTimeout(TimeUnit)}.  Note that the {@link TimeUnit} argument may
128  * be null only when the duration value is zero; there is no default unit that
129  * is used when null is specified.</p>
130  *
131  * <p>When a time duration is specified as a string value, the following format
132  * is used.</p>
133  *
134  * <pre>   {@code <value> [ <whitespace> <unit> ]}</pre>
135  *
136  * <p>The {@code <value>} is an integer.  The {@code <unit>} name, if present,
137  * must be preceded by one or more spaces or tabs.</p>
138  *
139  * <p>The following {@code <unit>} names are allowed.  Both {@link TimeUnit}
140  * names and IEEE standard abbreviations are allowed.  Unit names are case
141  * insensitive.</p>
142  *
143  * <table border="true">
144  * <tr><th>IEEE abbreviation</th>
145  *     <th>TimeUnit name</td>
146  *     <th>Definition</th>
147  * </tr>
148  * <tr><td>{@code ns}</td>
149  *     <td>{@code NANOSECONDS}</td>
150  *     <td>one billionth (10<sup>-9</sup>) of a second</td>
151  * </tr>
152  * <tr><td>{@code us}</td>
153  *     <td>{@code MICROSECONDS}</td>
154  *     <td>one millionth (10<sup>-6</sup>) of a second</td>
155  * </tr>
156  * <tr><td>{@code ms}</td>
157  *     <td>{@code MILLISECONDS}</td>
158  *     <td>one thousandth (10<sup>-3</sup>) of a second</td>
159  * </tr>
160  * <tr><td>{@code s}</td>
161  *     <td>{@code SECONDS}</td>
162  *     <td>1 second</td>
163  * </tr>
164  * <tr><td>{@code min}</td>
165  *     <td>&nbsp;</td>
166  *     <td>60 seconds</td>
167  * </tr>
168  * <tr><td>{@code h}</td>
169  *     <td>&nbsp;</td>
170  *     <td>3600 seconds</td>
171  * </tr>
172  * </table>
173  *
174  * <p>Examples are:</p>
175  * <pre>
176  * 3 seconds
177  * 3 s
178  * 500 ms
179  * 1000000 (microseconds is implied)
180  * </pre>
181  *
182  * <p>The maximum duration value is currently Integer.MAX_VALUE milliseconds.
183  * This translates to almost 25 days (2147483647999999 ns, 2147483647999 us,
184  * 2147483647 ms, 2147483 s, 35791 min, 596 h).</p>

187  * This default is supported for compatibility with JE 3.3 and earlier.  In JE

189  * implicitly specified in microseconds.  The older methods that do not have a
190  * {@link TimeUnit} argument, such as {@link #setLockTimeout(long)} and {@link

318      * Configures the number of bytes to be used as a secondary, off-heap cache.
319      *
320      * The off-heap cache is used to hold record data and Btree nodes when
321      * these are evicted from the "main cache" because it overflows. Eviction
322      * occurs according to an LRU algorithm and takes into account the user-
323      * specified {@link CacheMode}. When the off-heap cache overflows, eviction
324      * occurs there also according to the same algorithm.
325      * <p>
326      * The main cache is in the Java heap and consists primarily of the Java
327      * objects making up the in-memory Btree data structure. Btree objects are
328      * not serialized the main cache, so no object materialization is needed to
329      * access the Btree there. Access to records in the main cache is therefore
330      * very fast, but the main cache has drawbacks as well: 1) The larger the
331      * main cache, the more likely it is to have Java GC performance problems.
332      * 2) When the Java heap exceeds 32GB, the "compressed OOPs" setting no
333      * longer applies and less data will fit in the same amount of memory. For
334      * these reasons, JE applications often configure a heap of 32GB or less,
335      * and a main cache that is significantly less than 32GB, leaving any
336      * additional machine memory for use by the file system cache.
337      * <p>
338      * The use of the file system cache has performance benefits, but
339      * also has its own drawbacks: 1) There is a significant redundancy
340      * between the main cache and the file system cache because all data and
341      * Btree information that is logged (written) by JE appears in the file
342      * system and may also appear in the main cache. 2) It is not possible
343      * for <em>dirty</em> Btree information to be placed in the file system
344      * cache without logging it, this logging may be otherwise unnecessary, and
345      * the logging creates additional work for the JE cleaner; in other words,
346      * the size of the main cache alone determines the maximum size of the
347      * in-memory "dirty set".
348      * <p>
349      * The off-heap cache is stored outside the Java heap using a native
350      * platform memory allocator. The current implementation relies on
351      * internals that are specific to the Oracle and IBM JDKs; however, a
352      * memory allocator interface that can be implemented for other situations
353      * is being considered for a future release. Records and Btree objects are
354      * serialized when they are placed in the off-heap cache, and they must be
355      * materialized when they are moved back to the main cache in order to
356      * access them. This serialization and materialization adds some CPU
357      * overhead and thread contention, as compared to accessing data directly
358      * in the main cache. The off-heap cache can contain dirty Btree
359      * information, so it can be used to increase the maximum size of the
360      * in-memory "dirty set".
361      * <p>
362      * NOTE: If an off-heap cache is configured but cannot be used because
363      * that native allocator is not available in the JDK that is used, an
364      * {@code IllegalStateException} will be thrown by the {@link Environment}
365      * or {@link com.sleepycat.je.rep.ReplicatedEnvironment} constructor. In
366      * the current release, this means that the {@code sun.misc.Unsafe} class
367      * must contain the {@code allocateMemory} method and related methods, as
368      * defined in the Oracle JDK.
369      * <p>
370      * When configuring an off-heap cache you can think of the performance
371      * trade-offs in two ways. First, if the off-heap cache is considered to be
372      * a replacement for the file system cache, the serialization and
373      * materialization overhead is not increased. In this case, the use of
374      * the off-heap cache is clearly beneficial, and using the off-heap cache
375      * "instead of" the file system cache is normally recommended. Second, the
376      * off-heap cache can be used along with a main cache that is reduced in
377      * size in order to compensate for Java GC problems. In this case, the
378      * trade-off is between the additional serialization, materialization and
379      * contention overheads of the off-heap cache, as compared to the Java GC
380      * overhead.
381      * <p>
382      * When dividing up available memory for the JVM heap, the off-heap cache,
383      * and for other uses, please be aware that the file system cache and the
384      * off-heap cache are different in one important respect. The file system
385      * cache automatically shrinks when memory is needed by the OS or other
386      * processes, while the off-heap cache does not. Therefore, it is best to
387      * be conservative about leaving memory free for other uses, and it is not
388      * a good idea to size the off-heap cache such that all machine memory will
389      * be allocated. If off-heap allocations or other allocations fail because
390      * there is no available memory, the process is likely to die without any
391      * exception being thrown. In one test on Linux, for example, the process
392      * was killed abruptly by the OS and the only indication of the problem was
393      * the following shown by {@code dmesg}.
394      * <pre>
395      * Out of memory: Kill process 28768 (java) score 974 or sacrifice child
396      * Killed process 28768 (java)
397      *    total-vm:278255336kB, anon-rss:257274420kB, file-rss:0kB
398      * </pre>
399      * <p>
400      * WARNING: Although this configuration property is mutable, it cannot be
401      * changed from zero to non-zero, or non-zero to zero. In other words, the
402      * size of the off-heap cache can be changed after initially configuring a
403      * non-zero size, but the off-heap cache cannot be turned on and off
404      * dynamically. An attempt to do so will cause an {@code
405      * IllegalArgumentException} to be thrown by the {@link Environment} or
406      * {@link com.sleepycat.je.rep.ReplicatedEnvironment} constructor.
407      *
408      * <p><table border="1">
409      * <tr><td>Name</td><td>Type</td><td>Mutable</td>
410      * <td>Default</td><td>Minimum</td><td>Maximum</td></tr>
411      * <tr>
412      * <td>{@value}</td>
413      * <td>Long</td>
414      * <td>Yes</td>
415      * <td>0</td>
416      * <td>0</td>
417      * <td>-none-</td>
418      * </tr>
419      * </table></p>
420      *
421      * @see #setOffHeapCacheSize(long)
422      *
423      * @see <a href="EnvironmentStats.html#cacheSizing">Cache Statistics:
424      * Sizing</a>
425      */

462      * An upper limit on the number of bytes used for data storage. Works
463      * with {@link #FREE_DISK} to define the storage limit. If the limit is
464      * exceeded, write operations will be prohibited.
465      * <p>
466      * If set to zero (the default), no usage limit is enforced, meaning that
467      * all space on the storage volume, minus {@link #FREE_DISK}, may be used.
468      * If MAX_DISK is non-zero, FREE_DISK is subtracted from MAX_DISK to
469      * determine the usage threshold for prohibiting write operations. If
470      * multiple JE environments share the same storage volume, setting MAX_DISK
471      * to a non-zero value is strongly recommended.
472      *
473      *   <p style="margin-left: 2em">Note: An exception to the rule above is
474      *   when MAX_DISK is less than or equal to 10GB and FREE_DISK is not
475      *   explicitly specified. See {@link #FREE_DISK} more information.</p>
476      *
477      * Both the FREE_DISK and MAX_DISK thresholds (if configured) are checked
478      * during a write operation. If either threshold is crossed, the behavior
479      * of the JE environment is as follows:
480      * <ul>
481      *     <li>
482      *     Application write operations will throw {@link DiskLimitException}.
483      *     DiskLimitException extends {@link OperationFailureException} and
484      *     will invalidate the transaction, but will not invalidate the
485      *     environment. Read operations may continue even when write operations
486      *     are prohibited.
487      *     </li>
488      *     <li>
489      *     When using NoSQL DB, the above item applies to client CRUD
490      *     operations as well as operations performed on internal metadata.
491      *     When a disk limit is violated, NoSQL DB will throw exceptions for
492      *     client write operations and for operations that update internal
493      *     metadata. Related exceptions may be logged for other internal write
494      *     operations. Such exceptions will be derived from the JE
495      *     DiskLimitException.
496      *     </li>
497      *     <li>
498      *     {@link Environment#checkpoint}, {@link Environment#sync} and
499      *     {@link Database#sync} will throw DiskLimitException.
500      *     </li>
501      *     <li>
502      *     {@link Environment#close} may throw DiskLimitException when a final
503      *     checkpoint is performed. However, the environment will be properly
504      *     closed in other respects.
505      *     </li>
506      *     <li>
507      *     The JE evictor will not log dirty nodes when the cache overflows
508      *     and therefore dirty nodes cannot be evicted from cache. So
509      *     although read operations are allowed, cache thrashing may occur if
510      *     all INs do not fit in cache as {@link DbCacheSize recommended}.
511      *     </li>
512      *     <li>
513      *     In an HA environment a disk limit may be violated on a replica node
514      *     but not the master node. In this case, a DiskLimitException will not
515      *     be thrown by a write operation on the master node. Instead,
516      *     {@link com.sleepycat.je.rep.InsufficientAcksException} or
517      *     {@link com.sleepycat.je.rep.InsufficientReplicasException} will be
518      *     thrown if the {@link Durability#getReplicaAck() ack requirements}
519      *     are not met.
520      *     </li>
521      * </ul>
522      * <p>
523      * JE uses a log structured storage system where data files often become
524      * gradually obsolete over time (see {@link #CLEANER_MIN_UTILIZATION}). The
525      * JE cleaner is responsible for reclaiming obsolete space by cleaning and
526      * deleting data files. In a standalone (non-HA) environment, data files
527      * are normally deleted quickly after being cleaned, but may be reserved
528      * and protected temporarily by a {@link com.sleepycat.je.util.DbBackup} or
529      * {@link DiskOrderedCursor}. These reserved files will be deleted as soon
530      * as they are no longer protected.
531      * <p>
532      * In an HA environment, JE will retain as many reserved files as possible
533      * to support replication to nodes that are out of contact. All cleaned
534      * files are reserved (not deleted) until approaching a disk limit, at
535      * which time they are deleted, as long as they are not protected.
536      * Reserved files are protected when they are needed for
537      * replication to active nodes or for feeding an active network restore.
538      * <p>
539      * For more information on reserved and protected data files, see
540      * {@link EnvironmentStats#getActiveLogSize()},
541      * {@link EnvironmentStats#getReservedLogSize()},
542      * {@link EnvironmentStats#getProtectedLogSize()},
543      * {@link EnvironmentStats#getProtectedLogSizeMap()},
544      * {@link EnvironmentStats#getAvailableLogSize()} and
545      * {@link EnvironmentStats#getTotalLogSize}.
546      * <p>
547      * When multiple JE environments share the same storage volume, the
548      * FREE_DISK amount will be maintained for each environment. The following
549      * scenario illustrates use of a single shared volume with capacity 300GB:
550      * <ul>
551      *     <li>
552      *     JE-1 and JE-2 each have MAX_DISK=100GB and FREE_DISK=5GB,
553      *     </li>
554      *     <li>
555      *     100GB is used for fixed miscellaneous storage.
556      *     </li>
557      * </ul>
558      * <p>
559      * Each JE environment will use no more than 95GB each, so at least 10GB
560      * will remain free overall. In other words, if both JE environments reach
561      * their threshold and write operations are prohibited, each JE environment
562      * will have 5GB of free space for recovery (10GB total).
563      * <p>
564      * On the other hand, when an external service is also consuming disk
565      * space and its usage of disk space is variable over time, the situation
566      * is more complex and JE cannot always guarantee that FREE_DISK is
567      * honored. The following scenario includes multiple JE environments as
568      * well an external service, all sharing a 300GB volume.
569      * <ul>
570      *     <li>
571      *     JE-1 and JE-2 each have MAX_DISK=100GB and FREE_DISK=5GB,
572      *     </li>
573      *     <li>
574      *     an external service is expected to use up to 50GB, and
575      *     </li>
576      *     <li>
577      *     50GB is used for fixed miscellaneous storage.
578      *     </li>
579      * </ul>
580      * <p>
581      * Assuming that the external service stays within its 50GB limit then, as
582      * the previous example, each JE environment will normally use no more than
583      * 95GB each, and at least 10GB will remain free overall. However, if the
584      * external service exceeds its threshold, JE will make a best effort to
585      * prohibit write operations in order to honor the FREE_DISK limit, but
586      * this is not always possible, as illustrated by the following sequence
587      * of events:
588      * <ul>
589      *     <li>
590      *     If the external service uses all its allocated space, 50GB, and JE
591      *     environments are each using 75GB, then there will be 50GB free
592      *     overall (25GB for each JE environment). Write operations are allowed
593      *     in both JE environments.
594      *     </li>
595      *     <li>
596      *     If the external service then exceeds its limit by 25GB and uses
597      *     75GB, there will only 25GB free overall. But each JE environment is
598      *     still under its 90GB limit and there is still more than 5GB free
599      *     overall, so write operations are still allowed.
600      *     </li>
601      *     <li>
602      *     If each JE environment uses an additional 10GB of space, there will
603      *     only be 5GB free overall. Each JE environment is using only 85GB,
604      *     which is under its 95GB limit. But the 5GB FREE_DISK limit for the
605      *     volume overall has been reached and therefore JE write operations
606      *     will be prohibited.
607      *     </li>
608      * </ul>
609      * Leaving only 5GB of free space in the prior scenario is not ideal, but
610      * it is at least enough for one JE environment at a time to be recovered.
611      * The reality is that when an external entity exceeds its expected disk
612      * usage, JE cannot always compensate. For example, if the external service
613      * continues to use more space in the scenario above, the volume will
614      * eventually be filled completely.
615      *

1005      * modify a database will fail.
1006      *
1007      * <p>A read-only environment has several limitations and is recommended
1008      * only in special circumstances.  Note that there is no performance
1009      * advantage to opening an environment read-only.</p>
1010      *
1011      * <p>The primary reason for opening an environment read-only is to open a
1012      * single environment in multiple JVM processes.  Only one JVM process at a
1013      * time may open the environment read-write.  See {@link
1014      * EnvironmentLockedException}.</p>
1015      *
1016      * <p>When the environment is open read-only, the following limitations
1017      * apply.</p>
1018      * <ul>
1019      * <li>In the read-only environment no writes may be performed, as
1020      * expected, and databases must be opened read-only using {@link
1021      * DatabaseConfig#setReadOnly}.</li>
1022      * <li>The read-only environment receives a snapshot of the data that is
1023      * effectively frozen at the time the environment is opened. If the
1024      * application has the environment open read-write in another JVM process
1025      * and modifies the environment's databases in any way, the read-only
1026      * version of the data will not be updated until the read-only JVM process
1027      * closes and reopens the environment (and by extension all databases in
1028      * that environment).</li>
1029      * <li>If the read-only environment is opened while the environment is in
1030      * use by another JVM process in read-write mode, opening the environment
1031      * read-only (recovery) is likely to take longer than it does after a clean
1032      * shutdown.  This is due to the fact that the read-write JVM process is
1033      * writing and checkpoints are occurring that are not coordinated with the
1034      * read-only JVM process.  The effect is similar to opening an environment
1035      * after a crash.</li>
1036      * <li>In a read-only environment, the JE cache will contain information
1037      * that cannot be evicted because it was reconstructed by recovery and
1038      * cannot be flushed to disk.  This means that the read-only environment
1039      * may not be suitable for operations that use large amounts of memory, and
1040      * poor performance may result if this is attempted.</li>
1041      * <li>In a read-write environment, the log cleaner will be prohibited from
1042      * deleting log files for as long as the environment is open read-only in
1043      * another JVM process.  This may cause disk usage to rise, and for this
1044      * reason it is not recommended that an environment is kept open read-only
1045      * in this manner for long periods.</li>
1046      * </ul>
1047      *
1048      * <p>For these reasons, it is recommended that a read-only environment be
1049      * used only for short periods and for operations that are not performance
1050      * critical or memory intensive.  With few exceptions, all application
1051      * functions that require access to a JE environment should be built into a
