29     /**
30      * In all src and test code, the String(byte[], ...) constructor and
31      * String.getBytes method must always be passed a Charset, to avoid
32      * portability issues.  Otherwise, portability issues will occur when
33      * running on a JVM plataform with a non-western default charset, the
34      * EBCDIC encoding (on z/OS), etc.  [#20967]
35      * <p>
36      * In most cases, the UTF8 or ASCII charset should be used for portability.
37      * UTF8 should be used when any character may be represented.  ASCII can be
38      * used when all characters are in the ASCII range.  The default charset
39      * should only be used when handling user-input data directly, e.g.,
40      * console input/output or user-visible files.
41      * <p>
42      * Rather than passing the charset as a string (getBytes("UTF-8")), the
43      * Charset objects defined here should be passed (getBytes(UTF8)).  Not
44      * only is using a Charset object slightly more efficient because it avoids
45      * a lookup, even more importantly it avoids having to clutter code with a
46      * catch for java.io.UnsupportedEncodingException, which should never be
47      * thrown for the "UTF-8" or "US-ASCII" charsets.
48      */
49     public static byte[] toUTF8(String str) {
50         try {
51             return str.getBytes("UTF-8");
52         } catch (UnsupportedEncodingException e) {
53             /* Should never happen. */
54             throw new RuntimeException(e);
55         }
56     }
