18 /**
19  * Thrown when multiple threads are competing for a lock and the lock timeout
20  * interval is exceeded for the current operation. This is normally because
21  * another transaction or cursor holds a lock for longer than the timeout
22  * interval. It may also occur if the application fails to close a cursor, or
23  * fails to commit or abort a transaction, since any locks held by the cursor
24  * or transaction will be held indefinitely.
25  *
26  * <p>This exception is not thrown if a deadlock is detected, even if the
27  * timeout elapses before the deadlock is broken. If a deadlock is detected,
28  * {@link DeadlockException} is always thrown instead.</p>
29  *
30  * <p>The lock timeout interval may be set using
31  * {@link EnvironmentConfig#setLockTimeout} or
32  * {@link Transaction#setLockTimeout}.</p>
33  *
34  * <p>The {@link Transaction} handle is invalidated as a result of this
35  * exception.</p>
36  *
37  * <p>Normally, applications should catch the base class {@link
38  * LockConflictException} rather than catching one of its subclasses.  All lock
39  * conflicts are typically handled in the same way, which is normally to abort
40  * and retry the transaction.  See {@link LockConflictException} for more
41  * information.</p>

44  */
