21 /**
22  * Encodes database key and data items as a byte array.
23  *
24  * <p>Storage and retrieval for the {@link Database Database}
25  * and {@link Cursor Cursor} methods are based on key/data
26  * pairs. Both key and data items are represented by DatabaseEntry objects.
27  * Key and data byte arrays may refer to arrays of zero length up to arrays of
28  * essentially unlimited length.</p>
29  *
30  * <p>The DatabaseEntry class provides simple access to an underlying object
31  * whose elements can be examined or changed.  DatabaseEntry objects can be
32  * subclassed, providing a way to associate with it additional data or
33  * references to other structures.</p>
34  *
35  * <p>Access to DatabaseEntry objects is not re-entrant. In particular, if
36  * multiple threads simultaneously access the same DatabaseEntry object using
37  * {@link Database Database} or {@link Cursor
38  * Cursor} methods, the results are undefined.</p>
39  *
40  * <p>DatabaseEntry objects may be used in conjunction with the object mapping
41  * support provided in the {@link com.sleepycat.bind} package.</p>
42  *
43  * <h3><a name="params">Input and Output Parameters</a></h3>
44  *
45  * <p>DatabaseEntry objects are used for both input values (for example, when
46  * writing to a database or specifying a search parameter) and output values
47  * (for example, when reading from a database). For every CRUD method
48  * ({@code get}, {@code put}, etc), each of the method's DatabaseEntry
49  * parameters ({@code key}, {@code data}, etc) may be input or output
50  * parameters, and this is specified by the method's documentation.</p>
51  *
52  * <h4><a name="inParam">Input Parameters</a></h4>
53  *
54  * <p>An input parameter is required by the JE method. The parameter may not be
55  * null, and the caller is also responsible for initializing the data of the
56  * DatabaseEntry to a non-null byte array.</p>
57  *
58  * <p>Input parameters normally may not be {@link #setPartial(int,int,boolean)
59  * partial}. However, this is allowed under certain circumstances, namely
60  * the {@link Cursor#putCurrent} method allows specifying a partial data
61  * parameter in order to update only part of the record's data value. Input
62  * parameters are NOT allowed to be partial unless this is explicitly stated in
63  * the method documentation.</p>
64  *
65  * <p>Although an input parameter is always used for input, in some cases it
66  * may be also used for output. For example, the {@link
67  * Cursor#getSearchKeyRange} method is passed a key parameter that is used as
68  * input, but since a record with a different key (greater or equal to the key
69  * given) may be found, the key parameter is also used to return the key
70  * that was found. Such parameters are documented as "input/output"
71  * parameters.</p>
72  *
73  * <p>Another example is when a custom key comparator is used and a key
74  * parameter is passed to a search method. The input parameter may match a
75  * record's key even if the bytes are not equal, and the key of the record
76  * found will be returned via the parameter. The same thing is true of data (or
77  * primary key) parameters when a custom duplicate comparator is used. Because
78  * of this, all input parameters of "get" methods can potentially be used for
79  * output, however, they are not explicitly documented to be input/output
80  * parameters.</p>
81  *
82  * <h4><a name="outParam">Output Parameters</a></h4>
83  *
84  * <p>An output parameter is not required by the JE method. It is used to
85  * optionally return a value to the caller. Null may be passed for the
86  * parameter if no returned value is needed. Passing null is a common way to
87  * optimize read operations when only the record's key, and not the record's
88  * data, is required. By passing null for the data parameter, a read from
89  * disk can be avoided when the data is not already cached. In addition, all
90  * output parameters may be {@link #setPartial(int,int,boolean) partial} to
91  * allow only returning a part of the data byte array. See <a
92  * href="Cursor.html#partialEntry">Using Null and Partial DatabaseEntry
93  * Parameters</a> for more information.</p>
94  *
95  * <p>For output parameters, the byte array specified by the caller will not be
96  * used and may be null. The JE method will will always allocate a new byte
97  * array. Therefore, after calling a method that returns output parameters,
98  * the application can safely keep a reference to the byte array returned by
99  * {@link #getData} without danger that the array will be overwritten in a
100  * subsequent call.</p>
101  *
102  * <p>Historical note: Prior to JE 7.0, null could not be passed for output
103  * parameters. Instead, {@code DatabaseEntry.setPartial(0, 0, true)} was called
104  * for a data parameter to avoid reading the record's data. Now, null can be
105  * passed instead.</p>
106  *
107  * <h3>Offset and Size Properties</h3>
108  *
109  * <p>By default the Offset property is zero and the Size property is the
110  * length of the byte array.  However, to allow for optimizations involving the
111  * partial use of a byte array, the Offset and Size may be set to non-default
112  * values.</p>
113  *
114  * <p>For output parameters, the Size will always be set to the length of the
115  * byte array and the Offset will always be set to zero.</p>
116  *
117  * <p>However, for input parameters the Offset and Size are set to non-default
118  * values by the built-in tuple and serial bindings. For example, with a tuple
119  * or serial binding the byte array is grown dynamically as data is output, and
120  * the Size is set to the number of bytes actually used. For a serial binding,
121  * the Offset is set to a non-zero value in order to implement an optimization
122  * having to do with the serialization stream header.</p>
123  *
124  * <p>WARNING: In callbacks that are passed DatabaseEntry parameters, the
125  * application should always honor the Size and Offset properties, rather than
126  * assuming they have default values.</p>
127  */
