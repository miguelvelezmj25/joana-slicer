46 /**
47  * The secondary index for an entity class and a secondary key.
48  *
49  * <p>{@code SecondaryIndex} objects are thread-safe.  Multiple threads may
50  * safely call the methods of a shared {@code SecondaryIndex} object.</p>
51  *
52  * <p>{@code SecondaryIndex} implements {@link EntityIndex} to map the
53  * secondary key type (SK) to the entity type (E).  In other words, entities
54  * are accessed by secondary key values.</p>
55  *
56  * <p>The {@link SecondaryKey} annotation may be used to define a secondary key
57  * as shown in the following example.</p>
58  *
59  * <pre class="code">
60  * {@literal @Entity}
61  * class Employee {
62  *
63  *     {@literal @PrimaryKey}
64  *     long id;
65  *
66  *     {@literal @SecondaryKey(relate=MANY_TO_ONE)}
67  *     String department;
68  *
69  *     String name;
70  *
71  *     private Employee() {}
72  * }</pre>
73  *
74  * <p>Before obtaining a {@code SecondaryIndex}, the {@link PrimaryIndex} must
75  * be obtained for the entity class.  To obtain the {@code SecondaryIndex} call
76  * {@link EntityStore#getSecondaryIndex EntityStore.getSecondaryIndex}, passing
77  * the primary index, the secondary key class and the secondary key name.  For
78  * example:</p>
79  *
80  * <pre class="code">
81  * EntityStore store = new EntityStore(...);
82  *
83  * {@code PrimaryIndex<Long, Employee>} primaryIndex =
84  *     store.getPrimaryIndex(Long.class, Employee.class);
85  *
86  * {@code SecondaryIndex<String, Long, Employee>} secondaryIndex =
87  *     store.getSecondaryIndex(primaryIndex, String.class, "department");</pre>
88  *
89  * <p>Since {@code SecondaryIndex} implements the {@link EntityIndex}
90  * interface, it shares the common index methods for retrieving and deleting
91  * entities, opening cursors and using transactions.  See {@link EntityIndex}
92  * for more information on these topics.</p>
93  *
94  * <p>{@code SecondaryIndex} does <em>not</em> provide methods for inserting
95  * and updating entities.  That must be done using the {@link
96  * PrimaryIndex}.</p>
97  *
98  * <p>Note that a {@code SecondaryIndex} has three type parameters {@code <SK,
99  * PK, E>} or in the example {@code <String, Long, Employee>} while a {@link
100  * PrimaryIndex} has only two type parameters {@code <PK, E>} or {@code <Long,
101  * Employee>}.  This is because a {@code SecondaryIndex} has an extra level of
102  * mapping:  It maps from secondary key to primary key, and then from primary
103  * key to entity.  For example, consider this entity:</p>
104  *
105  * <div><table class="code" border="1" summary="">
106  *   <tr><th>ID</th><th>Department</th><th>Name</th></tr>
107  *   <tr><td>1</td><td>Engineering</td><td>Jane Smith</td></tr>
108  * </table></div>
109  *
110  * <p>The {@link PrimaryIndex} maps from id directly to the entity, or from
111  * primary key 1 to the "Jane Smith" entity in the example.  The {@code
112  * SecondaryIndex} maps from department to id, or from secondary key
113  * "Engineering" to primary key 1 in the example, and then uses the {@code
114  * PrimaryIndex} to map from the primary key to the entity.</p>
115  *
116  * <p>Because of this extra type parameter and extra level of mapping, a {@code
117  * SecondaryIndex} can provide more than one mapping, or view, of the entities
118  * in the primary index.  The main mapping of a {@code SecondaryIndex} is to
119  * map from secondary key (SK) to entity (E), or in the example, from the
120  * String department key to the Employee entity.  The {@code SecondaryIndex}
121  * itself, by implementing {@code EntityIndex<SK, E>}, provides this
122  * mapping.</p>
123  *
124  * <p>The second mapping provided by {@code SecondaryIndex} is from secondary
125  * key (SK) to primary key (PK), or in the example, from the String department
126  * key to the Long id key.  The {@link #keysIndex} method provides this
127  * mapping.  When accessing the keys index, the primary key is returned rather
128  * than the entity.  When only the primary key is needed and not the entire
129  * entity, using the keys index is less expensive than using the secondary
130  * index because the primary index does not have to be accessed.</p>
131  *
132  * <p>The third mapping provided by {@code SecondaryIndex} is from primary key
133  * (PK) to entity (E), for the subset of entities having a given secondary key
134  * (SK).  This mapping is provided by the {@link #subIndex} method.  A
135  * sub-index is convenient when you are interested in working with the subset
136  * of entities having a particular secondary key value, for example, all
137  * employees in a given department.</p>
138  *
139  * <p>All three mappings, along with the mapping provided by the {@link
140  * PrimaryIndex}, are shown using example data in the {@link EntityIndex}
141  * interface documentation.  See {@link EntityIndex} for more information.</p>
142  *
143  * <p>Note that when using an index, keys and values are stored and retrieved
144  * by value not by reference.  In other words, if an entity object is stored
145  * and then retrieved, or retrieved twice, each object will be a separate
146  * instance.  For example, in the code below the assertion will always
147  * fail.</p>
148  * <pre class="code">
149  * MyKey key = ...;
150  * MyEntity entity1 = index.get(key);
151  * MyEntity entity2 = index.get(key);
152  * assert entity1 == entity2; // always fails!
153  * </pre>
154  *
155  * <h3>One-to-One Relationships</h3>
156  *
157  * <p>A {@link Relationship#ONE_TO_ONE ONE_TO_ONE} relationship, although less
158  * common than other types of relationships, is the simplest type of
159  * relationship.  A single entity is related to a single secondary key value.
160  * For example:</p>
161  *
162  * <pre class="code">
163  * {@literal @Entity}
164  * class Employee {
165  *
166  *     {@literal @PrimaryKey}
167  *     long id;
168  *
169  *     {@literal @SecondaryKey(relate=ONE_TO_ONE)}
170  *     String ssn;
171  *
172  *     String name;
173  *
174  *     private Employee() {}
175  * }
176  *
177  * {@code SecondaryIndex<String, Long, Employee>} employeeBySsn =
178  *     store.getSecondaryIndex(primaryIndex, String.class, "ssn");</pre>
179  *
180  * <p>With a {@link Relationship#ONE_TO_ONE ONE_TO_ONE} relationship, the
181  * secondary key must be unique; in other words, no two entities may have the
182  * same secondary key value.  If an attempt is made to store an entity having
183  * the same secondary key value as another existing entity, a {@link
184  * DatabaseException} will be thrown.</p>
185  *
186  * <p>Because the secondary key is unique, it is useful to lookup entities by
187  * secondary key using {@link EntityIndex#get}.  For example:</p>
188  *
189  * <pre class="code">
190  * Employee employee = employeeBySsn.get(mySsn);</pre>
191  *
192  * <h3>Many-to-One Relationships</h3>
193  *
194  * <p>A {@link Relationship#MANY_TO_ONE MANY_TO_ONE} relationship is the most
195  * common type of relationship.  One or more entities is related to a single
196  * secondary key value.  For example:</p>
197  *
198  * <pre class="code">
199  * {@literal @Entity}
200  * class Employee {
201  *
202  *     {@literal @PrimaryKey}
203  *     long id;
204  *
205  *     {@literal @SecondaryKey(relate=MANY_TO_ONE)}
206  *     String department;
207  *
208  *     String name;
209  *
210  *     private Employee() {}
211  * }
212  *
213  * {@code SecondaryIndex<String, Long, Employee>} employeeByDepartment =
214  *     store.getSecondaryIndex(primaryIndex, String.class, "department");</pre>
215  *
216  * <p>With a {@link Relationship#MANY_TO_ONE MANY_TO_ONE} relationship, the
217  * secondary key is not required to be unique; in other words, more than one
218  * entity may have the same secondary key value.  In this example, more than
219  * one employee may belong to the same department.</p>
220  *
221  * <p>The most convenient way to access the employees in a given department is
222  * by using a sub-index.  For example:</p>
223  *
224  * <pre class="code">
225  * {@code EntityIndex<Long, Entity>} subIndex = employeeByDepartment.subIndex(myDept);
226  * {@code EntityCursor<Employee>} cursor = subIndex.entities();
227  * try {
228  *     for (Employee entity : cursor) {
229  *         // Do something with the entity...
230  *     }
231  * } finally {
232  *     cursor.close();
233  * }</pre>
234  *
235  * <h3>One-to-Many Relationships</h3>
236  *
237  * <p>In a {@link Relationship#ONE_TO_MANY ONE_TO_MANY} relationship, a single
238  * entity is related to one or more secondary key values.  For example:</p>
239  *
240  * <pre class="code">
241  * {@literal @Entity}
242  * class Employee {
243  *
244  *     {@literal @PrimaryKey}
245  *     long id;
246  *
247  *     {@literal @SecondaryKey(relate=ONE_TO_MANY)}
248  *     {@literal Set<String> emailAddresses = new HashSet<String>;}
249  *
250  *     String name;
251  *
252  *     private Employee() {}
253  * }
254  *
255  * {@code SecondaryIndex<String, Long, Employee>} employeeByEmail =
256  *     store.getSecondaryIndex(primaryIndex, String.class, "emailAddresses");</pre>
257  *
258  * <p>With a {@link Relationship#ONE_TO_MANY ONE_TO_MANY} relationship, the
259  * secondary key must be unique; in other words, no two entities may have the
260  * same secondary key value.  In this example, no two employees may have the
261  * same email address.  If an attempt is made to store an entity having the
262  * same secondary key value as another existing entity, a {@link
263  * DatabaseException} will be thrown.</p>
264  *
265  * <p>Because the secondary key is unique, it is useful to lookup entities by
266  * secondary key using {@link EntityIndex#get}.  For example:</p>
267  *
268  * <pre class="code">
269  * Employee employee = employeeByEmail.get(myEmailAddress);</pre>
270  *
271  * <p>The secondary key field for a {@link Relationship#ONE_TO_MANY
272  * ONE_TO_MANY} relationship must be an array or collection type.  To access
273  * the email addresses of an employee, simply access the collection field
274  * directly.  For example:</p>
275  *
276  * <pre class="code">
277  * Employee employee = primaryIndex.get(1); // Get the entity by primary key
278  * employee.emailAddresses.add(myNewEmail); // Add an email address
279  * primaryIndex.putNoReturn(1, employee);   // Update the entity</pre>
280  *
281  * <h3>Many-to-Many Relationships</h3>
282  *
283  * <p>In a {@link Relationship#MANY_TO_MANY MANY_TO_MANY} relationship, one
284  * or more entities is related to one or more secondary key values.  For
285  * example:</p>
286  *
287  * <pre class="code">
288  * {@literal @Entity}
289  * class Employee {
290  *
291  *     {@literal @PrimaryKey}
292  *     long id;
293  *
294  *     {@literal @SecondaryKey(relate=MANY_TO_MANY)}
295  *     {@literal Set<String> organizations = new HashSet<String>;}
296  *
297  *     String name;
298  *
299  *     private Employee() {}
300  * }
301  *
302  * {@code SecondaryIndex<String, Long, Employee>} employeeByOrganization =
303  *     store.getSecondaryIndex(primaryIndex, String.class, "organizations");</pre>
304  *
305  * <p>With a {@link Relationship#MANY_TO_MANY MANY_TO_MANY} relationship, the
306  * secondary key is not required to be unique; in other words, more than one
307  * entity may have the same secondary key value.  In this example, more than
308  * one employee may belong to the same organization.</p>
309  *
310  * <p>The most convenient way to access the employees in a given organization
311  * is by using a sub-index.  For example:</p>
312  *
313  * <pre class="code">
314  * {@code EntityIndex<Long, Entity>} subIndex = employeeByOrganization.subIndex(myOrg);
315  * {@code EntityCursor<Employee>} cursor = subIndex.entities();
316  * try {
317  *     for (Employee entity : cursor) {
318  *         // Do something with the entity...
319  *     }
320  * } finally {
321  *     cursor.close();
322  * }</pre>
323  *
324  * <p>The secondary key field for a {@link Relationship#MANY_TO_MANY
325  * MANY_TO_MANY} relationship must be an array or collection type.  To access
326  * the organizations of an employee, simply access the collection field
327  * directly.  For example:</p>
328  *
329  * <pre class="code">
330  * Employee employee = primaryIndex.get(1); // Get the entity by primary key
331  * employee.organizations.remove(myOldOrg); // Remove an organization
332  * primaryIndex.putNoReturn(1, employee);   // Update the entity</pre>
333  *
334  * <h3>Foreign Key Constraints for Related Entities</h3>
335  *
336  * <p>In all the examples above the secondary key is treated only as a simple
337  * value, such as a {@code String} department field.  In many cases, that is
338  * sufficient.  But in other cases, you may wish to constrain the secondary
339  * keys of one entity class to be valid primary keys of another entity
340  * class.  For example, a Department entity may also be defined:</p>
341  *
342  * <pre class="code">
343  * {@literal @Entity}
344  * class Department {
345  *
346  *     {@literal @PrimaryKey}
347  *     String name;
348  *
349  *     String missionStatement;
350  *
351  *     private Department() {}
352  * }</pre>
353  *
354  * <p>You may wish to constrain the department field values of the Employee
355  * class in the examples above to be valid primary keys of the Department
356  * entity class.  In other words, you may wish to ensure that the department
357  * field of an Employee will always refer to a valid Department entity.</p>
358  *
359  * <p>You can implement this constraint yourself by validating the department
360  * field before you store an Employee.  For example:</p>
361  *
362  * <pre class="code">
363  * {@code PrimaryIndex<String, Department>} departmentIndex =
364  *     store.getPrimaryIndex(String.class, Department.class);
365  *
366  * void storeEmployee(Employee employee) throws DatabaseException {
367  *     if (departmentIndex.contains(employee.department)) {
368  *         primaryIndex.putNoReturn(employee);
369  *     } else {
370  *         throw new IllegalArgumentException("Department does not exist: " +
371  *                                            employee.department);
372  *     }
373  * }</pre>
374  *
375  * <p>Or, instead you could define the Employee department field as a foreign
376  * key, and this validation will be done for you when you attempt to store the
377  * Employee entity.  For example:</p>
378  *
379  * <pre class="code">
380  * {@literal @Entity}
381  * class Employee {
382  *
383  *     {@literal @PrimaryKey}
384  *     long id;
385  *
386  *     {@literal @SecondaryKey(relate=MANY_TO_ONE, relatedEntity=Department.class)}
387  *     String department;
388  *
389  *     String name;
390  *
391  *     private Employee() {}
392  * }</pre>
393  *
394  * <p>The {@code relatedEntity=Department.class} above defines the department
395  * field as a foreign key that refers to a Department entity.  Whenever a
396  * Employee entity is stored, its department field value will be checked to
397  * ensure that a Department entity exists with that value as its primary key.
398  * If no such Department entity exists, then a {@link DatabaseException} is
399  * thrown, causing the transaction to be aborted (assuming that transactions
400  * are used).</p>
401  *
402  * <p>This begs the question:  What happens when a Department entity is deleted
403  * while one or more Employee entities have department fields that refer to
404  * the deleted department's primary key?  If the department were allowed to be
405  * deleted, the foreign key constraint for the Employee department field would
406  * be violated, because the Employee department field would refer to a
407  * department that does not exist.</p>
408  *
409  * <p>By default, when this situation arises the system does not allow the
410  * department to be deleted.  Instead, a {@link DatabaseException} is thrown,
411  * causing the transaction to be aborted.  In this case, in order to delete a
412  * department, the department field of all Employee entities must first be
413  * updated to refer to a different existing department, or set to null.  This
414  * is the responsibility of the application.</p>
415  *
416  * <p>There are two additional ways of handling deletion of a Department
417  * entity.  These alternatives are configured using the {@link
418  * SecondaryKey#onRelatedEntityDelete} annotation property.  Setting this
419  * property to {@link DeleteAction#NULLIFY} causes the Employee department
420  * field to be automatically set to null when the department they refer to is
421  * deleted.  This may or may not be desirable, depending on application
422  * policies.  For example:</p>
423  *
424  * <pre class="code">
425  * {@literal @Entity}
426  * class Employee {
427  *
428  *     {@literal @PrimaryKey}
429  *     long id;
430  *
431  *     {@code @SecondaryKey(relate=MANY_TO_ONE, relatedEntity=Department.class,
432  *                                       onRelatedEntityDelete=NULLIFY)}
433  *     String department;
434  *
435  *     String name;
436  *
437  *     private Employee() {}
438  * }</pre>
439  *
440  * <p>The {@link DeleteAction#CASCADE} value, on the other hand, causes the
441  * Employee entities to be automatically deleted when the department they refer
442  * to is deleted.  This is probably not desirable in this particular example,
443  * but is useful for parent-child relationships.  For example:</p>
444  *
445  * <pre class="code">
446  * {@literal @Entity}
447  * class Order {
448  *
449  *     {@literal @PrimaryKey}
450  *     long id;
451  *
452  *     String description;
453  *
454  *     private Order() {}
455  * }
456  *
457  * {@literal @Entity}
458  * class OrderItem {
459  *
460  *     {@literal @PrimaryKey}
461  *     long id;
462  *
463  *     {@code @SecondaryKey(relate=MANY_TO_ONE, relatedEntity=Order.class,
464  *                                       onRelatedEntityDelete=CASCADE)}
465  *     long orderId;
466  *
467  *     String description;
468  *
469  *     private OrderItem() {}
470  * }</pre>
471  *
472  * <p>The OrderItem orderId field refers to its "parent" Order entity.  When an
473  * Order entity is deleted, it may be useful to automatically delete its
474  * "child" OrderItem entities.</p>
475  *
476  * <p>For more information, see {@link SecondaryKey#onRelatedEntityDelete}.</p>
477  *
478  * <h3>One-to-Many versus Many-to-One for Related Entities</h3>
479  *
480  * <p>When there is a conceptual Many-to-One relationship such as Employee to
481  * Department as illustrated in the examples above, the relationship may be
482  * implemented either as Many-to-One in the Employee class or as One-to-Many in
483  * the Department class.</p>
484  *
485  * <p>Here is the Many-to-One approach.</p>
486  *
487  * <pre class="code">
488  * {@literal @Entity}
489  * class Employee {
490  *
491  *     {@literal @PrimaryKey}
492  *     long id;
493  *
494  *     {@literal @SecondaryKey(relate=MANY_TO_ONE, relatedEntity=Department.class)}
495  *     String department;
496  *
497  *     String name;
498  *
499  *     private Employee() {}
500  * }
501  *
502  * {@literal @Entity}
503  * class Department {
504  *
505  *     {@literal @PrimaryKey}
506  *     String name;
507  *
508  *     String missionStatement;
509  *
510  *     private Department() {}
511  * }</pre>
512  *
513  * <p>And here is the One-to-Many approach.</p>
514  *
515  * <pre class="code">
516  * {@literal @Entity}
517  * class Employee {
518  *
519  *     {@literal @PrimaryKey}
520  *     long id;
521  *
522  *     String name;
523  *
524  *     private Employee() {}
525  * }
526  *
527  * {@literal @Entity}
528  * class Department {
529  *
530  *     {@literal @PrimaryKey}
531  *     String name;
532  *
533  *     String missionStatement;
534  *
535  *     {@literal @SecondaryKey(relate=ONE_TO_MANY, relatedEntity=Employee.class)}
536  *     {@literal Set<Long> employees = new HashSet<Long>;}
537  *
538  *     private Department() {}
539  * }</pre>
540  *
541  * <p>Which approach is best?  The Many-to-One approach better handles large
542  * number of entities on the to-Many side of the relationship because it
543  * doesn't store a collection of keys as an entity field.  With Many-to-One a
544  * Btree is used to store the collection of keys and the Btree can easily
545  * handle very large numbers of keys.  With One-to-Many, each time a related
546  * key is added or removed the entity on the One side of the relationship,
547  * along with the complete collection of related keys, must be updated.
548  * Therefore, if large numbers of keys may be stored per relationship,
549  * Many-to-One is recommended.</p>
550  *
551  * <p>If the number of entities per relationship is not a concern, then you may
552  * wish to choose the approach that is most natural in your application data
553  * model.  For example, if you think of a Department as containing employees
554  * and you wish to modify the Department object each time an employee is added
555  * or removed, then you may wish to store a collection of Employee keys in the
556  * Department object (One-to-Many).</p>
557  *
558  * <p>Note that if you have a One-to-Many relationship and there is no related
559  * entity, then you don't have a choice -- you have to use One-to-Many because
560  * there is no entity on the to-Many side of the relationship where a
561  * Many-to-One key could be defined.  An example is the Employee to email
562  * addresses relationship discussed above:</p>
563  *
564  * <pre class="code">
565  * {@literal @Entity}
566  * class Employee {
567  *
568  *     {@literal @PrimaryKey}
569  *     long id;
570  *
571  *     {@literal @SecondaryKey(relate=ONE_TO_MANY)}
572  *     {@literal Set<String> emailAddresses = new HashSet<String>;}
573  *
574  *     String name;
575  *
576  *     private Employee() {}
577  * }</pre>
578  *
579  * <p>For sake of argument imagine that each employee has thousands of email
580  * addresses and employees frequently add and remove email addresses.  To
581  * avoid the potential performance problems associated with updating the
582  * Employee entity every time an email address is added or removed, you could
583  * create an EmployeeEmailAddress entity and use a Many-to-One relationship as
584  * shown below:</p>
585  *
586  * <pre class="code">
587  * {@literal @Entity}
588  * class Employee {
589  *
590  *     {@literal @PrimaryKey}
591  *     long id;
592  *
593  *     String name;
594  *
595  *     private Employee() {}
596  * }
597  *
598  * {@literal @Entity}
599  * class EmployeeEmailAddress {
600  *
601  *     {@literal @PrimaryKey}
602  *     String emailAddress;
603  *
604  *     {@literal @SecondaryKey(relate=MANY_TO_ONE, relatedEntity=Employee.class)}
605  *     long employeeId;
606  *
607  *     private EmployeeEmailAddress() {}
608  * }</pre>
609  *
610  * <h3>Key Placement with Many-to-Many for Related Entities</h3>
611  *
612  * <p>As discussed in the section above, one drawback of a to-Many relationship
613  * (One-to-Many was discussed above and Many-to-Many is discussed here) is that
614  * it requires storing a collection of keys in an entity.  Each time a key is
615  * added or removed, the containing entity must be updated.  This has potential
616  * performance problems when there are large numbers of entities on the to-Many
617  * side of the relationship, in other words, when there are large numbers of
618  * keys in each secondary key field collection.</p>
619  *
620  * <p>If you have a Many-to-Many relationship with a reasonably small number of
621  * entities on one side of the relationship and a large number of entities on
622  * the other side, you can avoid the potential performance problems by defining
623  * the secondary key field on the side with a small number of entities.</p>
624  *
625  * <p>For example, in an Employee-to-Organization relationship, the number of
626  * organizations per employee will normally be reasonably small but the number
627  * of employees per organization may be very large.  Therefore, to avoid
628  * potential performance problems, the secondary key field should be defined in
629  * the Employee class as shown below.</p>
630  *
631  * <pre class="code">
632  * {@literal @Entity}
633  * class Employee {
634  *
635  *     {@literal @PrimaryKey}
636  *     long id;
637  *
638  *     {@literal @SecondaryKey(relate=MANY_TO_MANY, relatedEntity=Organization.class)}
639  *     {@literal Set<String> organizations = new HashSet<String>;}
640  *
641  *     String name;
642  *
643  *     private Employee() {}
644  * }
645  *
646  * {@literal @Entity}
647  * class Organization {
648  *
649  *     {@literal @PrimaryKey}
650  *     String name;
651  *
652  *     String description;
653  * }</pre>
654  *
655  * <p>If instead a {@code Set<Long> members} key had been defined in the
656  * Organization class, this set could potentially have a large number of
657  * elements and performance problems could result.</p>
658  *
659  * <h3>Many-to-Many Versus a Relationship Entity</h3>
660  *
661  * <p>If you have a Many-to-Many relationship with a large number of entities
662  * on <em>both</em> sides of the relationship, you can avoid the potential
663  * performance problems by using a <em>relationship entity</em>.  A
664  * relationship entity defines the relationship between two other entities
665  * using two Many-to-One relationships.</p>
666  *
667  * <p>Imagine a relationship between cars and trucks indicating whenever a
668  * particular truck was passed on the road by a particular car.  A given car
669  * may pass a large number of trucks and a given truck may be passed by a large
670  * number of cars.  First look at a Many-to-Many relationship between these two
671  * entities:</p>
672  *
673  * <pre class="code">
674  * {@literal @Entity}
675  * class Car {
676  *
677  *     {@literal @PrimaryKey}
678  *     String licenseNumber;
679  *
680  *     {@literal @SecondaryKey(relate=MANY_TO_MANY, relatedEntity=Truck.class)}
681  *     {@literal Set<String> trucksPassed = new HashSet<String>;}
682  *
683  *     String color;
684  *
685  *     private Car() {}
686  * }
687  *
688  * {@literal @Entity}
689  * class Truck {
690  *
691  *     {@literal @PrimaryKey}
692  *     String licenseNumber;
693  *
694  *     int tons;
695  *
696  *     private Truck() {}
697  * }</pre>
698  *
699  * <p>With the Many-to-Many approach above, the {@code trucksPassed} set could
700  * potentially have a large number of elements and performance problems could
701  * result.</p>
702  *
703  * <p>To apply the relationship entity approach we define a new entity class
704  * named CarPassedTruck representing a single truck passed by a single car.  We
705  * remove the secondary key from the Car class and use two secondary keys in
706  * the CarPassedTruck class instead.</p>
707  *
708  * <pre class="code">
709  * {@literal @Entity}
710  * class Car {
711  *
712  *     {@literal @PrimaryKey}
713  *     String licenseNumber;
714  *
715  *     String color;
716  *
717  *     private Car() {}
718  * }
719  *
720  * {@literal @Entity}
721  * class Truck {
722  *
723  *     {@literal @PrimaryKey}
724  *     String licenseNumber;
725  *
726  *     int tons;
727  *
728  *     private Truck() {}
729  * }
730  *
731  * {@literal @Entity}
732  * class CarPassedTruck {
733  *
734  *     {@literal @PrimaryKey}
735  *     long id;
736  *
737  *     {@literal @SecondaryKey(relate=MANY_TO_ONE, relatedEntity=Car.class)}
738  *     String carLicense;
739  *
740  *     {@literal @SecondaryKey(relate=MANY_TO_ONE, relatedEntity=Truck.class)}
741  *     String truckLicense;
742  *
743  *     private CarPassedTruck() {}
744  * }</pre>
745  *
746  * <p>The CarPassedTruck entity can be used to access the relationship by car
747  * license or by truck license.</p>
748  *
749  * <p>You may use the relationship entity approach because of the potential
750  * performance problems mentioned above.  Or, you may choose to use this
751  * approach in order to store other information about the relationship.  For
752  * example, if for each car that passes a truck you wish to record how much
753  * faster the car was going than the truck, then a relationship entity is the
754  * logical place to store that property.  In the example below the
755  * speedDifference property is added to the CarPassedTruck class.</p>
756  *
757  * <pre class="code">
758  * {@literal @Entity}
759  * class CarPassedTruck {
760  *
761  *     {@literal @PrimaryKey}
762  *     long id;
763  *
764  *     {@literal @SecondaryKey(relate=MANY_TO_ONE, relatedEntity=Car.class)}
765  *     String carLicense;
766  *
767  *     {@literal @SecondaryKey(relate=MANY_TO_ONE, relatedEntity=Truck.class)}
768  *     String truckLicense;
769  *
770  *     int speedDifference;
771  *
772  *     private CarPassedTruck() {}
773  * }</pre>
774  *
775  * <p>Be aware that the relationship entity approach adds overhead compared to
776  * Many-to-Many.  There is one additional entity and one additional secondary
777  * key.  These factors should be weighed against its advantages and the
778  * relevant application access patterns should be considered.</p>
779  *
780  * @author Mark Hayes
781  */
