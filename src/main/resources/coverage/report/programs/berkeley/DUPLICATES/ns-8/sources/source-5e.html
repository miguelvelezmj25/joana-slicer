18 /**
19  * @hidden
20  * For internal use only.
21  *
22  * Provides a way to create an association between primary and secondary
23  * databases that is not limited to a one-to-many association.
24  * <p>
25  * By implementing this interface, a secondary database may be associated with
26  * one or more primary databases.  For example, imagine an application that
27  * wishes to partition data from a single logical "table" into multiple primary
28  * databases to take advantage of the performance benefits of {@link
29  * Environment#removeDatabase} for a single partition, while at the same having
30  * a single index database for the entire "table".  For read operations via a
31  * secondary database, JE reads the primary key from the secondary database and
32  * then must read the primary record from the primary database.  The mapping
33  * from a primary key to its primary database is performed by the user's
34  * implementation of the {@link #getPrimary} method.
35  * <p>
36  * In addition, a primary database may be divided into subsets of keys where
37  * each subset may be efficiently indexed by zero or more secondary databases.
38  * This effectively allows multiple logical "tables" per primary database.
39  * When a primary record is written, the secondaries for its logical "table"
40  * are returned by the user's implementation of {@link #getSecondaries}.
41  * During a primary record write, because {@link #getSecondaries} is called to
42  * determine the secondary databases, only the key creators/extractors for
43  * those secondaries are invoked.  For example, if you have a single primary
44  * database with 10 logical "tables", each of which has 5 distinct secondary
45  * databases, when a record is written only 5 key creators/extractors will be
46  * invoked, not 50.
47  * <p>
48  * <b>Configuring a SecondaryAssociation</b>
49  * <p>
50  * When primary and secondary databases are associated using a {@code
51  * SecondaryAssociation}, the databases must all be configured with the same
52  * {@code SecondaryAssociation} instance.  A common error is to forget to
53  * configure the {@code SecondaryAssociation} on the primary database.
54  * <p>
55  * A {@code SecondaryAssociation} is configured using {@link
56  * DatabaseConfig#setSecondaryAssociation} and {@link
57  * SecondaryConfig#setSecondaryAssociation} for a primary and secondary
58  * database, respectively. When calling {@link
59  * Environment#openSecondaryDatabase}, null must be passed for the
60  * primaryDatabase parameter when a {@code SecondaryAssociation} is configured.
61  * <p>
62  * Note that when a {@code SecondaryAssociation} is configured, {@code true}
63  * may not be passed to the {@link SecondaryConfig#setAllowPopulate} method.
64  * Population of new secondary databases in an existing {@code
65  * SecondaryAssociation} is done differently, as described below.
66  * <p>
67  * <b>Adding and removing secondary associations</b>
68  * <p>
69  * The state information defining the association between primary and secondary
70  * databases is maintained by the application, and made available to JE by
71  * implementing the {@code SecondaryAssociation} interface.  The application
72  * may add/remove databases to/from the association without any explicit
73  * coordination with JE.  However, certain rules need to be followed to
74  * maintain data integrity.
75  * <p>
76  * In the simplest case, there is a fixed (never changing) set of primary and
77  * secondary databases and they are added to the association at the time they
78  * are created, before writing or reading records.  In this case, since reads
79  * and writes do not occur concurrently with changes to the association, no
80  * special rules are needed.  For example:
81  * <ol>
82  *     <li>Open the databases.</li>
83  *     <li>Add databases to the association.</li>
84  *     <li>Begin writing and reading.</li>
85  * </ol>
86  * <p>
87  * In other cases, primary and secondary databases may be added to an already
88  * established association, along with concurrent reads and writes.  The rules
89  * for doing so are described below.
90  * <p>
91  * Adding an empty (newly created) primary database is no more complex than in
92  * the simple case above, assuming that writes to the primary database do not
93  * proceed until after it has been added to the association.
94  * <ol>
95  *     <li>Open the new primary database.</li>
96  *     <li>Add the primary database to the association such that {@link
97  *     #getSecondaries} returns the appropriate list when passed a key in the
98  *     new primary database, and {@link #getPrimary} returns the new database
99  *     when passed a key it contains.</li>
100  *     <li>Begin writing to the new primary database.</li>
101  * </ol>
102  * <p>
103  * Using the procedure above for adding a primary database, records will be
104  * indexed in secondary databases as they are added to the primary database,
105  * and will be immediately available to secondary index queries.
106  * <p>
107  * Alternatively, records can be added to the primary database without making
108  * them immediately available to secondary index queries by using the following
109  * procedure.  Records will be indexed in secondary databases as they are added
110  * but will not be returned by secondary queries until after the last step.
111  * <ol>
112  *     <li>Open the new primary database.</li>
113  *     <li>Modify the association such that {@link #getSecondaries} returns the
114  *     appropriate list when passed a key in the new primary database, but
115  *     {@link #getPrimary} returns null when passed a key it contains.</li>
116  *     <li>Write records in the new primary database.</li>
117  *     <li>Modify the association such that {@link #getPrimary} returns the new
118  *     database when passed a key it contains.</li>
119  * </ol>
120  * <p>
121  * The following procedure should be used for removing an existing (non-empty)
122  * primary database from an association.
123  * <ol>
124  *     <li>Remove the primary database from the association, such that {@link
125  *     #getPrimary} returns null for all keys in the primary.</li>
126  *     <li>Disable read and write operations on the database and ensure that
127  *     all in-progress operations have completed.</li>
128  *     <li>At this point the primary database may be closed and removed
129  *     (e.g., with {@link Environment#removeDatabase}), if desired.</li>
130  *     <li>For each secondary database associated with the removed primary
131  *     database, {@link SecondaryDatabase#deleteObsoletePrimaryKeys} should be
132  *     called to process all secondary records.</li>
133  *     <li>At this point {@link #getPrimary} may throw an exception (rather
134  *     than return null) when called for a primary key in the removed database,
135  *     if desired.</li>
136  * </ol>
137  * <p>
138  * The following procedure should be used for adding a new (empty) secondary
139  * database to an association, and to populate the secondary incrementally
140  * from its associated primary database(s).
141  * <ol>
142  *     <li>Open the secondary database and call {@link
143  *     SecondaryDatabase#startIncrementalPopulation}.  The secondary may not be
144  *     used (yet) for read operations.</li>
145  *     <li>Add the secondary database to the association, such that the
146  *     collection returned by {@link #getSecondaries} includes the new
147  *     database, for all primary keys that should be indexed.</li>
148  *     <li>For each primary database associated with the new secondary
149  *     database, {@link Database#populateSecondaries} should be called to
150  *     process all primary records.</li>
151  *     <li>Call {@link SecondaryDatabase#endIncrementalPopulation}. The
152  *     secondary database may now be used for read operations.</li>
153  * </ol>
154  * <p>
155  * The following procedure should be used for removing an existing (non-empty)
156  * secondary database from an association.
157  * <ol>
158  *     <li>Remove the secondary database from the association, such that it is
159  *     not included in the collection returned by {@link #getSecondaries}.</li>
160  *     <li>Disable read and write operations on the database and ensure that
161  *     all in-progress operations have completed.</li>
162  *     <li>The secondary database may now be closed and removed, if
163  *     desired.</li>
164  * </ol>
165  * <p>
166  * <b>Other Implementation Requirements</b>
167  * <p>
168  * The implementation must use data structures for storing and accessing
169  * association information that provide <em>happens-before</em> semantics.  In
170  * other words, when the association is changed, other threads calling the
171  * methods in this interface must see the changes as soon as the change is
172  * complete.
173  * <p>
174  * The implementation should use non-blocking data structures to hold
175  * association information to avoid blocking on the methods in this interface,
176  * which may be frequently called from many threads.
177  * <p>
178  * The simplest way to meet the above two requirements is to use concurrent
179  * structures such as those provided in the java.util.concurrent package, e.g.,
180  * ConcurrentHashMap.
181  * <p>
182  * For an example implementation, see:
183  * test/com/sleepycat/je/test/SecondaryAssociationTest.java
184  */

187     /*
188      * Implementation note on concurrent access and latching.
189      *
190      * When adding/removing primary/secondary DBs to an existing association,
191      * concurrency issues arise.  This section explains how they are handled,
192      * assuming the rules described in the javadoc below are followed when a
193      * custom association is configured.
194      *
195      * There are two cases:
196      *
197      * 1) A custom association is NOT configured.  An internal association is
198      * created that manages the one-many association between a primary and its
199      * secondaries.
200      *
201      * 2) A custom association IS configured.  It is used internally.
202      *
203      * For case 1, no custom association, DB addition and removal take place in
204      * the Environment.openSecondaryDatabase and SecondaryDatabase.close
205      * methods, respectively.  These methods, and the changes to the internal
206      * association, are protected by acquiring the secondary latch
207      * (EnvironmentImpl.getSecondaryAssociationLock) exclusively.  All write
208      * operations that use the association -- puts and deletes -- acquire this
209      * latch shared.  Therefore, write operations and changes to the
210      * association do not take place concurrently.  Read operations via a
211      * secondary DB/cursor do not acquire this latch, so they can take place
212      * concurrently with changes to the association; however, read operations
213      * only call getPrimary and they expect it may return null at any time (the
214      * operation ignores the record in that case).  It is impossible to close
215      * a primary while it is being read via a secondary, because secondaries
216      * must be closed before their associated primary.  (And the application is
217      * responsible for finishing reads before closing any DB.)
218      *
219      * For case 2, a custom association, the secondary latch does not provide
220      * protection because modifications to the association take place in the
221      * application domain, not in the Environment.openSecondaryDatabase and
222      * SecondaryDatabase.close methods.  Instead, protection is provided by the
223      * rules described in the javadoc here. Namely:
224      *
225      *   Primary/secondary DBs are added to the association AFTER opening them
226      *   (of course).  The DBs will not be be accessed until they are added to
227      *   the association.  
228      *
229      *   Primary/secondary DBs are removed from the association BEFORE closing
230      *   them.  The application is responsible for ensuring they are no longer
231      *   accessed before they are closed.
232      *
233      *   When a primary DB is added, its secondaries will be kept in sync as
234      *   soon as writing begins, since it must be added to the association
235      *   before writes are allowed.
236      *
237      *   When a secondary DB is added, the incremental population procedure
238      *   ensures that it is fully populated before being accessed.
239      */
240 
241     /**
242      * Returns true if there are no secondary databases in the association.
243      *
244      * This method is used by JE to optimize for the case where a primary
245      * has no secondary databases.  This allows a {@code SecondaryAssociation}
246      * to be configured on a primary database even when it has no associated
247      * secondaries, with no added overhead, and to allow for the possibility
248      * that secondaries will be added later.
249      * <p>
250      * For example, when a primary database has no secondaries, no internal
251      * latching is performed by JE during writes.  JE determines that no
252      * secondaries are present by calling {@code isEmpty}, prior to doing the
253      * write operation.
254      *
255      * @throws RuntimeException if an unexpected problem occurs.  The exception
256      * will be thrown to the application, which should then take appropriate
257      * action.
258      */
