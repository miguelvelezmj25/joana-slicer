16 /**
17  * Record lock modes for read operations. Lock mode parameters may be specified
18  * for all operations that retrieve data.
19  *
20  * <p><strong>Locking Rules</strong></p>
21  *
22  * <p>Together with {@link CursorConfig}, {@link TransactionConfig} and {@link
23  * EnvironmentConfig} settings, lock mode parameters determine how records are
24  * locked during read operations.  Record locking is used to enforce the
25  * isolation modes that are configured.  Record locking is summarized below for
26  * read and write operations.  For more information on isolation levels and
27  * transactions, see <a
28  * href="{@docRoot}/../TransactionGettingStarted/index.html"
29  * target="_top">Writing Transactional Applications</a>.</p>
30  *
31  * <p>With one exception, a record lock is always acquired when a record is
32  * read or written, and a cursor will always hold the lock as long as it is
33  * positioned on the record.  The exception is when {@link #READ_UNCOMMITTED}
34  * is specified, which allows a record to be read without any locking.</p>
35  *
36  * <p>Both read (shared) and write (exclusive) locks are used.  Read locks are
37  * normally acquired on read ({@code get} method) operations and write locks on
38  * write ({@code put} method) operations.  The only exception is that a write
39  * lock will be acquired on a read operation if {@link #RMW} is specified.</p>
40  *
41  * <p>Because read locks are shared, multiple accessors may read the same
42  * record.  Because write locks are exclusive, if a record is written by one
43  * accessor it may not be read or written by another accessor.  An accessor is
44  * either a transaction or a thread (for non-transactional operations).</p>
45  *
46  * <p>Whether additional locking is performed and how locks are released depend
47  * on whether the operation is transactional and other configuration
48  * settings.</p>
49  *
50  * <p><strong>Transactional Locking</strong></p>
51  *
52  * <p>Transactional operations include all write operations for a transactional
53  * database, and read operations when a non-null {@link Transaction} parameter
54  * is passed.  When a null transaction parameter is passed for a write
55  * operation for a transactional database, an auto-commit transaction is
56  * automatically used.</p>
57  *
58  * <p>With transactions, read and write locks are normally held until the end
59  * of the transaction (commit or abort).  Write locks are always held until the
60  * end of the transaction.  However, if {@link #READ_COMMITTED} is configured,
61  * then read locks for cursor operations are only held during the operation and
62  * while the cursor is positioned on the record.  The read lock is released
63  * when the cursor is moved to a different record or closed.  When {@link
64  * #READ_COMMITTED} is used for a database (non-cursor) operation, the read
65  * lock is released before the method returns.</p>
66  *
67  * <p>When neither {@link #READ_UNCOMMITTED} nor {@link #READ_COMMITTED} is
68  * specified, read and write locking as described above provide Repeatable Read
69  * isolation, which is the default transactional isolation level.  If
70  * Serializable isolation is configured, additional "next key" locking is
71  * performed to prevent "phantoms" -- records that are not visible at one point
72  * in a transaction but that become visible at a later point after being
73  * inserted by another transaction.  Serializable isolation is configured via
74  * {@link TransactionConfig#setSerializableIsolation} or {@link
75  * EnvironmentConfig#setTxnSerializableIsolation}.</p>
76  *
77  * <p><strong>Non-Transactional Locking</strong></p>
78  *
79  * <p>Non-transactional operations include all operations for a
80  * non-transactional database (including a Deferred Write database), and read
81  * operations for a transactional database when a null {@link Transaction}
82  * parameter is passed.</p>
83  *
84  * <p>For non-transactional operations, both read and write locks are only held
85  * while a cursor is positioned on the record, and are released when the cursor
86  * is moved to a different record or closed.  For database (non-cursor)
87  * operations, the read or write lock is released before the method
88  * returns.</p>
89  * 
90  * <p>This behavior is similar to {@link #READ_COMMITTED}, except that both
91  * read and write locks are released.  Configuring {@link #READ_COMMITTED} for
92  * a non-transactional database cursor has no effect.</p>
93  *
94  * <p>Because the current thread is the accessor (locker) for non-transactional
95  * operations, a single thread may have multiple cursors open without locking
96  * conflicts.  Two non-transactional cursors in the same thread may access the
97  * same record via write or read operations without conflicts, and the changes
98  * make by one cursor will be visible to the other cursor.</p>
99  *
100  * <p>However, a non-transactional operation will conflict with a transactional
101  * operation for the same record even when performed in the same thread.  When
102  * using a transaction in a particular thread for a particular database, to
103  * avoid conflicts you should use that transaction for all access to that
104  * database in that thread.  In other words, to avoid conflicts always pass the
105  * transaction parameter, not null, for all operations.  If you don't wish to
106  * hold the read lock for the duration of the transaction, specify {@link
107  * #READ_COMMITTED}.</p>
108  *
109  * <p><strong>Read Uncommitted (Dirty-Read)</strong></string></p>
110  *
111  * <p>When {@link #READ_UNCOMMITTED} is configured, no locking is performed
112  * by a read operation.  {@code READ_UNCOMMITTED} does not apply to write
113  * operations.</p>
114  *
115  * <p>{@code READ_UNCOMMITTED} is sometimes called dirty-read because records
116  * are visible to the caller in their current state in the Btree at the time of
117  * the read, even when that state is due to operations performed using a
118  * transaction that has not yet committed.  In addition, because no lock is
119  * acquired by the dirty read operation, the record's state may change at any
120  * time, even while a cursor used to do the dirty-read is still positioned on
121  * the record.</p>
122  *
123  * <p>To illustrate this, let's say a record is read with dirty-read
124  * ({@code READ_UNCOMMITTED}) by calling {@link Cursor#getNext Cursor.getNext}
125  * with a cursor C, and changes to the record are also being made in another
126  * thread using transaction T. When a locking (non-dirty-read) call to {@link
127  * Cursor#getCurrent Cursor.getCurrent} is subsequently made to read the same
128  * record again with C at the current position, a result may be returned that
129  * is different than the result returned by the earlier call to {@code
130  * getNext}. For example:
131  * <ul>
132  *     <li>If the record is updated by T after the dirty-read {@code getNext}
133  *     call, and T is committed, a subsequent call to {@code getCurrent} will
134  *     return the data updated by T.</li>
135  *
136  *     <li>If the record is updated by T before the dirty-read {@code getNext}
137  *     call, the {@code getNext} will returned the data updated by T.  But if
138  *     call, the {@code getNext} will return the data updated by T.  But if
139  *     T is then aborted, a subsequent call to {@code getCurrent} will return
140  *     the version of the data before it was updated by T.</li>
141  *
142  *     <li>If the record was inserted by T before the dirty-read {@code
143  *     getNext} call, the {@code getNext} call will return the inserted record.
144  *     But if T is aborted, a subsequent call to {@code getCurrent} will return
145  *     {@link OperationStatus#KEYEMPTY}.</li>
146  *
147  *     <li>If the record is deleted by T after the dirty-read {@code getNext}
148  *     call, and T is committed, a subsequent call to {@code getCurrent} will
149  *     return {@link OperationStatus#KEYEMPTY}.</li>
150  * </ul>
151  * </p>
152  *
153  * <p>Note that deleted records are handled specially in JE. Deleted records
154  * remain in the Btree until after the deleting transaction is committed, and
155  * they are removed from the Btree asynchronously (not immediately at commit
156  * time). When using {@code #READ_UNCOMMITTED}, any record encountered in the
157  * Btree that was previously deleted, whether or not the deleting transaction
158  * has been committed, will be ignored (skipped over) by the read operation.
159  * Of course, if the deleting transaction is aborted, the record will no longer
160  * be deleted.  If the application is scanning records, for example, this means
161  * that such records may be skipped by the scan. If this behavior is not
162  * desirable, {@link #READ_UNCOMMITTED_ALL} may be used instead. This mode
163  * ensures that records deleted by a transaction that is later aborted will not
164  * be skipped by a read operation.  This is accomplished in two different ways
165  * depending on the type of database and whether the record's data is requested
166  * by the operation.
167  * <ol>
168  *     <li>If the DB is configured for duplicates or the record's data
169  *     is not requested, then a record that has been deleted by an open
170  *     transaction is returned by the read operation.</li>
171  *
172  *     <li>If the DB is not configured for duplicates and the record's data is
173  *     requested, then the read operation must wait for the deleting
174  *     transaction to close (commit or abort).  After the transaction is
175  *     closed, the record will be returned if it is actually not deleted and
176  *     otherwise will be skipped.</li>
177  * </ol>
178  *
179  * <p>By "record data" we mean both the {@code data} parameter for a regular or
180  * primary DB, and the {@code pKey} parameter for a secondary DB. By "record
181  * data requested" we mean that all or part of the {@code DatabaseEntry} will
182  * be returned by the read operation.  Unless explicitly <em>not</em>
183  * requested, the complete {@code DatabaseEntry} is returned.  See
184  * <a href="Cursor.html#partialEntry">Using Partial DatabaseEntry
185  * Parameters</a> for more information.</p>
186  *
187  * <p>Because of this difference in behavior, although {@code
188  * #READ_UNCOMMITTED} is fully non-blocking, {@code #READ_UNCOMMITTED_ALL} is
189  * not (under the conditions described). As a result, when using {@code
190  * #READ_UNCOMMITTED_ALL} under these conditions, a {@link
191  * LockConflictException} will be thrown when blocking results in a deadlock or
192  * lock timeout.</p>
193  *
194  * <p>To summarize, callers that use {@code READ_UNCOMMITTED} or {@code
195  * #READ_UNCOMMITTED_ALL} should be prepared for the following behaviors.
196  * <ul>
197  *     <li>After a successful dirty-read operation, because no lock is acquired
198  *     the record can be changed by another transaction, even when the cursor
199  *     used to perform the dirty-read operation is still positioned on the
200  *     record.</li>
201  *
202  *     <li>After a successful dirty-read operation using a cursor C, say that
203  *     another transaction T deletes the record, and T is committed. In this
204  *     case, {@link OperationStatus#KEYEMPTY} will be returned by the following
205  *     methods if they are called while C is still positioned on the deleted
206  *     record: {@link Cursor#getCurrent Cursor.getCurrent}, {@link
207  *     Cursor#putCurrent Cursor.putCurrent} and {@link Cursor#delete
208  *     Cursor.delete}.</li>
209  *
210  *     <li>When using {@code READ_UNCOMMITTED}, deleted records will be skipped
211  *     even when the deleting transaction is still open. No blocking will occur
212  *     and {@link LockConflictException} is never thrown when using this
213  *     mode.</li>
214  *
215  *     <li>When using {@code #READ_UNCOMMITTED_ALL}, deleted records will not
216  *     be skipped even when the deleting transaction is open. If the DB is a
217  *     duplicates DB or the record's data is not requested, the deleted record
218  *     will be returned.  If the DB is not a duplicates DB and the record's
219  *     data is requested, blocking will occur until the deleting transaction is
220  *     closed. In the latter case, {@link LockConflictException} will be thrown
221  *     when this blocking results in a deadlock or a lock timeout.</li>
222  * </ul>
223  * </p>
224  */

292      *
