66 /**
67  * A VLSN (Virtual LSN) is used to identify every log entry shared between
68  * members of the replication group. Since a JE log is identified by LSNs, we
69  * must have a way to map VLSN->LSNs in order to fetch a replicated log record
70  * from the local log, using the VLSN.  The VLSNIndex implements those
71  * mappings. The VLSNIndex has these responsibilities:
72  *
73  * Generating new VLSNs.
74  *   Only masters need to generate VLSNs, but any node may have the potential
75  *   to be a master. The VLSN sequence must ascend over time and across
76  *   recoveries, so the VSLN id must be preserved much like the database, node
77  *   and txn ids.
78  * Maintaining the VLSN range.
79  *   Although each node needs to receive and store each log entry from the
80  *   replication stream, over time the part of the stream that is stored can be
81  *   reduced, either by log cleaning, or by syncups which can truncate the
82  *   replication stream. A node always holds a contiguous portion of the
83  *   replication stream. The VLSN range identifies that portion by having the
84  *   start and end VLSNs, as well as key landmarks such as the lastSync-able
85  *   log entry and the last commit log entry. VLSN range information is used by
86  *   elections and syncup.
87  * Gatekeeper for waiting for the most recently logged entries.
88  *   Feeders block upon the VLSNIndex when they are trying to fetch the most
89  *   recently logged entries. These recent log entries are held in a two level
90  *   cache within the VLSNIndex. A call to VLSNIndex.waitForLsn() goes through
91  *   this sequence:
92  *   1) check the log item stored in the vlsn wait latch, if the call did wait.
93  *   2) check the log item cache
94  *   If both fail, the FeederReader will fetch the required log entry from log
95  *   buffers or disk
96  * Providing the LSN mapping for a log record identified by its VLSN.
97  *   The Feeders and the syncup protocol both need to retrieve log records
98  *   by VLSN. To do that, we need an LSN mapping.
99  *
100  * Mappings are added to VLSNIndex when replicated log entries are written into
101  * the local log.  Although all mappings are registered, the VLSNIndex does not
102  * keep every one, in order to save on disk and in-memory storage. Only a
103  * sparse set is kept. When searching for a log entry by VLSN, the caller uses
104  * the closest available mapping and then scans the log looking for that entry.
105  *
106  * The VLSNIndex relies on the assumption that VLSN tagged log entries are
107  * ordered and contiguous in the log. That is, the LSN for VLSN 1 is < the LSN
108  * for VLSN 2 < LSN for VLSN 3, and there is never a gap in the VLSNs. However,
109  * at node syncup, the replication stream may need to be truncated when rolling
110  * back a non-committed log entry. We can't literally truncate the log files
111  * because the JE logs contain intermingled transactional and non transactional
112  * information. Instead, the truncation is done both logically by amending the
113  * VLSNIndex, and physically by overmarking those entries in the JE
114  * logs. Because of that, a physical dump of the log may show some VLSN tagged
115  * entries as duplicate and/or out of order because they're abandoned log
116  * entries that are not logically part of the replication stream any more
117  * For example, the log can look like this:
118  *  LSN 100, VLSN 1
119  *  LSN 200, VLSN 2  <- overmarked
120  *  LSN 300, VLSN 3  <- overmarked
121  *   --- syncup, rollback to VLSN 1, restart at VLSN 2
122  *  LSN 400, VLSN 2
123  *  LSN 500, VLSN 3
124  *
125  * VLSN->LSN mappings are created under the log write latch, which ensures that
126  * all VLSN tagged log entries are ordered in the logical replication stream in
127  * the log. However, the mapping is added to the VLSNIndex outside the log
128  * write latch, so the VLSNIndex database may have a momentary gap. For
129  * example,
130  *
131  *  t0- thread 1 logs entry at VLSN=1, LSN=100, within log write latch
132  *  t1- thread 2 logs entry at VLSN=2, LSN=150, within log write latch
133  *  t2- thread 3 logs entry at VLSN=3, LSN=200, within log write latch
134  *  t3- thread 1 calls VLSNIndex.put(VLSN=1/LSN=100)
135  *  t4- thread 3 calls VLSNIndex.put(VLSN=3/LSN=200)
136  *  t5- thread 2 calls VLSNIndex.put(VLSN=2/LSN=150)
137  *
138  * At t4, the VLSNIndex contains 1/100, 3/200, but not 2/150. However, we know
139  * that the VLSNIndex always represents a contiguous range of VLSNs, so the
140  * fact that 2/150 is not yet is handled, and is just like the case where
141  * the VLSNIndex optimized away the mapping in order to keep the index sparse.
142  *
143  * We do guarantee that the start and end VLSNs in the range have mappings, in
144  * order to always be able to provide a LTE and GTE mapping for all valid
145  * VLSNs. Because of that, if a VLSN comes out of order, it does not update the
146  * range. Care must be taken when truncating the VLSNIndex from the head or the
147  * tail to ensure that the guaranteed existence of the start and end range
148  * mapping remains valid.
149  *
150  * Cache and persistent storage:
151  *
152  *  The VLSN->LSN mappings in the range are grouped into instances of
153  *  com.sleepycat.je.util.VLSNBucket. Each bucket knows the first and last VLSN
154  *  within its mini-range. We observe these invariants
155  *    - buckets are ordered by VLSN in the database and the bucket cache,
156  *    - only the last bucket is the target of updates at any time,
157  *    - a single bucket corresponds to a single file, but a single file may
158  *  have multiple buckets covering it.
159  *
160  *  While it would be nice to also guarantee that there are no gaps between
161  *  buckets, ie:
162  *    bucket(N-1).last == bucket(N).first - 1
163  *    bucket(N).last  ==  bucket(N-1).first - 1
164  *  it is not possible to do so because we the put() call is not serialized
165  *  because we don't want to add overhead to the log write latch. In order
166  *  to permit out of order puts(), and to require that only the last bucket
167  *  is updated, we must permit gaps between buckets.
168  *
169  *  Mappings start out being cached in VLSNBuckets held in memory by the
170  *  VLSNTracker. As the tracker fills, the buckets are flushed to persistent
171  *  storage in a internal, non-replicated database. Both the database and
172  *  the tracker cache hold key/value pairs where
173  *
174  *    key = bucket.first
175  *    data = bucket
176  *
177  * Since the first valid VLSN is 1, key = -1 is reserved for storage of the
178  * VLSNRange.
179  *
180  * Buckets are filled up as new VLSNs arrive (either because they've been
181  * generated by write operations on the master, or because they're incoming
182  * operations on the replica). They're flushed to disk periodically rather than
183  * with every new VLSN, because the update rate would have too much of a
184  * performance impact. Since there is this level of caching happening, we must
185  * be careful to write in-memory buckets to disk at well known points to
186  * support recoverability. The flushing must be instigated by a third party
187  * activity, such as checkpointing, rather than by the action of adding a new
188  * mapping. That's because mappings are registered by the logging system, and
189  * although we are not holding the log write latch at that point, it seems
190  * inadvisable to recursively generate another logging call on behalf of the
191  * flush.  Currently the VLSNIndex is flushed to disk at every checkpoint.  It
192  * can also optionally happen more often, and (TODO) we may want to do so
193  * because we've seen cases where checkpoints take a very long time. Perhaps we
194  * should flush when we flip to a new log file?
195  *
196  * Once written to disk, the buckets are generally not updated. Updates can
197  * happen when the range is truncated, such as for syncup rollback, but the
198  * system is quiescent at that time, and there are no new mappings created. Log
199  * cleaning can read the vlsnIndex and delete buckets, but will not modify
200  * mappings. The VLSNRange does naturally change often, and that data record
201  * does get updated.
202  *
203  * Recovery:
204  *
205  * The VLSN database is restored at recovery time just as all other databases
206  * are. However, there may be a portion of the VLSN range that was not flushed
207  * to disk. At recovery, we piggyback onto the log scanning done and re-track
208  * the any mappings found within the recovery range. Those mappings are merged
209  * into those stored on disk, so that the VLSNIndex correctly reflects the
210  * entire replication stream at startup. For example, suppose a log has:
211  *
212  * LSN
213  * 100      firstActiveLSN
214  * 200      Checkpoint start
215  * 300      VLSN 78
216  * 400      VLSNIndex flushed here
217  * 500      Checkpoint end
218  * 600      VLSN 79
219  *
220  * The VLSNIndex is initially populated with the version of the index found
221  * at LSN 400. That doesn't include VLSN 79. A tracking pass is done from
222  * checkpoint start -> end of log, which sweeps up VLSN 78 and VLSN 79 into
223  * a temporary tracker. That tracker is merged in the VLSNIndex, to update
224  * its mappings to VLSN 79.
225  *
226  * Note that the checkpoint VLSNIndex must encompass all vlsn mappings that are
227  * prior to the checkpoint start of that recovery period. This follows the
228  * general philosophy that checkpoint flushes all metadata, and recovery reads
229  * from checkpoint start onewards to add on any neede extra data.
230  * Retrieving mappings:
231  *
232  * Callers who need to retrieve mappings obtain a VLSNScanner, which acts as a
233  * cursor over the VLSNIndex. A VLSNScanner finds and saves the applicable
234  * VLSNBucket, and queries the bucket directly as long as it can provide
235  * mappings. This reduces the level of contention between multiple readers
236  * (feeders) and writers (application threads, or the replay thread)
237  *
238  * Synchronization hierarchy:
239  *
240  * To write a new mapping, you must have the mutex on the VLSIndex, and then
241  * the tracker, which lets you obtain the correct bucket, and then you must
242  * have a mutex on the bucket. To read a mapping, you must have the tracker
243  * mutex to obtain the right bucket. If you already have the right bucket in
244  * hand, you only need the bucket mutex.
245  *
246  * In truth, buckets which are not the "currentBucket" are not modified again,
247  * so a future optimization would allow for reading a mapping on a finished
248  * bucket without synchronization.
249  *
250  * The VLSNRange is updated as an atomic assignment to a volatile field after
251  * taking the mutex on the current bucket. It is read without a mutex, by
252  * looking at it as a volatile field.
253  *
254  * The hierarchy is
255  *   VLSNIndex -> VLSNTracker -> VLSNBucket
256  *   VLSNIndex -> VLSNTracker -> VLSNRange
257  *   VLSNIndex -> VLSNIndex.mappingSynchronizer
258  *   VLSNIndex.flushSynchronizer -> VLSNTracker
259  *
260  * Removing mappings vs reading mappings - sync on the range.
261  *
262  * We also need to consider that fact that callers of the VLSNIndex may be
263  * holding other mutex, or IN latches, and that the VLSNIndex methods may do
264  * database operations to read or write to the internal VLSN database. That can
265  * result in a nested database operation, and we need to be careful to avoid
266  * deadlocks. To be safe, we disable critical eviction [#18475]
267  * VLSNBucket.writeDatabase().
268  *
269  * Writers
270  * -------
271  * Allocating a new VLSN: bump()
272  *  - sync on log write latch
273  *    Note that since there is no synchronization on the VLSNINdex itself,
274  *    [allocating  new VLSN, logging its entry] and [flushing the vlsn index
275  *     to disk] is not atomic. See awaitConsistency().
276  *
277  * Adding a mapping: put()
278  *   - sync on VLSNIndex
279  *        -sync on VLSNTracker to access the right bucket, and possibly
280  *         create a new bucket. Atomically modify the VLSNRange.
281  *
282  * Flushing mappings to disk: writeToDatabase()
283  *    - sync on VLSNIndex.flushSyncrhonizer -> VLSNTracker
284  *
285  * Replica side syncup truncates the VLSNIndex from the end:
286  *    - no synchronization needed, the system is quiescent, and we can assume
287  *      that VLSNs are neither read nor written by other threads.
288  *
289  * Log cleaning truncates the VLSNIndex from the beginning:
290  *    We assume that the log cleaner is prohibited from deleting files that are
291  *    being used for current feeding. We can also assume that the end of the
292  *    log is not being deleted, and that we're not conflict with put(). We do
293  *    have to worry about conflicting with backwards scans when executing
294  *    syncup as a feeder, and with flushing mappings to disk. Shall we
295  *    disable log file deletion at this point?
296  *
297  *   Steps to take:
298  *
299  *    First change the VLSNRange:
300  *    - sync on VLSNIndex
301  *           - atomically modify the VLSNRange to ensure that no readers or
302  *             writers touch the buckets that will be deleted.
303  *           - sync on VLSNTracker to delete any dead buckets. Do that before
304  *             updating the on-disk database, so that we don't lose any
305  *             buckets to writeToDatabase().
306  *     - without synchronization, scan the database and non-transactionally
307  *       delete any on-disk buckets that are <= the log cleaned file.
308  *
309  * Readers
310  * -------
311  * Active forward feeder checks if a mapping exists, and waits if necessary
312  *    - read the current VLSNRange w/out a mutex. If not satisfactory
313  *        - sync on VLSNIndex
314  *              - sync on VLSNIndex.mappingSynchronizer
315  *
316  * Active forward feeder reads a mapping:
317  *  first - getBucket()
318  *     - sync on VLSNTracker to access the right bucket
319  *  if bucket is in hand
320  *    - sync on target bucket to read bucket
321  */

536      * or the latch was explicitly terminated.
537      *
538      * @return the LogItem associated with the vlsn, or null if the entry is
539      * now present in the log, but is not available in the LogItemCache.
540      */
541     public LogItem waitForVLSN(VLSN vlsn, int waitTime)
542         throws InterruptedException, WaitTimeOutException {
543 
544         /* First check the volatile range field, without synchronizing. */
545         VLSNRange useRange = tracker.getRange();
546         if (useRange.getLast().compareTo(vlsn) >= 0) {
547             return logItemCache.get(vlsn);
548         }
549 
550         VLSNAwaitLatch waitLatch = null;
551         synchronized (this) {
552             useRange = tracker.getRange();
553             if (useRange.getLast().compareTo(vlsn) >= 0) {
554                 return logItemCache.get(vlsn);
555             }
556 
557             synchronized (mappingSynchronizer) {
558                 /* The target VLSN hasn't arrived yet, we'll wait. */
559                 setupWait(vlsn);
560 
561                 /* Copy the latch while synchronized. */
562                 waitLatch = vlsnPutLatch;
563             }
564         }
565 
566         /*
567          * Do any waiting outside the synchronization section. If the
568          * waited-for VLSN has already arrived, the waitLatch will have been
569          * counted down, and we'll go through.
570          */
571         if (!waitLatch.await(waitTime, TimeUnit.MILLISECONDS) ||
572             waitLatch.isTerminated()) {
573             /* Timed out waiting for an incoming VLSN, or was terminated. */
574             throw new WaitTimeOutException();
575         }
576 
577         if (! (tracker.getRange().getLast().compareTo(vlsn) >= 0)) {
578             throw EnvironmentFailureException.
579             unexpectedState(envImpl, "Waited for vlsn:" + vlsn +
580                             " should be greater than last in range:" +
581                             tracker.getRange().getLast());
582         }
583         LogItem logItem = waitLatch.getLogItem();
584         /* If we waited successfully, logItem can't be null. */
585         return logItem.header.getVLSN().equals(vlsn) ?
586                logItem :
587                /*
588                 * An out-of-order vlsn put, that is, a later VLSN arrived at
589                 * the index before this one. We could look for it in the log
590                 * item cache, but due to the very nature of the out of order
591                 * put it's unlikely to be there and we would rather not incur
592                 * the overhead of a failed lookup.
593                 */
594                null;
595     }

659      * VLSNIndex, we need to update the range before updating the buckets.
660      *

673      * case, we need to insert a ghost bucket.

676      * deletion occurs. [#20702]

685             return false;

720     private synchronized void truncateDatabaseFromHead(VLSN deleteEnd,
721                                                        long deleteFileNum) {
722 
723         /*
724          * Be sure that the changes are fsynced before deleting any files.  The
725          * changed vlsn index must be persisted so that there are no references
726          * to the deleted, cleaned files. Instead of using COMMIT_SYNC, use
727          * COMMIT_NO_SYNC with an explicit environment flush and fsync, because
728          * the latter ends the txn and releases locks sooner, and reduces
729          * possible lock contention on the VLSNIndex. Both feeders and write
730          * operations need to lock the VLSNIndex, so keeping lock contention
731          * minimal is essential.
732          * [#20702]
733          */
734         TransactionConfig config = new TransactionConfig();
735         config.setDurability(Durability.COMMIT_NO_SYNC);
736         Txn txn = Txn.createLocalTxn(envImpl, config);
737         boolean success = false;
738         try {
739             synchronized (flushSynchronizer) {
740                 pruneDatabaseHead(deleteEnd, deleteFileNum, txn);
741                 flushToDatabase(txn);
742             }
743             txn.commit();
744             envImpl.flushLog(true /*fsync required*/);
745             success = true;
746         } finally {
747             if (!success) {
748                 txn.abort();
749             }
750         }
751     }

753     /**
754      * Remove all information from the VLSNIndex for VLSNs >= deleteStart Used
755      * by replica side syncup, when the log is truncated. Assumes that the
756      * vlsnIndex is quiescent, and no writes are happening, although the
757      * cleaner may read the vlsnIndex.
758      * @throws DatabaseException
759      */
760     public synchronized void truncateFromTail(VLSN deleteStart, long lastLsn)
761         throws DatabaseException {
762 
763         logItemCache.clear();
764         VLSNRange currentRange = tracker.getRange();
765         if (currentRange.getLast().getNext().equals(deleteStart)) {
766 
767             /*
768              * deleteStart directly follows what's in this range, no need to
769              * delete anything.
770              */
771             return;
772         }
773 
774         /*
775          * The VLSNIndex has two parts -- the in-memory cache, and the
776          * database. Update the tracker, which holds the cache first, and then
777          * update the database.
778          */
779         tracker.truncateFromTail(deleteStart, lastLsn);
780 
781         TransactionConfig config = new TransactionConfig();
782 
783         /*
784          * Be sure to commit synchronously so that changes to the vlsn index
785          * are persisted before the log is truncated. There are no feeders or
786          * repstream write operations at this time, so the use of COMMIT_SYNC
787          * does not introduce any lock contention. [#20702]
788          */
789         config.setDurability(Durability.COMMIT_SYNC);
790         Txn txn = Txn.createLocalTxn(envImpl, config);
791         boolean success = false;
792         try {
793             /*
794              * The tracker knows the boundary between VLSNs that are on disk
795              * and VLSNs that are within its cache, and maintains that info
796              * as mappings are added, and as the tracker/cache is flushed.
797              * But since we're potentially truncating mappings that were on
798              * disk, we need to update the tracker's notion of where the flush
799              * boundary is.
800              */
801             VLSN lastOnDisk = pruneDatabaseTail(deleteStart, lastLsn, txn);
802             tracker.setLastOnDiskVLSN(lastOnDisk);
803 
804             /*
805              * Because mappings can come out of order, it's possible that
806              * buckets are not completely contiguous, and that truncating
807              * will result in the loss of the mapping for the end of the range.
808              * For example, suppose the buckets are like this:
809              *  On disk: vlsn 13 -> bucket for vlsns 13-16
810              *  In tracker: vlsn 18 -> bucket for vlsns 18 -23
811              * Truncating the vlsnIndex at 18 will make the last VLSN become
812              * 17, and removing the vlsn 18 bucket will result in no mapping
813              * for the new end range, vlsn 17. If so, the tracker should
814              * create a new mapping, of vlsn 17 -> lastlsn, to cap off the
815              * range and ensure that there are mappings for the start and end
816              * lsns.
817              */
818             tracker.ensureRangeEndIsMapped(deleteStart.getPrev(), lastLsn);
819             flushToDatabase(txn);
820             txn.commit();
821             success = true;
822         } finally {
823             if (!success) {
824                 txn.abort();
825             }
826         }
827     }

976      * called, we guarantee that a non-null bucket will be returned.
977      *
978      * At this point, we are sure that the target vlsn is within the range of
979      * vlsns held in the database. However, note that there is no explicit

981      * VLSNTracker.flushToDatabase, which might be writing additional buckets
982      * to this database. This may affect the cases when the cursor search
983      * does not return a equality match on a bucket. [#20726]
984      *
985      * For example, suppose the database looks like this:
986      * key=vlsn 10, data = bucket: vlsn 10 -> lsn 0x10/100
987      *                             vlsn 15 -> lsn 0x10/150
988      * key=vlsn 20, data = bucket: vlsn 20 -> lsn 0x11/100
989      *                             vlsn 25 -> lsn 0x11/150
990      * If we are looking for a bucket for vlsn 22, there will not be a match
991      * from the call to cursor.getSearchKeyRange(key=22). The code that
992      * accounts for that will need to consider that new buckets may be flushed
993      * to the database while the search for a new bucket is going on. For
994      * example,
995      *
996      * key=vlsn 30, data = bucket: vlsn 30 -> lsn 0x12/100
997      *                             vlsn 35 -> lsn 0x12/150
998      *
999      * may be written to the database while we are searching for a bucket that
1000      * owns vlsn 22.
1001      */
1002     private VLSNBucket getGTEBucketFromDatabase(VLSN target,
1003                                                 VLSNBucket currentBucketInUse)
1004         throws DatabaseException {
1005 
1006         Cursor cursor = null;
1007         Locker locker = null;
1008         try {
1009             locker = BasicLocker.createBasicLocker(envImpl);
1010             cursor = makeCursor(locker);
1011 
1012             /*
1013              * Look at the bucket at key >= target.Will return null if no GTE
1014              * bucket.
1015              */
1016             VLSNBucket bucket = examineGTEBucket(target, cursor);
1017             if (bucket != null) {
1018                 return bucket;
1019             }
1020 
1021             /*
1022              * We're here because we did not find a bucket >= target. Let's
1023              * examine the last bucket in this database. We know that it will
1024              * either be:
1025              *
1026              * 1) a bucket that's < target, but owns the mapping
1027              * 2) if the index was appended to by VLSNTracker.flushToDatabase
1028              *    while the search is going on, the last bucket may be one
1029              *    that is > or >= target.
1030              * Using the example above, the last bucket could be case 1:
1031              *
1032              * a bucket that is < target 22:
1033              *    key=vlsn 20, data = bucket: vlsn 20 -> lsn 0x11/100
1034              *                                vlsn 25 -> lsn 0x11/150
1035              *
1036              * or case 2, a bucket that is >= target 22, because the index grew
1037              *    key=vlsn 30, data = bucket: vlsn 30 -> lsn 0x12/100
1038              *                                vlsn 35 -> lsn 0x12/150
1039              */

1042             DatabaseEntry key = new DatabaseEntry();
1043             DatabaseEntry data = new DatabaseEntry();
1044             OperationStatus status = cursor.getLast(key, data,
1045                                                     LockMode.DEFAULT);
1046             if (isValidBucket(status, key)) {
1047                 endBucket = VLSNBucket.readFromDatabase(data);
1048                 if (endBucket.owns(target)) {
1049                     return endBucket;
1050                 }
1051 
1052                 /*
1053                  * If this end bucket is not the owner of the target VLSN, we
1054                  * expect it to be a greaterThan bucket which was inserted
1055                  * because of a concurrent VLSNTracker.flushToDatabase call
1056                  * that did not exist when we did the previous
1057                  * cursor.getKeyRangeSearch (case 2), In that case, we can
1058                  * search again for the owning bucket.
1059                  */
1060                 if (endBucket.follows(target)) {
1061                     bucket = examineGTEBucket(target, cursor);
1062                     if (bucket != null) {
1063                         return bucket;
1064                     }
1065                 }
1066             }
1067 
1068             /*
1069              * Shouldn't get here! There should have been a bucket in this
1070              * database >= this target.
1071              */
1072 
1073             /* Dump the bucket database for debugging. */
1074             int count = 0;
1075             StringBuilder sb = new StringBuilder();
1076             status = cursor.getFirst(key, data, LockMode.DEFAULT);

1079                 sb.append("key => " + keyValue + "\n");
1080

1082                     VLSNRange range = VLSNRange.readFromDatabase(data);
1083                     sb.append("range =>" + range + "\n");
1084                 } else {
1085                     bucket = VLSNBucket.readFromDatabase(data);
1086                     sb.append("bucket => " + bucket + "\n");
1087                 }
1088 
1089                 count++;
1090                 status = cursor.getNext(key, data, LockMode.DEFAULT);
1091             }

1097                 (envImpl, "Couldn't find bucket for GTE VLSN " + target +
1098                  " in database. EndBucket=" + endBucket + "currentBucket=" +
1099                  currentBucketInUse + " tracker = " + tracker);
1100         } finally {
1101             if (cursor != null) {
1102                 cursor.close();
1103             }
1104

1106                 locker.operationEnd(true);
1107             }
1108         }

1114      * @return null if no GTE bucket was found.

1116     private VLSNBucket examineGTEBucket(VLSN target, Cursor cursor) {
1117 
1118         /* getSearchKeyRange will return a bucket >= target if one exists */
1119         DatabaseEntry key = new DatabaseEntry();
1120         DatabaseEntry data = new DatabaseEntry();
1121         LongBinding.longToEntry(target.getSequence(), key);
1122         OperationStatus status =
1123             cursor.getSearchKeyRange(key, data, LockMode.DEFAULT);
1124 
1125         if (status == OperationStatus.SUCCESS) {
1126             VLSNBucket bucket = VLSNBucket.readFromDatabase(data);
1127             if (bucket.owns(target)) {
1128                 return bucket;
1129             }
1130

1132              * The bucket we found is > than our target. Look at the

1535     /*
1536      * Update this index, which was initialized with what's on disk, with
1537      * mappings found during recovery. These mappings ought to either overlap
1538      * what's on disk, or cover the range immediately after what's on disk.  If
1539      * it doesn't, the recovery mechanism, which flushes the mapping db at
1540      * checkpoint is faulty and we've lost mappings.
1541      *
1542      * In other words, if this tracker holds the VLSN range a -> c, then the
1543      * recovery tracker will have the VLSN range b -> d, where
1544      *
1545      *   a <= b
1546      *   c <= d
1547      *   if c < b, then b == c+1
1548      *
1549      * This method must be called when the index and tracker are quiescent, and
1550      * there are no calls to track().
1551      *
1552      * The recoveryTracker is the authoritative voice on what should be in the
1553      * VLSN index.
1554      */
1555     void merge(VLSNRecoveryTracker recoveryTracker) {
1556 
1557         if (recoveryTracker == null) {
1558             flushToDatabase(Durability.COMMIT_SYNC);
1559             return;
1560         }
1561 
1562         if (recoveryTracker.isEmpty()) {
1563 
1564             /*
1565              * Even though the recovery tracker has no mappings, it may have
1566              * seen a rollback start that indicates that the VLSNIndex should
1567              * be truncated. Setup the recovery tracker so it looks like
1568              * it has a single mapping -- the matchpoint VLSN and LSN and
1569              * proceed. Take this approach, rather than truncating the index,
1570              * because we may need that matchpoint mapping to cap off the
1571              * VLSN range.
1572              *
1573              * For example, suppose an index has mappings for VLSN 1, 5, 10,
1574              * and the rollback is going to matchpoint 7. A pure truncation
1575              * would lop off VLSN 10, making VLSN 5 the last mapping. We
1576              * would then need to add on VLSN 7.
1577              */
1578             VLSN lastMatchpointVLSN = recoveryTracker.getLastMatchpointVLSN();
1579             if (lastMatchpointVLSN.isNull()) {
1580                 return;
1581             }
1582 
1583             /*
1584              * Use a MATCHPOINT log entry to indicate that this is a syncable
1585              * entry. This purposefully leaves the recovery tracker's range's
1586              * lastTxnEnd null, so it will not overwrite the on disk
1587              * tracker. This assumes that we will never rollback past the last
1588              * txn end.
1589              */
1590             recoveryTracker.track(lastMatchpointVLSN,
1591                                   recoveryTracker.getLastMatchpointLsn(),
1592                                   LogEntryType.LOG_MATCHPOINT.getTypeNum());
1593         }
1594 
1595         /*
1596          * The mappings held in the recoveryTracker must either overlap what's
1597          * on disk or immediately follow the last mapping on disk. If there
1598          * is a gap between what is on disk and the recovery tracker, something
1599          * went awry with the checkpoint scheme, which flushes the VLSN index
1600          * at each checkpoint. We're in danger of losing some mappings. Most
1601          * importantly, the last txnEnd VLSN in the range might not be right.
1602          *
1603          * The one exception is when the Environment has been converted from
1604          * non-replicated and there are no VLSN entries in the VLSNIndex. In
1605          * that case, it's valid that the entries seen from the recovery
1606          * tracker may have a gap in VLSNs. For example, in a newly converted
1607          * environment, the VLSN index range has NULL_VLSN as its last entry,
1608          * but the first replicated log entry will start with 2.
1609          *
1610          * Note: EnvironmentImpl.needRepConvert() would more accurately convey
1611          * the fact that this is the very first recovery following a
1612          * conversion.  But needRepConvert() on a replica is never true, and we
1613          * need to disable this check on the replica's first recovery too.
1614          */
1615         VLSN persistentLast = tracker.getRange().getLast();
1616         VLSN recoveryFirst = recoveryTracker.getRange().getFirst();
1617         if ((!(envImpl.isRepConverted() && persistentLast.isNull()) ||
1618              !envImpl.isRepConverted()) &&
1619             recoveryFirst.compareTo(persistentLast.getNext()) > 0) {
1620 
1621             throw EnvironmentFailureException.unexpectedState
1622                 (envImpl, "recoveryTracker should overlap or follow on disk " +
1623                  "last VLSN of " + persistentLast + " recoveryFirst= " +
1624                  recoveryFirst);
1625         }
1626 
1627         VLSNRange currentRange = tracker.getRange();
1628         if (currentRange.getLast().getNext().equals(recoveryFirst)) {
1629             /* No overlap, just append mappings found at recovery. */
1630             tracker.append(recoveryTracker);
1631             flushToDatabase(Durability.COMMIT_SYNC);
1632             return;
1633         }
1634 
1635         /*
1636          * The mappings in the recovery tracker should overwrite those in the
1637          * VLSN index.
1638          */
1639         TransactionConfig config = new TransactionConfig();
1640         config.setDurability(Durability.COMMIT_SYNC);
1641         Txn txn = Txn.createLocalTxn(envImpl, config);
1642         boolean success = false;
1643         VLSN lastOnDiskVLSN;
1644         try {
1645             lastOnDiskVLSN = pruneDatabaseTail(recoveryFirst, DbLsn.NULL_LSN,
1646                                                txn);
1647             tracker.merge(lastOnDiskVLSN, recoveryTracker);
1648             flushToDatabase(txn);
1649             txn.commit();
1650             success = true;
1651         } finally {
1652             if (!success) {
1653                 txn.abort();
1654             }
1655         }
1656     }

2075     /**
2076      * Ensure that the in-memory vlsn index encompasses all logged entries
2077      * before it is flushed to disk. A No-Op for non-replicated systems.
2078      *
2079      * The problem is in the interaction of logging and VLSN
2080      * tracking. Allocating an new VLSN and logging a replicated log entry is
2081      * done within the log write latch, without any VLSNINdex
2082      * synchronization. That must be done to keep the log write latch critical
2083      * section as small as possible, and to avoid any lock hiearchy issues.
2084      *
2085      * The VLSNIndex is updated after the log write latch critical section. The
2086      * VLSNIndex is flushed to disk by checkpoint, and it is assumed that this
2087      * persistent version of the index encompasses all VLSN entries prior to
2088      * checkpoint start. Since the logging of a new VLSN, and the flushing of
2089      * the index are not atomic, it's possible that the checkpointer may start
2090      * the flush of the vlsnIndex before the last vlsn's mapping is recorded
2091      * in the index. To obey the requirement that the checkpointed vlsn index
2092      * encompass all mappings < checkpoint start, check that the vlsn index
2093      * is up to date before the flush.
2094      * [#19754]
2095      *
2096      * awaitConsistency() works by using the same waitForVLSN() method used by
2097      * the Feeders. WaitForVLSN asserts that all feeders are waiting on single
2098      * vlsn, to assure that no feeders are left in limbo, awaiting a vlsn that
2099      * has gone by. This contract is valid for the feeders, because they wait
2100      * for vlsns sequentially, consuming each one by one. However, this ckpter
2101      * awaitConsistency functionality uses the nextVLSNCounter, which can
2102      * leapfrog ahead arbitrarily, in this case:
2103      *
2104      *    vlsn range holds 1 -> N-1
2105      *    Feeder is present, awaiting vlsn N
2106      *    thread A bumps vlsn to N  and writes record under log write latch
2107      *    thread B bumps vlsn to N + 1 and writes record under log write latch
2108      *    ckpter awaits consistency, using N+1, while feeders are awaiting N
2109      *    thread A puts VLSN N outside log write latch
2110      *    thread B puts VLSN N+1 outside log write latch
2111      *
2112      * Because of this, the ckpter must distinguish between what it is really
2113      * waiting on (VLSN N+1) and what is can next wait on to fulfil the
2114      * feeder waiting contract (VLSN N)
2115      */
2116     public void awaitConsistency() {
2117 
2118         /* VLSNIndex is not initialized and in use yet, no need to wait. */
2119         if (nextVLSNCounter == null) {
2120             return;
2121         }

2129              * previously determined vlsnAllocatedBeforeCkpt was decremented
2130              * due to a logging failure.
2131              */
2132             if (vlsnAllocatedBeforeCkpt == null) {
