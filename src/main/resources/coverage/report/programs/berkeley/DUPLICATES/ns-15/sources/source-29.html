24 /**
25  * This asynchronous exception indicates that a new master has been selected,
26  * this <code>Replica</code>'s log is ahead of the current <code>Master</code>,
27  * and in this case, the <code>Replica</code> was unable to rollback without a
28  * recovery. As a consequence, it is possible that one or more of the most
29  * recently committed transactions may need to be rolled back, before the
30  * <code>Replica</code> can synchronize its state with that of the current
31  * <code>Master</code>. Note that any CommitTokens obtained before restarting
32  * this <code>Replica</code> shouldn't be used after {@link RollbackException}
33  * is thrown because the token may no longer exist on the current
34  * <code>Master</code> node, due to failover processing.
35  * <p>
36  * Existing {@link ReplicatedEnvironment}, and consequently {@link Database}
37  * handles, are invalidated as a result of this exception. The application must
38  * close all old handles and create new handles before it can proceed. The
39  * actual rollback of any recently committed transactions is done when the
40  * application re-instantiates and thereby reopens the {@link
41  * ReplicatedEnvironment}.  The application is responsible for discarding and
42  * recreating any transient state that may be associated with the committed
43  * transactions that were rolled back. {@link #getEarliestTransactionId} and 
44  * {@link #getEarliestTransactionCommitTime} provide information to help
45  * determine which transactions might be rolled back. Note that it is possible
46  * that no committed transactions have been rolled back and that the 
47  * application need do no adjustments, in which case 
48  * {@link #getEarliestTransactionCommitTime} will return null.
49  * <p>
50  * This exception should be encountered relatively infrequently in practice,
51  * since the election mechanism favors nodes with the most advanced log when
52  * deciding upon a master. The exception, due to its nature, can only be
53  * encountered when the node is in the <code>Replica</code> state, or the node
54  * is trying to transition to the <code>Replica</code> state.
55  * <p>
56  * Use of weak durability requirements like
57  * {@link com.sleepycat.je.Durability.ReplicaAckPolicy#NONE} or a
58  * {@link com.sleepycat.je.rep.ReplicationMutableConfig#NODE_PRIORITY} of zero
59  * increases the likelihood of this exception.
60  * @see RollbackProhibitedException
61  */
